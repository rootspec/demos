# Level 2: Stable Truths

**Product:** RootWeather
**Version:** 7.3.7
**Status:** Baseline

---

## Truth: Data-first, display-second

The application fetches real weather data from Open-Meteo (free, no API key required) and derives all display logic — alerts, emojis, descriptions — from that data. We choose reliable open data over premium APIs.

**We choose:** Zero-cost, no-registration data access
**Over:** Richer proprietary APIs that require keys, billing, or rate-limit management

---

## Truth: Local persistence is the only persistence

All user preferences (unit, wind unit, time format, default city, favorites) are stored in browser `localStorage`. There is no backend, no account, no sync. The browser is the database.

**We choose:** Zero-infrastructure persistence with full user ownership
**Over:** Server-side accounts and cloud sync that require authentication and infrastructure

---

## Truth: One UI, multiple views — not multiple pages

RootWeather is a single-page application. The three views (weather, dashboard, compare) are rendered conditionally by view state, not by routing or page navigation. This keeps the app fast and stateless.

**We choose:** View-state toggling within a single component tree
**Over:** Client-side routing (React Router, etc.) that adds complexity for a single-domain app

---

## Truth: Condition-derived alerts, not push notifications

Weather alerts are computed client-side from current condition thresholds (UV index, wind speed, humidity, temperature). There is no alert subscription, no notification system, no server push. Alerts appear when and only when the data warrants them.

**We choose:** Always-relevant, zero-latency inline alerts
**Over:** Push notification infrastructure that introduces permission flows and false positives

---

## Truth: Favorites are the unit of multi-location

The favorites list is the source of truth for saved locations. Both the dashboard view and the comparison view operate exclusively on favorites. A city must be saved before it can appear in either multi-location feature.

**We choose:** A single, unified saved-location list driving all multi-city features
**Over:** Separate "pinned" or "watched" lists for dashboard vs. comparison

---

## Truth: Settings apply globally and immediately

Changing a unit or format preference in SettingsPanel updates all active views instantly. There is no "Apply" button, no confirmation step. The change takes effect on state update.

**We choose:** Immediate feedback on preference changes
**Over:** Deferred application of settings

---

## Definition of Success

A user can:
1. Search for any city in the world and see current weather within seconds
2. Save multiple cities and compare them side-by-side
3. Return to the app and find their preferences and favorites exactly as they left them
4. Understand current hazard conditions (UV, wind, temperature, humidity) without reading raw numbers
