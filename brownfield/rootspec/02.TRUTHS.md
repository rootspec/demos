# Level 2: Stable Truths

> References: `01.PHILOSOPHY.md`

## Truth: Data over chrome

The primary product is the weather data. UI elements exist to surface data, not to fill space or demonstrate technical capability. Every component earns its presence by communicating information.

**Trade-off:** We choose functional density over visual polish. A plain card with accurate data is better than a beautiful widget with misleading data.

## Truth: Local state is the right backend

User preferences — units, saved locations, default city, time format — belong in the browser. This eliminates the need for authentication, accounts, or server-side user state. The product is stateless from the server's perspective.

**Trade-off:** We choose local-first simplicity over cross-device sync. Users who care about sync can use the same browser profile.

## Truth: The API should be invisible

The Open-Meteo API requires no registration and no API key. The product design reflects this: there is no setup step, no configuration, no credentials to manage. The API is an implementation detail the user never encounters.

**Trade-off:** We choose zero-friction access over advanced features that would require auth (e.g., push notifications, personalized forecasts). The Open-Meteo constraint is a feature, not a limitation.

## Truth: Views are modes, not pages

The app has three views — single-city weather, multi-location dashboard, and side-by-side comparison. These are modes of the same app, not separate pages with separate URLs. Navigation between modes is immediate and stateful.

**Trade-off:** We choose fluid mode-switching over deep-linkable URLs. This keeps the architecture simple without a router dependency.

## Truth: Alerts are derived, not fetched

Weather alerts (UV warnings, wind warnings, temperature extremes, humidity flags) are computed client-side from the raw API data. There is no separate alert API, no push subscription, no delay. If the conditions warrant an alert, it appears.

**Trade-off:** We choose computed client-side alerts over official meteorological alert services. This is adequate for the general use case and eliminates a dependency.

## Truth: Settings are progressive, not required

Settings are available and persistent but not mandatory. Default values are sensible (Celsius, km/h, 24-hour time). The app is fully functional without the user ever opening the settings panel.

**Trade-off:** We choose sane defaults with optional customization over forced setup. A first-time user sees weather immediately.

## Definition of Success

A user opens RootWeather on a new device with no prior configuration, searches for their city, and has actionable weather information within ten seconds — without an account, without dismissing any modal, and without seeing an ad.

A returning user opens the app and sees weather for their default city immediately, before touching anything.
