# Level 2: Stable Truths

## Design Philosophy

### Truth: Simplicity over completeness
Show the weather data that matters most — current conditions, today's hours, the week ahead. Do not attempt to be a meteorological tool. Radar maps, historical data, and minute-by-minute forecasts are out of scope.

### Truth: Client-only architecture
The entire application runs in the browser with no backend. All data comes from public APIs. All user state lives in localStorage. This eliminates infrastructure cost, deploy complexity, and privacy concerns.

### Truth: Open data, zero friction
The product depends only on free, keyless APIs. Users never configure API keys, create accounts, or accept terms. If a data source requires authentication, it is not a candidate.

### Truth: Personalization through persistence, not profiles
User preferences (units, time format, saved locations, default city) are stored locally and applied silently. There is no concept of a user profile, sync, or export. Losing a browser means losing preferences — this is an accepted trade-off.

## Trade-offs

| We choose | Over | Why |
|-----------|------|-----|
| Glanceability | Comprehensiveness | Supports Effortless Clarity — users want a quick read, not a deep analysis |
| Client-only | Server infrastructure | Supports zero-friction access — nothing to deploy, no accounts, no cost |
| localStorage | Cloud sync | Supports Familiar Comfort without requiring accounts — accepted data loss risk |
| Free public APIs | Premium data sources | Supports open access principle — no API keys, no cost, no terms gates |
| Emoji + text indicators | Icon libraries or images | Supports Ambient Awareness — universally understood, zero asset loading |

## Definition of Success

The product succeeds when a user can search a city, understand current conditions, glance at the forecast, and save the location — all within [brief duration] of their first visit, with no instructions or onboarding.

## Constraints

- **No build-time secrets** — no API keys in environment variables or config
- **Single-page app** — no routing, no page transitions, no server rendering
- **Responsive but not mobile-first** — works on mobile but designed for desktop-width viewports
- **No real-time updates** — weather refreshes on navigation or city change, not on a polling interval
- **Subpath deployment** — the app is served at `/demos/brownfield/` on GitHub Pages; the build tool's base path must be set so all asset URLs and internal links resolve correctly at that subpath
