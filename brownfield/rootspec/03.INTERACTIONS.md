# Level 3: Interaction Architecture

> References: `01.PHILOSOPHY.md`, `02.TRUTHS.md`

## Interaction Loop Architecture

| Scale    | Scope                         | Duration     | Key Patterns                            |
|----------|-------------------------------|--------------|-----------------------------------------|
| Immediate | Single action                 | Seconds      | Search → select → weather appears       |
| Session  | Explore multiple locations    | Minutes      | Switch locations, adjust settings, compare |
| Extended | Return visit                  | Across days  | Default city auto-loads, favorites persist |
| Lifetime | Build a personal weather view | Ongoing      | Maintain favorites, refine default city  |

---

## Core Interaction Loops

### Loop 1: Look Up a City

**Trigger:** User wants to know the weather somewhere.

1. User types in the search bar (minimum [search_min_chars] characters)
2. Typeahead fires after [search_debounce_ms] ms of inactivity
3. Dropdown shows up to [search_max_results] city suggestions with region and country
4. User selects a city
5. App fetches weather for that city's coordinates
6. Current conditions, hourly forecast, and 7-day chart appear
7. A "Save Location" button appears if the city is not already saved

**Feedback:** Loading state while fetching. Error state if fetch fails.

**Exit:** City is selected; weather view is active.

---

### Loop 2: Save and Return to a Favorite

**Trigger:** User wants to remember a location for quick access.

1. User looks up a city (Loop 1)
2. User clicks "Save Location"
3. City is added to the favorites list (rendered as chips above the weather card)
4. On a return visit, the favorites list is immediately visible
5. User clicks a chip to switch to that city's weather

**Feedback:** "Save Location" button disappears once city is saved. Active city chip is highlighted.

**Exit:** Favorites persisted to localStorage; user can navigate between saved cities.

---

### Loop 3: Configure Preferences

**Trigger:** User wants to change units or set a default city.

1. User opens the Settings panel (toggle button in the main interface)
2. User selects temperature unit (°C / °F), wind speed unit (km/h / mph), or time format (24h / 12h)
3. User optionally selects a default city from their favorites
4. Settings close (or stay open — panel is a toggle)
5. All displayed values immediately reflect the new settings

**Feedback:** Changes are immediate and visible. Settings persist on reload.

**Exit:** Preferences saved to localStorage.

---

### Loop 4: View All Saved Locations (Dashboard)

**Trigger:** User has [min_favorites_for_dashboard] or more saved locations and wants an overview.

1. User switches to the Dashboard view via the view toggle in the header
2. Dashboard fetches current conditions for all saved cities in parallel
3. Each city renders as a card: name, weather emoji, temperature, humidity, wind
4. User can click any card to switch to that city's detailed weather view
5. User can initiate comparison from this view

**Feedback:** "Loading all locations..." while fetching. Cards render as data arrives (all-at-once after Promise.all).

**Exit:** Overview visible; user can navigate to single city or start comparison.

---

### Loop 5: Compare Multiple Cities

**Trigger:** User wants to compare weather across two or more saved cities.

1. User clicks "Compare" in the dashboard view (visible when [min_favorites_for_compare] or more favorites exist)
2. App enters compare selection mode: a prompt appears, and each location card becomes selectable
3. User selects cities by clicking cards (up to [max_compare_cities] cities)
4. When [min_compare_cities] or more cities are selected, app automatically transitions to comparison view
5. Comparison view shows selected cities in side-by-side columns
6. Each column contains: city name, alerts, current conditions, hourly forecast, 7-day chart
7. User can remove a city from comparison; if fewer than [min_compare_cities] remain, returns to dashboard
8. "Back to Dashboard" returns to the multi-location overview

**Feedback:** Selected cards are highlighted with a border. Column layout expands the max width of the app.

**Exit:** Comparison complete; user returns to dashboard or navigates away.

---

### Loop 6: Return Visit (Default City)

**Trigger:** User returns to the app with a default city configured.

1. App loads
2. App reads default city from localStorage
3. If the default city exists in favorites, it is auto-selected as the active city
4. Weather loads automatically without any user action

**Feedback:** Weather appears immediately on load (after API response).

**Exit:** User is on the weather view for their default city without any interaction.

---

## Failure Modes

| Scenario | Behavior |
|----------|----------|
| Search returns no results | Dropdown does not open; input stays active |
| Weather fetch fails | Error message replaces the weather card; loading state clears |
| Dashboard fetch fails for one city | That card shows "Unable to load" |
| Default city not found in favorites | App loads with no active city; user must search |
| Fewer than [min_compare_cities] cities selected before leaving compare mode | comparedCities cleared; view returns to dashboard |

---

## View State Machine

```
[No city selected] → search → [City selected: Weather view]
[Weather view] → save location → [Favorites populated]
[Favorites populated] → view toggle → [Dashboard view]
[Dashboard view] → Compare button → [Compare selection mode]
[Compare selection mode] → select ≥ 2 cities → [Comparison view]
[Comparison view] → Back to Dashboard → [Dashboard view]
[Comparison view] → remove city below minimum → [Dashboard view]
[Any view] → search + select → [Weather view for new city]
```

---

## Settings Interaction

Settings do not require navigation — the panel is a slide-in/toggle within the current view. Changing a setting takes immediate effect on all displayed values. The panel can be dismissed by toggling again.
