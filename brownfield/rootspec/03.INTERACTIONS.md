# Level 3: Interaction Architecture

## Core Journeys

### Journey: Check Weather
The primary loop. User arrives, searches or selects a city, sees conditions and forecast.

**Trigger:** User opens the app or searches a new city.
**Flow:**
1. App loads → if default city is set and saved, show its weather immediately
2. User types in search bar → debounced autocomplete shows city matches
3. User selects a city → current conditions, alerts, hourly forecast, and 7-day forecast load
4. User glances at information and leaves

**Feedback:** Loading indicator during fetch. Error message if API fails. Empty state prompt if no city selected.

### Journey: Manage Locations
User builds a personal set of cities for quick access.

**Trigger:** User finds a city worth revisiting.
**Flow:**
1. After viewing weather for a city → "Save Location" button appears (if not already saved)
2. User saves → city appears as a chip in the favorites bar
3. User clicks a favorite chip → weather view switches to that city
4. User removes a favorite → chip disappears; if it was the default city, default is cleared

**Feedback:** Favorite chips highlight the active city. Remove affordance is inline on each chip.

### Journey: Compare Locations
User views saved locations as an overview grid or side by side in a detailed comparison.

**Trigger:** User has saved at least one location and switches to Dashboard view.
**Flow:**
1. View toggle appears in header when favorites exist
2. User clicks "Dashboard" → grid of location cards loads, each showing current temp and conditions
3. User clicks a card → switches to Weather view for that city
4. If [min_comparison_cities]+ favorites exist, a "Compare" button appears on the dashboard
5. User clicks "Compare" → enters comparison selection mode; cards become selectable with visual toggle
6. User selects [min_comparison_cities] to [max_comparison_cities] cities by clicking cards → selected cards are highlighted
7. Once enough cities are selected, comparison view renders: side-by-side columns showing each city's current conditions, hourly forecast, and 7-day forecast
8. User deselects a city by clicking its column header → that column disappears; if fewer than [min_comparison_cities] remain, returns to dashboard

**Feedback:** Loading state while all locations' weather is fetched in parallel. Cards show "Unable to load" on failure. In comparison selection mode, selected cards show a highlight/check indicator. Compare button shows selection count. "Select at least [min_comparison_cities] cities" prompt when not enough are selected.

### Journey: Personalize
User adjusts display preferences to match their conventions.

**Trigger:** User wants different units or formats.
**Flow:**
1. User opens Settings panel (collapsed by default)
2. User toggles temperature unit (°C / °F), wind unit (km/h / mph), time format (24h / 12h)
3. User optionally sets a default city from saved locations
4. All changes apply immediately and persist to next visit

**Feedback:** Active setting is visually highlighted. Changes reflect instantly in all displayed data.

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Patterns |
|-------|-------|----------|--------------|
| Immediate | Search, select, toggle | Seconds | Debounced input, instant setting changes, alert badges |
| Session | Check weather, compare, save | Minutes | View switching, favorites management, settings adjustment |
| Extended | Return visits | Days/weeks | Default city auto-load, persisted preferences, stable favorites list |
| Lifetime | Ongoing use | Indefinite | localStorage accumulates locations and preferences over time |

## Failure Modes

- **API unreachable** — show error message in the weather view area; favorites bar and settings remain functional
- **City not found** — autocomplete returns empty; no dropdown shown
- **localStorage full or unavailable** — preferences revert to defaults on each visit; no error surfaced (silent degradation)
- **Stale data** — no staleness indicator; data refreshes only on city change or page reload

## Edge Cases

- **Duplicate favorites** — prevented by lat/lon comparison before adding
- **Default city removed from favorites** — default city setting is cleared
- **Extremely long city names** — chips and cards truncate via CSS overflow
- **Rapid city switching** — previous fetch is cancelled (cleanup in useWeather hook)
- **Comparison with fewer than minimum cities** — exits comparison and returns to dashboard automatically
- **Compared city removed from favorites** — that column is dropped from comparison; if fewer than minimum remain, exits to dashboard
