# Level 1: Foundational Philosophy

## Competitors

- **Test-Driven Development (TDD)** — Code-level constraints that catch bugs but don't capture human intent
- **Behavior-Driven Development (BDD)** — Structured but verbose; expensive to maintain, doesn't stick
- **GitHub Copilot / Cursor** — AI code assistants that generate implementation without anchoring to intent
- **Linear / Jira** — Issue trackers that manage work but allow specs to drift from decisions
- **Notion / Confluence** — Flexible docs that become unread artifacts

## Table Stakes

- A clear explanation of what RootSpec is and why it exists
- Getting-started path for developers who want to try it
- Honest treatment of limitations and trade-offs
- Version information so visitors know what they're evaluating
- Links to source code and further documentation

## Pain Points

- **Spec drift** — By the time code ships, the original intent has been diluted through layers of interpretation
- **Philosophy-implementation gap** — Implementation decisions accumulate without any anchor to the original "why"
- **Unreliable AI output** — AI generates plausible code but has no mechanism to validate that it satisfies intent
- **Unread documentation** — Google Docs specs are written for compliance, not comprehension; nobody reads them
- **TDD/BDD overhead** — Existing constraint-based approaches are expensive, tedious, and brittle to maintain
- **Drift at the source** — Even the original intent author holds unstated assumptions that cause mid-cycle pivots

## Influences

- **Stripe / Linear / Vercel Docs** — Technical documentation as a first-class product experience
- **Literary and editorial web publishing** — Long-form essay sites that feel authored, not generated
- **Hard-copy technical writing** — Typography-first, whitespace-generous, content-forward presentation
- **Thoughtful developer blogs** — Confident, non-preachy technical writing by practitioners

## Mission

RootSpec exists to keep human intent intact from the first question to the last test — making validation the primary artifact of software development rather than an afterthought.

## Design Pillars

### 1. Clarity Over Cleverness
Users feel they immediately understand what RootSpec is and why it matters — no jargon, no buzzwords, no sales pitch.

**User perspective:** "I understood exactly what this does and why I'd want it within two minutes of arriving."

**Examples:**
- Plain-language explanations before technical detail
- Every claim backed by a concrete example or demo
- No acronyms without immediate definition

### 2. Proof Over Promise
Every claim on this site is demonstrated through the site itself, not merely stated.

**User perspective:** "This site was built with RootSpec. I can see the spec. I can see the code. The methodology isn't theoretical."

**Examples:**
- Meta-banner making the demo context explicit and prominent
- Links to actual spec files and seed document in the repo
- Interactive elements that let visitors experience the methodology firsthand

### 3. Editorial Restraint
The aesthetic signals careful thought: generous whitespace, considered typography, and content that does the work without decorative chrome.

**User perspective:** "This feels like a place where someone thinks, not a place where someone sells."

**Examples:**
- Serif body type that rewards reading
- Light mode default with dark mode toggle
- No gradient blobs, no stock illustrations, no animated counters

### 4. Approachability Without Condescension
Users find the methodology substantive but never intimidating, with a clear and immediate path to getting started.

**User perspective:** "This is clearly a real system, but I can see how to start using it. It doesn't require me to upend everything first."

**Examples:**
- Interactive wizard that produces a real skeleton spec from a single idea
- Four skills explained as a sequence, not a framework to memorize
- Honest acknowledgment of what RootSpec is not

## Inviolable Principles

- Never use marketing language that overstates what RootSpec does
- Never obscure the demo context — the meta-banner must be honest and visible above the fold
- Never default to dark mode (editorial sites read in light; dark mode is a tell of AI-generated design)
- Never use decorative visual effects that compete with content (gradients, parallax, glassmorphism, glows)
- Never list social proof that doesn't exist (no "trusted by" logos, no fake usage statistics)
- The site must itself demonstrate the methodology — if it doesn't look like a RootSpec output, it undermines the thesis

## North-Star Experience

Six months from now, a developer who has been burned by scope drift arrives at this site. Within the first scroll, they recognize their problem described accurately without exaggeration. By the end of the page, they've used the wizard to sketch their own spec, seen the hierarchy explorer, and understand that RootSpec isn't promising magic — it's promising a disciplined process with AI handling the tedious parts. They click "View the spec" in the meta-banner and find a real, structured document. They leave with `/rs-init` running in a terminal.
