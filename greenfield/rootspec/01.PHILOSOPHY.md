# Level 1: Foundational Philosophy

**Product:** RootSpec Marketing Site
**Purpose:** Explain what RootSpec is, why it exists, and invite developers to start using it.

---

## Competitors

- **Linear** — Linear's approach to issues/specs is workflow-oriented, not philosophically grounded
- **Notion** — Flexible docs tool commonly used for specs; no enforcement of hierarchy or traceability
- **Confluence** — Enterprise wiki for specifications; creates documentation nobody reads
- **GitHub Issues/Projects** — Task-centric, not spec-centric; loses the "why" behind work
- **Traditional PRDs (Google Docs)** — Unstructured, unmaintained, disconnected from code

## Table Stakes

- Clear explanation of what RootSpec is and why it exists
- Demonstration that the methodology is approachable, not academic
- Links to GitHub repositories for the framework and this demo
- Interactive features that let visitors experience the methodology firsthand
- Responsive design that works on mobile and desktop

## Pain Points

- **Spec drift** — Documentation written once, never updated, diverges from actual product
- **Philosophy-implementation gap** — Implementation decisions quietly reshape product intent over time
- **Unreliable AI output** — AI has knowledge but not wisdom; without a validation layer, output diverges from intent
- **Unread specs** — Google Docs specs that no one references after sprint planning
- **Untraceable decisions** — Features exist without any link to a user need or product principle

## Influences

- **Stripe documentation** — Clear, developer-focused, technically precise without being cold
- **Linear.app** — Confident product philosophy communicated without corporate buzzwords
- **Tailwind CSS docs** — Shows, doesn't just tell; concrete examples over abstract explanations
- **Y Combinator essays** — Direct, opinionated, written by practitioners for practitioners

---

## Mission

RootSpec exists because the real value in software development is not generating code — it is validating that what gets built matches what was intended. When AI can produce code trivially, the bottleneck becomes human wisdom: the philosophy, the trade-offs, the intent. RootSpec extracts that wisdom into a structured, executable specification that bridges the gap between human purpose and implementation output.

---

## Design Pillars

### 1. Immediate Clarity

Visitors understand what RootSpec is and why it matters within seconds of landing on the page.

**User perspective:** "I immediately get it — this solves a problem I've actually had."

**Examples:**
- Tagline and one-sentence explanation visible above the fold
- No jargon, no buzzwords — language a developer would use with a colleague
- The meta-banner makes the demo nature immediately apparent

### 2. Earned Credibility

The site demonstrates the methodology by being a live product of it — not just describing what RootSpec does, but showing it.

**User perspective:** "This isn't just marketing — they actually built this using the process they're describing."

**Examples:**
- Meta-banner links directly to the spec and seed files
- Version badge showing the exact RootSpec version used
- Honest framing: rough edges are intentional, not careless

### 3. Approachable Depth

The methodology feels like something a developer could actually adopt — not an academic framework or a heavy process.

**User perspective:** "I could start using this today. It's not overwhelming."

**Examples:**
- Interactive wizard lets visitors try the methodology with their own idea
- Before/after comparison grounds the abstract in the familiar
- The hierarchy explorer makes the five levels tangible, not theoretical

### 4. Developer Respect

The site communicates like a thoughtful developer talking to peers — direct, honest, technically precise.

**User perspective:** "They're not selling me. They're showing me something they believe in."

**Examples:**
- No corporate language or buzzword-heavy copy
- Acknowledges trade-offs and limitations directly
- Confident without being preachy

---

## Inviolable Principles

- Never obscure the demo nature of the site — the meta-banner must always be visible and honest
- Never use lorem ipsum or placeholder content in interactive sections
- Never make external API calls — all interactivity is client-side
- Always link to real source files (spec, seed, GitHub) — transparency is core to the message
- Never sacrifice clarity for cleverness in copy

---

## North-Star Experience

A technical lead who has struggled with spec drift visits the site, immediately understands the problem being solved, tries the spec wizard with their current project idea, and closes the tab thinking: "This is exactly what I've been missing. I'm going to try this with my team this sprint." The site doesn't overwhelm them — it gives them just enough to understand the methodology and one clear path to start.
