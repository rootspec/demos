# L1: Philosophy

## Mission

To showcase RootSpec methodology by example through a marketing website that demonstrates purpose-aligned declarative specification in practice.

The site explains what RootSpec is, why it exists, and how to start using it, while itself being proof that the methodology works for real products.

## Design Pillars

### Proof by Example
Every claim about RootSpec is demonstrated by this site's own creation process.

### Transparent Process
Visitors understand immediately that this site was generated from minimal guidance using RootSpec's pipeline.

### Interactive Learning
Core concepts are made tangible through hands-on exploration rather than abstract descriptions.

### Developer-First Clarity
Technical depth without buzzwords; show concrete value to people who build software.

### Honest Limitations
Acknowledge what's possible with minimal input rather than overselling perfection.

## Inviolable Principles

- The meta banner must always be visible, explaining this site is a RootSpec demo
- All interactive features must work without external API calls
- The spec-to-implementation pipeline must be fully traceable
- Technical accuracy over marketing polish

## North Star

Six months from now: Developers visiting this site immediately understand both what RootSpec does and how it works. They can explore the methodology interactively, see real examples of the spec hierarchy, and walk away with enough knowledge to evaluate whether RootSpec fits their needs. The site serves as both effective marketing and a reference implementation that builds confidence in the framework.

## Market Position

**Positioning:** The methodology for AI-assisted development where philosophy guides implementation, not vice versa.

**Competing Against:**
- Specification tools that become documentation graveyards (linear specs nobody reads)
- Code generation without validation (unreliable AI output)  
- Ad-hoc requirement gathering (Google Docs and scattered conversations)
- Implementation-first development (where architecture emerges accidentally)

**Table Stakes:**
- Clear explanation of the methodology
- Working examples and demonstrations  
- Getting started instructions
- Integration with existing workflows
- Evidence that it scales to real projects

**Unique Value:**
- Declarative, executable specifications that act as validation gates
- Hierarchical structure prevents circular dependencies and philosophy drift
- Natural language approach (not config-first)
- Works with or without AI, any tooling, any team size
- Extracts and validates human intent rather than generating code blindly
