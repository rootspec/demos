# Level 1: Foundational Philosophy
# RootSpec Marketing Site

---

## Competitors

- **Notion** — Document-first workspace, used informally as a spec tool by many teams; no hierarchy enforcement
- **Linear** — Issue-tracking with lightweight spec support; treats specs as tickets, not as methodology
- **Confluence** — Wiki-style documentation; docs drift rapidly and are rarely consulted during build
- **GitHub Issues / Discussions** — Ad hoc requirement capture; no philosophy-to-implementation traceability
- **Spec-by-example tools (Cucumber/Gherkin)** — Test-first acceptance criteria without a broader specification philosophy

## Table Stakes

- Clear explanation of what RootSpec is and why it exists
- Links to documentation, GitHub repo, and installation instructions
- Interactive features that let visitors experience the methodology
- Mobile-responsive layout with dark/light theme
- Fast initial load — no blocking API calls

## Pain Points

- **Spec drift** — Traditional docs become stale the moment code diverges; no enforcement mechanism
- **Philosophy-implementation gap** — Teams know their mission but can't trace it to a ticket or test
- **Unreliable AI output** — AI generates code confidently but without any anchor to product intent
- **Unread Google Docs** — Specs exist as documents nobody returns to; they're written, not used
- **Circular dependencies** — Implementation details bleed upward into strategy, corrupting foundational decisions

## Influences

- **Stripe Docs** — Developer-facing precision without coldness; earns trust through clarity
- **Linear's product blog** — Confident, opinionated writing that respects the reader's intelligence
- **The Pragmatic Programmer** — Practitioner wisdom, not academic theory
- **Basecamp's writing style** — Honest, direct, unadorned by marketing language

---

## Mission

RootSpec exists to close the gap between what a product should be and what gets built — by giving teams a structured, executable specification that AI and humans can both validate against.

---

## Design Pillars

### 1. Earned Confidence
Users feel that the methodology is proven, not theoretical — they trust it because it's shown, not claimed.

**User perspective:** "I can see exactly how this works, and it matches problems I've actually had."

**Examples:**
- Every claim is backed by a concrete example or visual
- The site itself is built with RootSpec — the meta-demo is the proof

### 2. Intellectual Clarity
Users feel the concept click into place immediately — no jargon fog, no ambiguity about what to do next.

**User perspective:** "I understand this in under two minutes, and I know what to try first."

**Examples:**
- Hierarchy is shown visually before it's explained in text
- Interactive demos let users touch the concept, not just read it

### 3. Respectful Directness
Users feel spoken to as peers — technical, capable, and already burned by the problems being described.

**User perspective:** "This feels like advice from someone who has shipped real software."

**Examples:**
- No buzzwords, no corporate softening
- Pain points named plainly; no hedging about why existing tools fall short

### 4. Methodical Transparency
Users can see exactly how the site was made — the process is visible, not hidden behind polish.

**User perspective:** "I can inspect the spec, the seed, the code. Nothing is a black box."

**Examples:**
- Meta banner explains the build process with links to raw artifacts
- Version badge shows exactly which framework version generated this

---

## Inviolable Principles

- Never hide rough edges — the site is a demo of a real process, not a polished product illusion
- Always link to source artifacts (SEED.md, spec files, GitHub) — transparency is structural, not optional
- Never use buzzwords that obscure meaning ("synergy," "leverage," "scalable solutions")
- No external API calls at runtime — the site must work without any backend
- Philosophy must precede implementation in every decision — if a feature doesn't serve a Design Pillar, it doesn't ship

---

## North-Star Experience

A developer lands on this page after a frustrating sprint where requirements changed midway and the AI-generated code had to be thrown out. Within two minutes they understand what RootSpec is, why it's different, and how to start. They click through the hierarchy explorer and feel the concept click. They run the spec wizard and see their own product idea mapped to the methodology. They leave with a GitHub link open in another tab and a concrete first step in mind. They don't feel like they just read a product page — they feel like they just had a conversation with someone who has solved their problem.
