# Philosophy

## Mission

Create a marketing website that demonstrates RootSpec's power by being itself a proof-of-concept — generated from minimal human guidance with no manual code or design mockups.

RootSpec addresses the fundamental problem of specification drift in software development: when philosophy and implementation diverge, products lose coherence. We solve this by enforcing a strict hierarchy where higher levels guide lower levels, never the reverse.

## Design Pillars

These emotional experiences guide every design decision:

### IMMEDIATE_CLARITY
Visitors instantly understand what RootSpec is and why it matters.

### TANGIBLE_DEMONSTRATION
Interactive features make abstract concepts concrete and explorable.

### METHODICAL_CONFIDENCE
The systematic approach feels reliable, not academic or buzzwordy.

### AUTHENTIC_TRANSPARENCY
Honest about what you get from minimal guidance; no overselling.

### ACTIONABLE_NEXT_STEPS
Clear path from curiosity to implementation.

## Inviolable Principles

- **Philosophy drives implementation, never vice versa** — Every feature must trace to a design pillar
- **Show, don't tell** — Demonstrate methodology through working examples, not explanatory text
- **No external dependencies** — All interactive features run client-side to ensure reliability
- **Accessibility first** — Keyboard navigation and screen reader support are non-negotiable
- **Framework agnostic** — Content and concepts apply regardless of implementation technology

## North Star (6 months)

A developer visits the site after hearing about RootSpec in passing. Within 5 minutes, they understand the core concept through the interactive hierarchy explorer. They try the spec wizard with their own product idea and see how their input maps to structured levels. They view the honest meta banner explaining this site's own genesis and check out the linked spec files. By the time they leave, they're convinced that RootSpec could solve their team's specification drift problem and know exactly how to start — run `/rs-init` in their project. 

The site feels like a thoughtful developer explaining a methodology they believe in, not a corporate product page.

## Audience

Primary: Developers and technical leads who have experienced specification drift, unclear requirements, or unreliable AI-generated code. They value systematic approaches but are skeptical of academic solutions.

Secondary: Teams using AI development tools who want more reliable, validatable output.

## Competitive Positioning

**Against traditional specification tools:** We don't generate code — we extract and validate human intent first, then generate from validated intent.

**Against documentation platforms:** We're executable and testable, not just readable. The spec acts as a validation gate.

**Against AI code generators:** We solve the reliability problem by inserting human philosophy as a bridge between AI knowledge and implementation wisdom.

**Against heavyweight methodologies:** Natural language specification that works with any tooling, any team size, any level of AI assistance.
