# Level 1: Foundational Philosophy

## Competitors

- **Test-Driven Development (TDD)** — Constraint-based methodology that automates validation but creates brittle, implementation-coupled tests
- **Behavior-Driven Development (BDD)** — Natural language specs that drift from code over time; expensive to maintain
- **GitHub Copilot / Cursor** — AI code generation with no validation layer; amplifies drift rather than preventing it
- **Jira / Linear** — Issue tracking that fragments intent across tickets; specs live nowhere coherent
- **Notion / Confluence** — Freeform documentation that nobody reads; no executable link to implementation

## Table Stakes

- Clear explanation of what the methodology is and why it exists
- Demonstration of the five-level hierarchy and how it works
- Entry point to the framework: GitHub repo, npm package, getting-started docs
- Honest meta-framing: this site is itself a RootSpec demo

## Pain Points

- **Spec drift** — Requirements documents diverge from code; nobody knows what was intended
- **Philosophy-implementation gap** — By the time code ships, the original intent is unrecognizable
- **AI without validation** — Code generators produce plausible-looking output that satisfies no actual spec
- **Unread documentation** — Google Docs specs that developers ignore because they have no executable power
- **Solo-validator problem** — The person who defined intent is rarely the person who verifies the result

## Influences

- **Long-form essay sites** (Substack, The Browser) — Reading should feel like arriving at a place where someone thinks
- **Stripe / Linear docs** — Technical rigor with editorial clarity
- **TDD community** — The insight that code without proof is incomplete; adapted for intent rather than implementation
- **Star Trek replicator thought experiment** — Validation as the defining challenge of faithful reproduction

## Mission

RootSpec exists to keep human intent intact across the full software development lifecycle — from the first question to the last test — by making the specification itself the source of truth that validates everything downstream.

## Design Pillars

### 1. Intent Preserved
The methodology guarantees that the original human intent survives implementation, AI generation, team handoffs, and time.

**User perspective:** "What I meant to build is what got built."

**Examples:**
- Spec validated directly against running code, not maintained in parallel
- Philosophy levels remain stable when implementation changes

### 2. Proof Over Plausibility
Valid output is proven output. A passing test suite that doesn't trace to intent is not a proof — it's a coincidence.

**User perspective:** "I know this works because I can show why it works."

**Examples:**
- Every user story traces to a design pillar
- Validation is a gate, not a ceremony

### 3. Approachable Rigor
The methodology is rigorous enough to catch drift and approachable enough that developers actually use it. Complexity is earned, never imposed.

**User perspective:** "This is a process I can actually follow without resenting it."

**Examples:**
- Four commands drive the entire workflow
- Natural language spec, not YAML-first configuration

### 4. Human Philosophy, AI Execution
Humans supply the intent — the why, the values, the trade-offs. AI handles the downstream work. The spec is the bridge that makes this division reliable.

**User perspective:** "AI does the tedious parts; I remain responsible for what matters."

**Examples:**
- Interview phase extracts latent intent before any code is written
- Implementation agents operate within validated constraints

## Inviolable Principles

- Never let implementation details bleed upward into philosophy
- Never treat validation as optional — it is the mechanism that makes everything else meaningful
- Never generate code without a spec to validate against
- The site must be honest about being a demo: the meta-banner is not optional
- No buzzwords, no corporate tone, no startup-pitch aesthetics — the site's quality is the argument

## North-Star Experience

A developer who has been burned by scope drift discovers this site, reads it in one sitting as if it were a well-written essay, uses the interactive hierarchy explorer to understand the five levels in under two minutes, runs `/rs-spec` on their own product idea from the wizard output, and feels — for the first time — that specification is something they want to do rather than something imposed on them.
