# Level 1: Foundational Philosophy

## Competitors

- **Traditional BDD/TDD tooling** — Behavior- and test-driven frameworks (Cucumber, RSpec, Jasmine) that enforce testing discipline but require manual authoring of specs and tests separately from intent
- **AI code generators** — Copilot, Cursor, and similar tools that generate code from natural language but have no mechanism for validating alignment with original intent
- **Project management / spec tools** — Linear, Notion, Confluence, Jira — places where "the spec" lives as a document that drifts from the codebase over time
- **Formal specification languages** — TLA+, Alloy, and similar tools used in safety-critical systems; precise but inaccessible outside specialized engineering contexts

## Table Stakes

- A way to capture human intent about what a product should be and why
- A connection between that intent and the working code
- Enough structure for team collaboration across the development lifecycle
- Works with modern development tooling (version control, CI/CD, AI assistants)

## Pain Points

- **Spec drift** — Documents capturing intent fall out of sync with the code; by the time validation happens it is far removed from the original intent
- **Implicit assumptions** — The intent author holds internal assumptions and biases; unstated expectations trigger mid-cycle pivots and quiet disconnects with end users
- **TDD/BDD overhead** — Existing constraint-based approaches are expensive and tedious: they transform a few pages of specification into a book of pedantic, finely-tipped instructions; they force meticulous back-and-forth between dev and QA; they turn problem-solving into rote assembly of brick-like functions
- **Unreliable AI output** — AI has knowledge but not wisdom; without a validated spec as a guardrail, AI-generated code satisfies the literal prompt rather than the underlying human intent
- **Philosophy-implementation gap** — No clear hierarchy prevents low-level implementation decisions from corrupting high-level principles; philosophy drifts when implementation changes

## Influences

- **Long-form editorial writing** — The thoughtful Substack, the literary magazine: a place where someone thinks, not a place where someone sells
- **Stripe, Linear, Vercel docs** — Documentation that feels considered: clean typography, generous whitespace, functional UI
- **Test-driven development** — The philosophical commitment that code doesn't exist without proof that it works as intended
- **Declarative systems** — CSS, SQL, infrastructure-as-code: define the desired state, let the system figure out how to achieve it

## Mission

RootSpec exists to carry human intent intact from the first question to the last test — giving developers and teams a way to define what their product is and why it exists, then prove that everything built from that point forward remains in alignment.

## Design Pillars

### 1. Intent Is the Asset
Users feel that what they wrote at the beginning is what was built at the end — the spec is the source of truth, not a description of it.

**User perspective:** "I said what I meant, and I can prove the software does what I said."

**Examples:**
- The spec validates the code — not the other way around
- Changes to the spec cascade downstream; changes to code propagate upstream as signals to revisit the spec
- Every feature traces to a design pillar; every test traces to a user story; every user story traces to the spec

### 2. Philosophy Before Implementation
Users feel confident that pivots and rewrites do not erode the original purpose because foundational decisions were locked in before implementation began.

**User perspective:** "I can completely change how it's built without losing what it's for."

**Examples:**
- Five levels, each only able to reference higher levels — never lower
- Refactoring L4 systems leaves L1-L3 untouched
- The spec can be written before a single line of code exists

### 3. Validation Is Everything
Users feel that shipping is not a gamble but a verified act — because the methodology produces proof, not approximations.

**User perspective:** "I know it works because I can show it works."

**Examples:**
- User stories are the tests; the story is the test
- Only verified solutions pass through; risk plummets
- Validation gates the entire workflow

### 4. Approachable Without Being Simplified
Users feel capable, not overwhelmed — like a thoughtful colleague explained something they already half understood.

**User perspective:** "I get it, and I can start using it today."

**Examples:**
- Natural language specification, not config or YAML-first
- Four commands drive all workflows
- The interactive site demonstrates the methodology by being the methodology

## Inviolable Principles

- Never sacrifice philosophical clarity for implementation convenience — the hierarchy flows one direction only
- Never position RootSpec as a blocker or gatekeeper; it identifies problems, it does not stop work
- Never use buzzwords or corporate abstraction language; speak like a thoughtful developer, not a product marketing team
- Never treat the spec as separate from the codebase; they are one artifact
- The site's visual quality is part of the argument — a polished aesthetic signals the methodology produces real results

## North-Star Experience

Six months from now, a developer who found RootSpec through the demo site is using it on a team project. When a new engineer joins, they read the spec top to bottom and understand exactly what was built and why. When a stakeholder asks "why does it work this way?", the answer traces cleanly from the code to a user story to a design pillar. When an AI agent writes implementation, it is constrained by the spec and its output is verified against it. The spec and the codebase move together, and the original intent is still recognizable in the shipped product.
