# Level 1: Foundational Philosophy

## Competitors

- **Traditional spec tools (Jira, Linear, Notion)** — Task and doc management with no philosophical hierarchy; specs drift from intent as soon as they're written
- **Test-driven development frameworks** — Enforce constraints but require expensive manual maintenance; tedious to keep aligned as products evolve
- **AI code generators (GitHub Copilot, Cursor)** — Generate code fluently but have no mechanism to validate against original intent; drift is faster, not slower
- **BDD tools (Cucumber, Gherkin)** — Structured behavior specs but transform philosophy into pedantic, brittle test scripts; expensive to maintain

## Table Stakes

- A structured way to capture product intent before writing code
- Some mechanism to validate that implementation satisfies the original intent
- Works alongside existing tooling (any language, any framework, any team size)
- Clear entry points for developers who want to start quickly

## Pain Points

- **Spec drift** — Documentation is written once and immediately starts diverging from what gets built
- **Philosophy-implementation gap** — The "why" of a product is lost by the time engineers are writing code; decisions are made without reference to original intent
- **AI output unreliability** — AI tools generate plausible code that may not satisfy the actual requirements; there is no validation loop
- **Unread specs** — Google Docs and Notion pages that nobody looks at after the kickoff meeting; specs become archaeological artifacts
- **Expensive maintenance** — TDD and BDD produce brittle test suites that break constantly and require dedicated QA cycles to maintain

## Influences

- **Long-form essay sites (Substack, The Browser)** — Editorial quality signals seriousness of thought; writing is the primary product
- **Stripe and Linear docs** — Technical but human; documentation as product experience
- **TDD principles** — Tests as specification, code as proof — but executed at the philosophy level, not the function level
- **The Star Trek replicator problem** — How do you prove you've built the right thing? Validation is the answer, not generation

## Mission

RootSpec exists because generating code is now trivial and proof is what matters. It captures human intent at the philosophical level and carries it intact through every stage of development — from the first question to the last test.

## Design Pillars

### 1. Validated Confidence
Users feel certain that what they've built actually satisfies their original intent — not just that it compiles and runs.

**User perspective:** "I know this is right, not just that it works."

**Examples:**
- Spec hierarchy traces every feature back to a design pillar
- User stories run as actual tests; the story is the test
- Validation gates the workflow; only verified solutions pass through

### 2. Clarity of Thought
Users feel the methodology helping them think more clearly about their product — surfacing assumptions, exposing contradictions, forcing precision without demanding formality.

**User perspective:** "This helped me understand what I actually wanted."

**Examples:**
- The interview phase asks "why" before "what"
- Philosophy level stays stable even as implementation evolves
- Trade-off choices are explicit, not implied

### 3. Unhurried Mastery
Users feel the depth and rigor of the methodology without feeling overwhelmed — the entry point is approachable, the ceiling is high.

**User perspective:** "I could start in five minutes and still be learning six months later."

**Examples:**
- A one-line seed is enough to begin
- Five commands cover the entire workflow
- The spec grows through use, not upfront investment

### 4. Human Authority
Users feel that the methodology amplifies their judgment rather than replacing it — AI executes; humans supply wisdom.

**User perspective:** "I'm still the one who decides what matters."

**Examples:**
- Philosophy level is always human-authored, never AI-generated by default
- AI asks questions; humans answer them
- The spec is the human's voice; the code is the AI's execution

## Inviolable Principles

- Never let implementation details influence philosophical foundations — the hierarchy flows one direction only
- Never replace human judgment with AI inference at the philosophy level — AI surfaces options, humans decide
- Never treat validation as a phase — it is the continuous thread connecting intent to output
- Never obscure the methodology behind tooling complexity — five commands, no black boxes
- The spec is always readable by a non-engineer — if it isn't, it has drifted toward implementation

## North-Star Experience

A technical lead at a ten-person startup opens RootSpec for the first time at 9am. By lunch, they have a validated spec for their product's core loop — one that their whole team can read, that traces every feature to a stated philosophy, and that runs as actual acceptance tests. The next morning, their AI coding agent produces a working implementation that passes every test on the first try. The lead didn't write a single line of code. They wrote what they believed, and the system built it.
