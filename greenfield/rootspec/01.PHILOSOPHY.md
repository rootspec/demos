# Level 1: Foundational Philosophy — WHY & WHAT EXPERIENCE

## Competitors

- **Traditional spec tools (Confluence, Notion docs)** — Unstructured prose that drifts from implementation; nobody reads them
- **Code generators (GitHub Copilot, v0, Cursor)** — Produce code without validating intent; generate confidently wrong things
- **Formal modeling tools (Alloy, TLA+)** — Too academic, steep learning curve, disconnected from working teams
- **User story trackers (Jira, Linear)** — Track work items but don't enforce philosophical coherence or hierarchy

## Table Stakes

- Specification must be human-readable, not just machine-parseable
- Works with any technology stack and any team size
- Integrates with AI-assisted development workflows
- Produces testable, verifiable output

## Pain Points

- **Spec drift** — Specifications decay as code evolves; nobody keeps docs in sync
- **Philosophy-implementation gap** — Foundational decisions get lost; new features contradict original intent
- **Unreliable AI output** — AI tools generate plausible-but-wrong solutions when there's no validation layer
- **Google Docs graveyard** — Spec documents exist but have no executable weight; they can't catch violations

## Influences

- **Dependency inversion principle** — High-level policy should not depend on low-level details
- **Architecture Decision Records (ADRs)** — Capturing "why" alongside "what"
- **Behavioral-driven development (BDD)** — Making specifications executable

## Mission

RootSpec exists to close the gap between human intent and AI-assisted implementation — giving developers a methodology where philosophy guides implementation, never the reverse.

## Design Pillars

### 1. Philosophy as Foundation
Users feel their intent is preserved and protected — every product decision traces to a human-defined purpose, not accumulated shortcuts.

**User perspective:** "I can look at any feature and trace it back to why we built this product."

**Examples:**
- Spec hierarchy enforces that mission informs systems, never the reverse
- Violations surface explicitly rather than accumulating silently

### 2. Validation as Liberation
Users feel confident that what they've built matches what they intended — a validation gate replaces hope as the quality mechanism.

**User perspective:** "I can accept AI suggestions because I have a validation gate, not just hope."

**Examples:**
- Specs act as executable contracts, not aspirational documents
- Test generation flows from intent, not from reverse-engineering code

### 3. Approachable Without Being Simple
Users feel that structure is helping them, not constraining them — the methodology is learnable by real teams under real time pressure.

**User perspective:** "This is how I already think — the framework just makes it explicit."

**Examples:**
- Natural language specification (not config or YAML-first)
- Four commands cover the entire workflow

### 4. Human Wisdom, AI Execution
Users feel in control of what matters while offloading what doesn't — humans supply philosophy, AI executes implementation.

**User perspective:** "The AI is a capable junior engineer who builds what I've specified, not a black box that decides what I meant."

**Examples:**
- `/rs-spec` extracts and validates human intent before any implementation begins
- The spec is a bridge, not a bottleneck

## Inviolable Principles

- Never generate code or output without a validated spec backing it — the spec comes first, always
- Never treat a specification as complete if it can't be traced back to a mission or design pillar
- Never create a validation process that blocks work — RootSpec identifies problems, it doesn't gatekeep
- Philosophy levels must remain stable when implementation changes; if philosophy needs to change, that's a deliberate product decision

## North-Star Experience

Six months from now, a developer using RootSpec leads a team building a complex product with AI assistance. When a new AI-generated feature is proposed, the team runs `/rs-validate` and gets a clear report: "This feature contradicts Design Pillar 2 and is missing a system boundary in L4." They update the spec intentionally, the feature is revised, and it ships. No lengthy review meetings. No "wait, why did we build it this way?" post-mortems. The spec answered those questions before the code was written.
