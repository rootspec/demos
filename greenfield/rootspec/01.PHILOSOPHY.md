# Level 1: Foundational Philosophy

**Product:** RootSpec Marketing Site
**Level:** L1 — WHY & WHAT EXPERIENCE
**References:** External only

---

## Competitors

- **Traditional PRDs** — Google Docs, Notion pages, Confluence wikis that drift out of date and are ignored by developers
- **User Story Tools** — Jira, Linear, Shortcut: task trackers that capture work items without philosophical grounding
- **Code Generators** — GitHub Copilot, v0, Bolt: produce code but cannot validate intent or enforce traceability
- **Spec-by-example tools** — Cucumber, Gherkin: test DSLs without a hierarchy linking tests to product philosophy

## Table Stakes

- Clear explanation of what RootSpec is and why it exists
- Demonstration of the five-level specification hierarchy
- Concrete examples of how the methodology works
- Links to documentation, source code, and getting-started resources

## Pain Points

- **Spec drift** — Requirements docs become outdated the moment code ships; nobody trusts them
- **Philosophy-implementation gap** — Engineers build features that technically satisfy stories but miss the original intent
- **AI unreliability** — Language models have knowledge but not wisdom; they generate plausible output that still violates product intent
- **Untraceable decisions** — When requirements live in Slack threads and meeting notes, nobody can explain why a feature works the way it does
- **Gatekeeping fatigue** — Traditional spec processes block progress rather than catching real problems

## Influences

- **The Unix philosophy** — Small, composable tools with clear boundaries
- **Domain-driven design** — Structure reflects the problem domain, not the tech stack
- **Literate programming** — Specification and implementation as a unified, readable artifact
- **Agile retrospectives** — Validation as a continuous feedback loop, not a waterfall gate

## Mission

To demonstrate that when code and specifications can be generated trivially, the real value is validation and proof — and that humans supply philosophy while AI executes implementation, with a spec as the bridge.

## Design Pillars

### 1. Intellectual Honesty
Visitors encounter a site that tells the truth about what it is — a minimal-guidance demo with visible rough edges — rather than a polished facade hiding the process.

**User perspective:** "I trust what I'm reading because it doesn't oversell itself."

**Examples:**
- The meta banner is prominent and candid about how the site was built
- Rough edges are contextualized, not hidden

### 2. Methodological Clarity
The five-level hierarchy feels intuitive and learnable through direct exploration, not through reading documentation.

**User perspective:** "I understand how this works by interacting with it, not by reading about it."

**Examples:**
- The hierarchy explorer makes level relationships visible
- The spec wizard walks through the methodology in minutes

### 3. Confident Accessibility
The methodology is technical but never academic — presented with the directness of a developer who has thought hard about a problem and found a better approach.

**User perspective:** "This was made by someone who has actually shipped software, not a consultant selling process."

**Examples:**
- Tone is direct, not corporate
- Examples use real content, not lorem ipsum

### 4. Earned Trust
Every claim is backed by evidence the visitor can examine: the spec itself, the seed, the source code.

**User perspective:** "I can verify everything they're telling me."

**Examples:**
- Links to the actual spec and seed files on GitHub
- Version badge shows which framework version built the site

## Inviolable Principles

- Never hide how the site was built — the meta banner is non-negotiable
- Never use lorem ipsum or placeholder content — show real methodology content
- Never add buzzwords or corporate language — the tone must remain direct and developer-authored
- The site itself must demonstrate the methodology it explains
