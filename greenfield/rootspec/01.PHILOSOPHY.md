# L1: Philosophy

## WHY — Mission Statement

RootSpec exists because software specification has become either trivial (copy-paste docs) or impossible (the spec that nobody reads). As AI tools make implementation nearly free, the real challenge shifts from "how to build" to "how to validate what was built against human intent."

RootSpec implements purpose-aligned declarative specification where philosophy guides implementation, never vice versa. When code and specs can be generated trivially, the real value is validation and proof.

## WHAT EXPERIENCE — Design Pillars

### Clarity Over Convenience
Users should immediately understand what RootSpec is and why they need it. No cognitive overhead, no buzzwords, no hiding complexity behind marketing speak.

### Hierarchy Over Chaos  
The strict five-level hierarchy (Philosophy → Truths → Interactions → Systems → Implementation) creates order. Users feel confident that every feature traces back to a design pillar.

### Validation Over Generation
Users feel empowered, not replaced. RootSpec identifies problems and validates intent rather than generating solutions. AI executes; humans supply wisdom.

### Demonstration Over Documentation
Users see RootSpec working, not just described. Interactive elements and real examples prove the methodology works rather than asking users to take it on faith.

### Honesty Over Perfection
Users understand this site itself is a demo with minimal human input. Rough edges are acknowledged upfront as the result of sparse guidance, not carelessness.

## Inviolable Principles

### Philosophy Stability
Implementation can change, but mission and design pillars remain constant. Changes flow downward only - never upward in the hierarchy.

### No Circular Dependencies
Each level can only reference higher levels, never lower. This prevents circular reasoning and keeps the specification coherent.

### Declarative Not Imperative
The spec describes what the product is, not how to build it. Implementation details live in Level 5 user stories, not scattered throughout.

### Human Intent Preservation
AI has knowledge but not wisdom. The spec bridges the gap between human philosophy and AI execution by maintaining traceability from every feature back to design pillars.

## North Star

Six months from now, developers open RootSpec when starting any complex project. They spend 30 minutes defining philosophy and truths, then let the spec guide every implementation decision. When AI suggests a feature, they check: "Does this trace to a design pillar?" When stakeholders request changes, they ask: "How does this affect our trade-offs?"

The spec becomes the single source of truth for product direction - not a document that gets written once and forgotten, but the foundation that makes every other decision clear and defensible.
