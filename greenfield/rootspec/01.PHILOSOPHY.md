# Level 1: Foundational Philosophy

## Competitors

- **Traditional spec tools (Confluence, Notion docs)** — Freeform documents that drift from code; no enforced structure, no traceability
- **BDD frameworks (Cucumber, Gherkin)** — Test-focused, bottom-up; no philosophy layer, no hierarchy enforcement
- **OpenAPI / JSON Schema** — Machine-readable specs for APIs only; no product intent, no experience layer
- **Storybook / Design Tokens** — UI-component specs; don't capture why or how systems relate
- **AI code generators (GitHub Copilot, Cursor)** — Generate implementation without validation; no wisdom layer

## Table Stakes

- A way to capture product intent in structured, human-readable form
- Traceability from high-level goals to testable implementation details
- Version control-friendly format (plain text, diffable)
- Works with any tech stack or team size
- Supports AI-assisted development workflows

## Pain Points

- **Spec drift** — Written specs go stale; code diverges without anyone noticing
- **Philosophy-implementation gap** — Devs implement without understanding why; decisions get made in isolation
- **Unreliable AI output** — AI generates code without validation against product intent; confident but wrong
- **Google Docs specs nobody reads** — Freeform documents are unsearchable, untraceable, and ignored
- **Circular dependencies** — Implementation decisions contaminate philosophy; product loses coherence over time

## Influences

- **Declarative configuration (Terraform, k8s)** — Define desired state; tooling figures out how to get there
- **Type systems** — Catch errors at the right level of abstraction, not at runtime
- **Domain-Driven Design** — Bounded contexts, ubiquitous language, philosophy before structure
- **Agile user stories** — Human-centered, testable units of value — but extended upward to mission and design

## Mission

RootSpec must exist because AI can generate code but not wisdom — humans supply the philosophy, and the spec bridges the gap between intent and implementation, making validation automatic and spec drift impossible.

## Design Pillars

### 1. Clarity Before Cleverness
Users feel immediately oriented — what RootSpec is, why it matters, and how to start are obvious within seconds of landing.

### 2. Show, Don't Tell
Visitors experience the methodology firsthand through interactive tools, not through abstract descriptions of what it does.

### 3. Honest Demo Transparency
Rough edges are owned, not hidden — visitors see exactly what a minimal seed produces.

### 4. Approachable Rigor
Developers feel invited into a principled methodology, not lectured at by an academic framework.

### 5. Technical Credibility
Every claim is grounded — precise language, real examples, and correctly behaving tools earn trust with skeptical developers.

## Inviolable Principles

- Never claim features RootSpec doesn't have
- Never hide or apologize for rough edges — own them as evidence of the methodology
- The site must itself pass RootSpec validation — no spec drift allowed in the demo
- No external API calls — all interactive features run client-side
- Every link to spec files must use absolute GitHub URLs, never relative paths

## North-Star Experience

A developer lands on the site, skeptical of another spec tool. Within two minutes, they've clicked through the Hierarchy Explorer, run the Spec Wizard with their own idea, and seen the before/after comparison. They leave thinking: "This is different — it's not about generating docs, it's about keeping humans accountable for philosophy while AI handles the rest." They open a new tab and go to the GitHub repo.
