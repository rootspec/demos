# Level 1: Foundational Philosophy

**Version:** 7.3.2
**Status:** Draft

---

## Competitors

- **Traditional spec tools (Confluence, Google Docs)** — Unstructured documents that drift from reality, rarely consulted after writing
- **AI code generators (GitHub Copilot, Cursor)** — Generate code quickly but lack validation that output aligns with intent
- **BDD frameworks (Cucumber, SpecFlow)** — Test-first but not philosophy-first; don't address the WHY behind requirements
- **Requirements management tools (Jira, Linear)** — Track work but don't encode decisions or prevent spec drift

## Table Stakes

- Clear explanation of what the methodology is and why it exists
- Actionable path to getting started (four skills, one command each)
- Interactive demonstration of core concepts
- Evidence that the methodology works (this site as living proof)
- Open source access and community links

## Pain Points

- **Spec drift** — Specs written at project start become obsolete; teams work from memory, not documentation
- **Philosophy-implementation gap** — Code reflects developer convenience, not product intent
- **Unreliable AI output** — AI tools have knowledge but not wisdom; without philosophical grounding, AI-generated code drifts
- **Unread documentation** — Google Docs specs are written once, ignored forever; they don't act as validation gates
- **Circular dependencies** — Implementation details influence philosophy, leading to rationalized decisions rather than principled ones

## Influences

- **Unix philosophy** — Simple, composable tools with clear responsibilities
- **Declarative programming** — Specify what, not how; outcomes over procedures
- **Domain-Driven Design** — Ubiquitous language, bounded contexts, philosophy that guides architecture
- **Test-Driven Development** — Validation as a first-class concern, not an afterthought

## Mission

RootSpec exists to close the gap between human intent and software reality — giving developers a structured methodology to extract, encode, and enforce product philosophy so AI tools and teams produce systems that actually match what was meant.

## Design Pillars

### 1. Intellectual Clarity
Users feel the fog lift — complex ideas become structured, navigable, and actionable.

**User perspective:** "I finally understand how my product decision relates to the code I'm writing."

**Examples:**
- Five-level hierarchy makes abstract philosophy concrete and traceable
- Each spec level answers a single, clear question

### 2. Earned Confidence
Users feel validated, not judged — the spec proves alignment rather than policing compliance.

**User perspective:** "I know my implementation matches my intent, and I can prove it."

**Examples:**
- Validation reports show what passes, not just what fails
- The spec acts as a gate that identifies problems, not blocks work

### 3. Approachable Rigor
Users feel the methodology is serious without being academic — rigorous enough to matter, approachable enough to use.

**User perspective:** "This is how I actually think about my product, just made explicit."

**Examples:**
- Natural language specification, not config-first
- Interactive features let visitors experience the methodology before adopting it

### 4. Methodological Trust
Users feel the framework is honest about tradeoffs — it works with or without AI, any tooling, any team.

**User perspective:** "This isn't magic or hype — it's a discipline that scales with my needs."

**Examples:**
- Transparent about what RootSpec does and doesn't solve
- Demo site itself built with the methodology (meta-proof)

## Inviolable Principles

- Never present spec compliance as bureaucracy — it is the mechanism by which philosophy survives implementation
- Never suggest the tool replaces human judgment — humans supply philosophy; the framework preserves it
- Always show, don't tell — the site itself is the primary demonstration of the methodology
- Never obscure rough edges of the demo — the meta-banner must be honest about what minimal guidance produces
- The spec hierarchy flows downward only — implementation never reaches back to change philosophy
