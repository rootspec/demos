# Level 1: Foundational Philosophy — WHY & WHAT EXPERIENCE

## Competitors

- **Traditional spec tools (Confluence, Notion docs)** — Static documentation that drifts from implementation; nobody reads them; no validation mechanism
- **BDD frameworks (Cucumber, Behave)** — Test-centric, bottom-up; forces developers into verbose Gherkin without connecting to product philosophy
- **Product roadmap tools (Productboard, Aha!)** — Strategic planning without technical traceability; philosophy and code remain disconnected
- **AI code generators (Copilot, Cursor)** — Fast output, unreliable intent; produces code without validating that it reflects human purpose
- **Architecture decision records (ADRs)** — Useful but scattered; no hierarchy enforcing that decisions trace to philosophy

## Table Stakes

- Ability to define product intent in natural language
- Structured hierarchy connecting purpose to implementation
- Validation that implementation reflects specification
- Works with standard developer tooling (CLI, version control)
- Accessible to non-specialists without sacrificing rigor

## Pain Points

- **Spec drift** — Requirements documents written once and never updated; code diverges silently from stated intent
- **Philosophy-implementation gap** — Teams know the "what" but lose the "why"; decisions lack principled grounding
- **Unreliable AI output** — AI has knowledge but not wisdom; without a validation gate, generated code may be syntactically correct but philosophically wrong
- **Google Docs specs** — Living in the wrong medium; no traceability, no tests, no enforcement
- **Circular reasoning** — Implementation choices creeping up to "inform" philosophy, reversing the dependency

## Influences

- **Dependency inversion principle** — High-level policy should not depend on low-level details; this principle applied to product specification
- **Declarative configuration** — Define desired state, not procedure; the spec describes what should be true, not how to achieve it
- **Test-driven development** — Write tests before code; specification-driven development writes philosophy before tests
- **Unix philosophy** — Small, composable tools doing one thing well; four commands, clear boundaries

## Mission

RootSpec exists because philosophy should guide implementation — never the reverse. When code and specs can be generated trivially, the real value is validation and proof that what was built reflects what was meant.

## Design Pillars

### 1. Wisdom Over Knowledge
The spec captures human intent — the "why" — that AI cannot derive from context alone. Users feel that their purpose is preserved, not just their requirements.

**User perspective:** "My product's philosophy is protected even when implementation changes rapidly."

**Examples:**
- Hierarchy enforces that L1 (philosophy) never references L5 (implementation)
- Validation surfaces when code diverges from stated intent, not just when it breaks

### 2. Clarity Without Ceremony
The methodology is rigorous but not bureaucratic. Users feel productive and unencumbered — spec work feels like focused thinking, not paperwork.

**User perspective:** "I can express complex product intent without learning a new language or filling out forms."

**Examples:**
- Natural language at all levels; YAML only at L5 for machine-readability
- Four commands handle the entire workflow: no sprawling toolchain

### 3. Traceable Trust
Every feature connects to a design pillar. Users feel confident that what shipped reflects what was intended, and can prove it.

**User perspective:** "I can show exactly why this feature exists and which principle it serves."

**Examples:**
- User stories reference the systems they test
- Validation fails when stories lack system references or pillars lack substance

### 4. Human-AI Partnership
Humans supply philosophy; AI executes implementation. The spec is the bridge. Users feel in control of AI-assisted development rather than dependent on it.

**User perspective:** "AI amplifies my intent rather than replacing my judgment."

**Examples:**
- Spec acts as a validation gate for AI-generated code
- The methodology works with or without AI; AI accelerates but doesn't own the philosophy

## Inviolable Principles

- Never allow lower levels to define or constrain higher levels (dependency inversion is non-negotiable)
- Never sacrifice natural language expressiveness for machine-readability at L1–L3
- Always make validation results actionable — report what's wrong and where, not just that something failed
- Never block work — identify problems, surface them clearly, let teams decide what to do

## North Star Experience

Six months from now, a developer leads a complex product through a significant pivot. The product philosophy stays intact because it was written down and validated — not because someone remembered it. They run `/rs-validate`, see which user stories would be violated by the proposed change, present the tradeoff clearly to stakeholders, and make a principled decision. The spec didn't prevent the pivot; it made the cost visible. AI generated the implementation; the human validated the intent. That's the experience this product must deliver.
