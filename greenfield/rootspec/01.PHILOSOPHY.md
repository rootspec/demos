# Level 1: Foundational Philosophy

**Product:** RootSpec Marketing Site
**Purpose:** Explain what RootSpec is, why it exists, and how to start using it — while demonstrating the methodology in practice.

---

## Competitors

- **Linear / Jira** — Issue trackers that manage work but don't enforce specification integrity
- **Notion / Confluence** — Flexible docs that serve as specs, but drift from implementation freely
- **GitHub Copilot / Cursor** — AI coding tools that generate code but lack validation against human intent
- **Swagger / OpenAPI** — Machine-readable specs for APIs, not human-intent-first product specs
- **Storybook** — Component-level documentation, not product-level philosophy alignment

## Table Stakes

- Clear explanation of what RootSpec is and why it matters
- Demonstration of the five-level hierarchy in an understandable way
- Easy path to getting started (links, commands, GitHub)
- Professional, trustworthy presentation appropriate for a developer audience
- Works on mobile and desktop without degradation

## Pain Points

- **Spec drift** — Existing docs (Google Docs, Notion pages) diverge from implementation over time with no enforcement mechanism
- **AI without wisdom** — LLMs generate code confidently but have no access to human intent or product philosophy
- **Untraceable decisions** — Features accumulate without connection to original design pillars, making refactoring dangerous
- **Gatekeeping tools** — Most "spec" tools block progress instead of identifying problems early
- **YAML-first specs** — Config-driven specifications require technical fluency before expressing intent

## Influences

- **Stripe Docs** — Technical precision without condescension; developer-first voice
- **Linear's philosophy pages** — Confident, direct language about methodology without being preachy
- **Basecamp's Shape Up** — Explaining a methodology through concrete examples and honest trade-offs

---

## Mission

RootSpec exists because AI has knowledge but not wisdom. When humans supply philosophy and AI executes implementation, a structured bridge is needed — one that validates output against intent, not just syntax. RootSpec is that bridge: a specification language where philosophy guides implementation, never vice versa.

---

## Design Pillars

### 1. Earned Clarity

Users feel the methodology clicking into place — not through abstract explanation but through direct experience of the hierarchy.

**User perspective:** "I understand what RootSpec does because I felt it work, not just because I read about it."

**Examples:**
- The Hierarchy Explorer lets visitors expand levels and feel the dependency constraints
- The Spec Wizard produces a real skeleton spec from a visitor's own idea

### 2. Confident Credibility

Users feel they're reading from someone who has thought deeply and solved a real problem.

**User perspective:** "This person has been burned by the same problems I have. I trust their solution."

**Examples:**
- Honest meta banner acknowledging this site was generated from a seed with minimal iteration
- Before/after comparison uses real content, not lorem ipsum
- Tone is that of a thoughtful developer, not a corporate product page

### 3. Approachable Depth

Users feel the methodology is sophisticated enough to take seriously but accessible enough to start immediately.

**User perspective:** "This is serious thinking, and I can actually try it right now."

**Examples:**
- Interactive wizard lets visitors apply the methodology to their own idea in minutes
- Four skills (init/spec/impl/validate) are concrete and actionable, not abstract

### 4. Demonstrable Proof

Users feel the site itself is evidence that RootSpec works — the medium is part of the message.

**User perspective:** "The fact that this site exists as it does proves the point."

**Examples:**
- Meta banner displays how the site was built: seed → spec → impl → validate
- Version badge shows which RootSpec version the site was built with
- Links to the actual spec files let skeptics audit the claim

---

## Inviolable Principles

- **Never use buzzwords** — No "AI-powered," "game-changing," "paradigm shift," or similar language
- **Never be preachy** — State the methodology, show it, let visitors decide
- **Always be honest about rough edges** — The meta banner must acknowledge limitations, not hide them
- **Never make external API calls** — All interactivity runs client-side; no third-party dependencies at runtime
- **Philosophy first, implementation second** — Every design decision on this site must trace to one of the four pillars above

---

## North-Star Experience

Some weeks after visiting, a developer who was skeptical about specification tooling returns while debugging a drift problem on their team. They'd understood the hierarchy, tried the wizard with their own product idea, and opened the GitHub repo. They don't feel sold to — they feel understood. The site's honest acknowledgment of its own limitations made them trust it more, not less.
