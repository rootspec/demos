# Level 1: Foundational Philosophy
# RootSpec Marketing Site

## Competitors

- **Linear Docs / Notion** — General documentation tools used as spec-writing environments; unstructured, no hierarchy enforcement
- **Confluence** — Wiki-style specs that drift from code; philosophical intent buried in meeting notes
- **GitHub Issues** — Requirement tracking without philosophical grounding; stories exist in isolation
- **Storybook / Chromatic** — UI component documentation tools, not product specification systems
- **Blueprint / RFC systems** — Engineering-focused; miss the human philosophy layer entirely

## Table Stakes

- Explain what RootSpec is within seconds of landing
- Show the five-level hierarchy clearly
- Provide a path to get started (install, commands)
- Link to the GitHub repo and source materials
- Work on mobile and desktop

## Pain Points

- **Spec drift** — Docs describe what was planned, not what exists; nobody trusts them
- **The Google Doc spec** — Exists, nobody reads it, disappears when someone leaves
- **AI without grounding** — AI tools generate code without understanding product intent; produces locally correct but globally wrong output
- **Philosophy-implementation gap** — The why lives in someone's head; the how lives in the code; they are never reconciled
- **No validation gate** — Specs are advisory, not executable; teams can't tell if they've built what was intended

## Influences

- **Plain language writing** — Strunk & White clarity: direct, no padding, respect the reader's time
- **Thoughtful developer documentation** — The Stripe docs tone: technical but human, example-first
- **Methodology sites like Atomic Design** — Shows how to explain a system visually without talking down to developers

## Mission

RootSpec exists because software teams lose the thread. The why gets buried, the spec drifts, and AI tools amplify confusion at scale. RootSpec bridges philosophy and implementation through a strict hierarchy that keeps human intent in control — not as constraint, but as compass.

## Design Pillars

### 1. Clarity Over Cleverness
Visitors understand what RootSpec is and why it matters without needing to decode marketing language.

**User perspective:** "I immediately understood what this does and whether I need it."

**Examples:**
- Plain language explanations over buzzwords
- Real content, not lorem ipsum, in all examples
- Every section answers a concrete question, not a vague theme

### 2. Show, Don't Tell
The site demonstrates the methodology rather than merely describing it.

**User perspective:** "The site itself proved that RootSpec works — I could see the spec, the output, and the gap between them."

**Examples:**
- Meta banner explaining the site was generated from a sparse seed
- Interactive hierarchy explorer that makes the spec tangible
- Before/after comparison with actual content

### 3. Earned Trust
Transparency about limitations and rough edges builds credibility more than polish.

**User perspective:** "They were honest about what this produces from minimal input — that made me trust it more."

**Examples:**
- Meta banner acknowledging rough edges as expected from sparse seed
- Real spec files linked, not idealized versions
- No overclaiming; the product speaks for itself

### 4. Accessible Depth
The site works for both casual visitors and technical leads doing serious evaluation.

**User perspective:** "I could skim it in 2 minutes or dig into the interactive tools for 20."

**Examples:**
- Hero gives the one-sentence summary
- Interactive wizard for hands-on engagement
- Linking to actual spec and source materials for deep-divers

### 5. Developer Credibility
The tone, content, and design signal that this was made by developers, for developers.

**User perspective:** "This doesn't feel like a corporate product page — it feels like a developer explaining something they actually believe in."

**Examples:**
- Technical but accessible prose
- No buzzwords, no fluff
- Version badge and build metadata visible

## Inviolable Principles

- Never claim the tool solves what it doesn't — if a rough edge exists due to sparse input, say so
- Never hide the meta-nature of the demo — the site must explain it is itself a RootSpec output
- Never obscure the spec hierarchy or make it seem more complex than it is
- Always link to real source materials (SEED.md, spec files, GitHub repo)
- Never use stock imagery or placeholder content in the final product — everything must be real

## North-Star Experience

A technical lead lands on the site at 11pm, skeptical of another specification methodology. Within two minutes they've seen the hierarchy, played with the wizard, and understood how their Google Doc spec could become something executable. They open the GitHub link. They come back the next day to try it on a real project. They didn't feel sold to — they felt like they found something that understood their problem.
