# Level 1: Foundational Philosophy

## Competitors

- **GitHub Copilot / Cursor** — AI code generation with no structural validation layer
- **Notion / Confluence specs** — Unstructured documentation nobody reads or enforces
- **OpenAPI / JSON Schema** — Machine-readable contracts for APIs, not product philosophy
- **Shape Up / Basecamp** — Opinionated project methodology, not a specification framework
- **Architecture Decision Records (ADRs)** — Captures decisions but not hierarchy or validation

## Table Stakes

- Clear explanation of what the product does and who it's for
- Concrete methodology that can be applied to real projects
- Evidence that the approach produces better outcomes
- A way to get started quickly (not just theory)

## Pain Points

- **Spec drift** — Documentation falls out of sync with code; nobody trusts it
- **Philosophy-implementation gap** — Business intent can't be traced through to code
- **Unreliable AI output** — AI generates code that satisfies requests but violates intent
- **Unread specs** — Google Docs or Confluence pages that accumulate and get ignored
- **Circular dependencies** — Implementation constraints bleed upward and corrupt product direction

## Influences

- **Dependency inversion principle** — High-level policy should not depend on low-level detail
- **Literate programming** — Code and explanation unified into one coherent whole
- **Test-driven development** — Specification as executable validation, not aspirational prose
- **Constitutional design** — Foundational documents that constrain all decisions beneath them

## Mission

RootSpec exists because when philosophy doesn't guide implementation, implementation corrupts philosophy. We give developers and teams a structured method to extract, record, and validate human intent — so that AI-assisted development produces systems that are not just functional, but purposeful.

## Design Pillars

### 1. Clarity Over Completeness
The spec makes intent undeniable — a well-structured skeleton beats a bloated document nobody reads.

**User perspective:** "I can see exactly what this product is trying to be and why."

**Examples:**
- Each level answers exactly one question (why, what, how, how much)
- No cross-level ambiguity — every decision lives at exactly one level

### 2. Validation as First Principle
Every claim in the spec must be traceable to a test or a measurable outcome — specs that can't be validated are opinions.

**User perspective:** "I can prove the spec matches reality."

**Examples:**
- User stories generate executable acceptance criteria
- Validation runs on the spec itself before implementation begins

### 3. Philosophy Stability
The hierarchy enforces one-directional dependency so that implementation changes never corrupt the foundational product intent.

**User perspective:** "My core product vision doesn't shift every sprint."

**Examples:**
- L1 changes require conscious decision, not accidental drift
- Trade-offs at L2 are explicitly stated so conflicts surface early

### 4. Approachability Without Sacrifice
Rigor should not require ceremony — four commands drive all workflows in natural language, accessible to a solo developer and a large team alike.

**User perspective:** "I can adopt this without a consulting engagement."

**Examples:**
- Four commands drive all workflows
- Natural language specification — not config files or YAML-first

## Inviolable Principles

- Never sacrifice clarity for completeness — a shorter, unambiguous spec beats a thorough ambiguous one
- Never allow lower levels to redefine higher-level intent — philosophy flows downward only
- Never present validation as gatekeeping — the spec identifies problems, it does not block work
- The spec serves the team, not the other way around — overhead must always be justified by value
