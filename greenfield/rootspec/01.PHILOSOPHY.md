# Level 1: Foundational Philosophy

## Competitors

- **Google Docs specs** — The default "write a doc and share it" approach. Nobody reads them, they drift from implementation, and they can't validate anything.
- **Notion/Confluence wikis** — Better organized than Google Docs, but still passive documents that diverge from code over time.
- **AI code generators (Copilot, Cursor, Claude Code)** — Generate code fast but have no mechanism to ensure output aligns with product intent. Knowledge without wisdom.
- **Traditional requirements management (Jira, Linear specs)** — Track what to build but not why. No hierarchy enforcement, no validation loop.
- **BDD/Cucumber** — Executable specs but limited to test-level concerns. No philosophical foundation, no hierarchy.

## Table Stakes

- Clear explanation of what the product is and why it exists
- Getting started instructions that work
- Interactive demonstrations — not just text descriptions
- Open source access (GitHub)
- Responsive, accessible design
- Professional visual quality

## Pain Points

- **Spec drift** — Requirements documents diverge from implementation within weeks. Nobody notices until it's too late.
- **Philosophy-implementation gap** — The "why" behind a product lives in founders' heads and degrades at every handoff. AI accelerates this degradation.
- **Unreliable AI output** — AI tools generate code confidently but have no way to validate alignment with intent. More code, same drift.
- **Dead documentation** — Specs written once, shared once, read never. No living connection to what gets built.
- **Methodology resistance** — Developers avoid spec methodologies because they feel academic, bureaucratic, or disconnected from actual work.

## Influences

- **Stripe's documentation** — Technical depth with clarity. Respects the reader's intelligence without being obtuse.
- **Interactive technical explanations (Nicky Case, Bret Victor)** — Learning through doing, not reading. The medium IS the message.
- **"Eat your own dogfood" culture** — Products that demonstrate their value by using themselves. Credibility through self-reference.
- **Developer tool landing pages (Vercel, Linear)** — Confident, clean, substance-first. No enterprise fluff.

## Mission

Bridge the gap between human intent and software implementation. When code and specs can be generated trivially, the real value is validation and proof — ensuring what gets built reflects what was meant.

## Design Pillars

### 1. Vindicated Clarity

Visitors feel the relief of recognition: "Finally, someone articulated the problem I've been living with."

**User perspective:** "I've fought spec drift for years and couldn't explain why everything felt broken. This nails it."

**Examples:**
- The Problem section names specific frustrations developers already have
- Language is direct and specific, not abstract

### 2. Empowered Confidence

Visitors feel capable and ready: "I could actually use this — it's not academic."

**User perspective:** "This isn't another methodology I'll read about and never adopt. I can see exactly how it works."

**Examples:**
- Interactive features let visitors experience the methodology hands-on
- The wizard produces tangible output from their own input
- Getting started is four commands, not a certification

### 3. Honest Credibility

Visitors feel trust through proof: "These people practice what they preach."

**User perspective:** "The site itself was built with this methodology. That's either very bold or very confident — either way, I'm paying attention."

**Examples:**
- Meta banner explains the site is a RootSpec demo with links to the spec and seed
- Rough edges are acknowledged, not hidden — transparency builds trust
- No inflated claims or buzzword marketing

## Inviolable Principles

- Never use buzzwords or marketing fluff — if it could appear on any product's landing page, it's too generic
- Show, don't tell — every claim about the methodology must be demonstrated, not just stated
- No external API calls — all interactive features run client-side
- Respect the visitor's intelligence — write for developers who've been burned, not executives who need convincing
- Accessibility is non-negotiable — keyboard navigation, screen readers, color contrast

## North Star

Six months from now, a developer who's frustrated with spec drift visits the site, immediately understands what RootSpec is from the hero, feels their frustration validated by the Problem section, tries the interactive hierarchy explorer and spec wizard, and leaves thinking: "I'm using this on my next project." They share the link with their team lead. The meta banner — showing the site was built with the methodology — is what sealed it.
