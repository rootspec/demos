# Level 1: Foundational Philosophy

## Competitors

- **Traditional BDD/TDD frameworks** — Gherkin, Cucumber, RSpec — pedantic, expensive to maintain, rarely survive contact with real teams
- **Design docs and PRDs** — Google Docs or Notion pages that capture intent at a point in time and then drift from reality
- **AI code generators** — Copilot, Cursor, and similar tools that generate code fluently but have no mechanism to validate intent
- **No-spec development** — Most teams operating on shared mental models, tribal knowledge, and hope

## Table Stakes

- Ability to define product requirements before coding
- Some form of traceability from requirement to implementation
- Works with existing tooling and programming languages
- Can be understood by both technical and non-technical stakeholders

## Pain Points

- **Spec drift** — Specifications written at project start become stale; by launch they describe a different product than what shipped
- **Philosophy-implementation gap** — The "why" of a product exists in founders' heads; implementation teams work from Jira tickets
- **AI output without validation** — LLMs generate plausible code that satisfies stated instructions but violates unstated intent; no mechanism exists to catch this
- **Documents nobody reads** — Long PRDs and design docs accumulate; engineers guess when they can't find answers; decisions get re-litigated constantly
- **Drift starts at the source** — Even the intent author holds implicit assumptions that create mid-cycle pivots and quiet disconnects with end users
- **Expensive existing approaches** — BDD/TDD are known to work but transform a few pages of specification into books of pedantic instructions; teams abandon them

## Influences

- **Dependency inversion principle** — Philosophy should drive implementation, never vice versa; high-level policy should be stable when low-level details change
- **Test-driven development** — Constraints as guardrails to keep intent intact; the spec and tests as the primary artifacts
- **Long-form editorial writing** — Clarity of thought expressed with precision; the reader should feel someone thought carefully
- **Star Trek replicator metaphor** — Generating something that exactly satisfies intent requires first capturing that intent with precision

## Mission

RootSpec must exist because generating software from intent is now tractable — but validating that the result satisfies the original intent is not. RootSpec bridges the gap: it captures human wisdom and philosophy in a structured, executable specification, then uses that specification as a validation gate through which only proven implementations may pass.

## Design Pillars

### 1. Validated Intent

Users feel confident that what ships actually matches what they meant — not just what they said.

### 2. Philosophy First

Users feel grounded in purpose when making decisions — features have reasons, trade-offs have rationale, and implementation details have philosophical backing.

### 3. Approachable Rigor

Users feel the methodology is demanding without being punishing — it asks hard questions but makes answering them tractable.

### 4. Honest Progress

Users feel they are making real progress they can measure and trust — not bureaucratic compliance theater.

## Inviolable Principles

- Never generate code as a primary artifact — the spec is the primary artifact; code is downstream
- Never block work — identify problems, make them visible, do not gate human judgment
- Never require a specific AI tool, editor, or platform — the methodology must be tool-agnostic
- Never sacrifice the hierarchy — lower levels cannot influence higher levels; this is the core constraint that makes everything else stable
- Always surface the "why" before the "what" or "how" — intent precedes strategy precedes implementation

## North-Star Experience

Six months from now, a solo developer using RootSpec to build a complex web application should be able to sit down, describe their vision in plain language, run four commands, and have a running, tested, production-ready application. More importantly: when they ship a new feature a year later, they should be able to read the spec and immediately understand why every decision was made — and trust that the system still reflects their original intent, not some accumulated drift from a hundred small compromises.
