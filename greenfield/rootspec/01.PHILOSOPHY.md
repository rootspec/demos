# L1: Foundational Philosophy

## Mission

RootSpec exists to close the gap between human intent and machine execution. Most software fails not because developers lack skill, but because the philosophy driving the product was never captured — leaving teams to guess at trade-offs, AI to hallucinate requirements, and products to drift away from their original purpose.

RootSpec is a specification methodology that makes intent explicit, hierarchical, and testable. Philosophy guides implementation. Always.

## Problem Being Solved

Software teams routinely suffer from:
- **Spec drift** — requirements documents that diverge from the actual product over time
- **Philosophy-implementation gap** — code that works but serves no coherent purpose
- **Unreliable AI output** — AI tools that generate plausible code disconnected from product intent
- **Dead documents** — Google Docs specs nobody reads, references, or maintains

## Target Audience

Developers and technical leads who:
- Have been burned by scope drift or unclear requirements
- Use AI tools (Copilot, Claude, Cursor) and want more reliable, intent-aligned output
- Lead teams where "the spec" is a Google Doc nobody reads
- Build complex, long-lived products that must stay coherent over time

## Design Pillars

These describe how visitors should FEEL when experiencing this site — not features, but emotional outcomes:

### Clarity over cleverness
Every explanation lands immediately — no jargon, no hand-waving.

### Credibility through honesty
Rough edges are acknowledged; we show real results, not polished demos.

### Tangible over abstract
Concepts are demonstrated, not just described — the site is itself a proof.

### Confident but not preachy
This is a methodology we believe in, presented without lecturing.

### Approachable depth
Technical enough to be credible; accessible enough to invite exploration.

## Inviolable Principles

- The site must itself be a working demo of RootSpec — generated from a sparse seed, no manual mockups.
- The meta banner is non-negotiable — visitors must understand what they're looking at and why it exists.
- No external API calls — all interactive features run entirely client-side.
- The version of RootSpec used to build the site is always visible.
- We do not oversell. If something is rough, we say so.

## North Star

Six months from now, a developer burned by scope drift visits this site, plays with the hierarchy explorer, runs the spec wizard, and thinks: "This is exactly the problem I have. And this is actually how you fix it." They don't email us asking what RootSpec is — the site answers that. They clone the repo and run `/rs-init` before the week is out.

## Key Messages

- "When code and specs can be generated trivially, the real value is validation and proof."
- AI has knowledge but not wisdom. Humans supply philosophy; AI executes implementation. The spec bridges the gap.
- Four skills drive all workflows: `/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate`
- RootSpec is not a code generator — it extracts and validates human intent.
- Not a gatekeeper — it identifies problems, doesn't block work.
