# Level 1: Foundational Philosophy

## Competitors

- **Confluence / Google Docs** — Unstructured documentation that becomes stale and unread
- **Jira / Linear** — Ticket-based systems that track work but don't validate intent
- **Cucumber / Gherkin** — BDD spec tools that focus on test syntax, not philosophy
- **OpenAPI / Swagger** — Schema-first specs that describe APIs, not product intent
- **README-driven development** — Informal practice without enforcement or hierarchy

## Table Stakes

- A way to capture product intent in natural language
- Structure that prevents circular dependencies between spec levels
- Compatibility with any tech stack and tooling
- Support for AI-assisted workflows without surrendering human judgment
- Validation to confirm the spec is internally consistent

## Pain Points

- **Spec drift** — Documentation diverges from implementation as teams iterate, leaving specs that describe a product that no longer exists
- **Philosophy-implementation gap** — Implementation details get decided without grounding in product purpose, leading to features that technically work but miss the point
- **Unreliable AI output** — AI tools generate plausible-looking code without knowing whether it's right for the product, because there's no validated intent to check against
- **Unread specs** — Traditional specs live in Google Docs or wikis that no one consults, because they're disconnected from the actual development workflow
- **Untraceable decisions** — Teams can't explain why a feature exists or what principle it serves

## Influences

- **Dependency inversion principle** — High-level policy should not depend on low-level details
- **Domain-driven design** — The importance of shared language and bounded contexts
- **Systems thinking** — Understanding products as interconnected systems with stable rules
- **Test-driven development** — The value of specifying behavior before implementation

## Mission

RootSpec must exist because software teams need a way to keep human intent — the philosophy behind a product — stable and authoritative as AI accelerates implementation. When code and specs can be generated trivially, the real value is validation and proof that implementation matches intent.

## Design Pillars

### 1. Grounded Clarity
Users feel anchored to the real purpose of their product, not lost in implementation details.

**User perspective:** "I always know *why* we're building this and can trace any decision back to first principles."

**Examples:**
- Every feature traces to a Design Pillar
- Philosophy files remain readable and stable while implementation evolves

### 2. Trustworthy Confidence
Users feel confident that what gets built actually matches what was intended — not just syntactically, but philosophically.

**User perspective:** "I can hand this to an AI — or a new team member — and know the output will actually fit the product."

**Examples:**
- Validation gates catch drift before it compounds
- Acceptance criteria are derived from spec, not written after the fact

### 3. Approachable Rigor
The methodology feels principled without being academic or heavyweight.

**User perspective:** "This is structured enough to be useful and light enough that I'll actually use it."

**Examples:**
- Four skills cover every workflow
- Natural language specification — not config or YAML-first

### 4. Collaborative Transparency
Teams and AI systems work from the same shared truth, making handoffs and corrections visible.

**User perspective:** "Everyone on the team — human or AI — is working from the same document and the same intent."

**Examples:**
- Specs are checked in alongside code, not stored in a separate wiki
- The spec acts as a communication layer between human philosophy and AI execution

## Inviolable Principles

- Never let implementation details contaminate philosophy-level documents
- Never sacrifice human intent for AI convenience — AI executes, humans supply wisdom
- Never make the spec a gatekeeper — it identifies problems, it does not block work
- Always keep higher-level documents stable when lower levels change
- The methodology must work without AI, with any tooling, and for any team size
