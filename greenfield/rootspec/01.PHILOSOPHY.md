# Level 1: Foundational Philosophy

**Product:** RootSpec Marketing Site
**Version:** 1.0.0
**Last Updated:** 2026-04-12

---

## Competitors

- **Traditional spec tools** — Confluence, Notion, Google Docs: flexible but unstructured; specs drift and become stale
- **AI code generators** — GitHub Copilot, Cursor, Claude: generate code quickly but lack grounding in human intent
- **Project management tools** — Jira, Linear, Asana: track work but don't encode reasoning or philosophy
- **Architecture documentation tools** — C4 model, ADRs, RFC templates: document decisions but lack enforced hierarchy

## Table Stakes

- Clear explanation of what RootSpec is and why it exists
- Demonstration of the methodology, not just description
- Path to getting started (GitHub, documentation, commands)
- Works on mobile and desktop without friction

## Pain Points

- **Spec drift** — Existing documentation approaches fail because specs and code diverge the moment development begins; nobody updates the Google Doc
- **Philosophy-implementation gap** — AI tools have knowledge but not wisdom; they execute without understanding intent, producing technically correct but philosophically wrong output
- **Unvalidatable requirements** — Vague requirements docs can't be tested, so nobody knows when they're done
- **Tool-first thinking** — Most methodology tools define their format first and ask teams to adapt; philosophy gets stuffed into a ticket description
- **Unreliable AI output** — Without a validation gate, AI-generated code satisfies prompts, not intent

## Influences

- **Thoughtful developer blogs** — Clear, direct writing that explains methodology through practice, not theory
- **Unix philosophy** — Simple tools that compose; constraints as features
- **Test-driven development** — Spec before implementation, validation as the arbiter of truth
- **Design system documentation** — Structured, opinionated, every decision traceable

---

## Mission

RootSpec must exist because AI has made implementation trivially cheap while making intent dangerously implicit. When code can be generated in seconds, the real value is the spec that validates that code against human philosophy — and the spec can only do that job if it's structured, hierarchical, and treated as the source of truth, never the other way around.

## Design Pillars

### 1. Earned Clarity
Visitors leave knowing exactly what RootSpec is and why it matters — not from a tagline, but from experiencing the methodology in action.

**User perspective:** "I came in skeptical and left convinced — because it showed me, not just told me."

**Examples:**
- Interactive hierarchy explorer makes the five-level concept tangible before reading about it
- Meta banner explains the site is itself a RootSpec demo — the medium is the message
- Before/after comparison uses real content, not lorem ipsum

### 2. Respectful Intelligence
Visitors feel respected as developers who think carefully — not talked down to by marketing copy.

**User perspective:** "This is written by someone who actually builds things. I trust it."

**Examples:**
- Tone is confident but not preachy — explains without lecturing
- Rough edges from minimal seed are acknowledged openly in the meta banner
- No buzzwords; methodology is explained through concrete examples

### 3. Tangible Methodology
The site lets visitors touch and use the methodology, not just read about it — making RootSpec approachable rather than academic.

**User perspective:** "I can see how this would work for my project."

**Examples:**
- "Spec Your Idea" wizard walks through three spec levels for a user's own product idea
- Hierarchy explorer shows reference rules through interaction, not just prose
- Four-skill workflow is visualized as a real before/after, not an abstract diagram

### 4. Transparent Provenance
Every aspect of the site — its origin, its limitations, its version — is visible and honest.

**User perspective:** "I know exactly what this was built from and how."

**Examples:**
- Meta banner links to the SEED.md and spec files in the GitHub repo
- RootSpec version displayed prominently in header and hero
- Footer identifies the builder and build date

---

## Inviolable Principles

- Never obscure rough edges — the site is a demo, and demos should be honest about their inputs
- Never use feature marketing language; every claim must be demonstrable on this page
- The methodology must be experienced, not just read about — at least three interactive sections are required
- Philosophy stays stable even if the implementation changes; don't let tech choices drive messaging
- No external API calls — everything runs client-side, keeping the demo self-contained and trustworthy

---

## North-Star Experience

Six months from now, a developer who found RootSpec through this site should be able to say: "I read the site, tried the wizard, explored the hierarchy, and immediately understood how to structure my next project's spec. I cloned the repo the same day." The site's sparse-seed origins are a feature, not a bug — they prove that the methodology produces coherent, useful output even without exhaustive human direction.
