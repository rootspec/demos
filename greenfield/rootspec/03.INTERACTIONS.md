# Level 3: Interaction Architecture

> References: L1 (01.PHILOSOPHY.md), L2 (02.TRUTHS.md)

## Core User Journeys

### Journey 1: First-Time Arrival (Primary)

1. Visitor lands on page → Meta banner is immediately visible, framing context
2. Hero section loads → Tagline and one-line explanation establish what RootSpec is
3. Visitor scrolls → Problem section resonates (or doesn't); they continue or leave
4. How It Works section → Four-skill walkthrough with visual before/after
5. Interactive sections → One or more of three tools is engaged
6. CTA section → GitHub link presented; visitor converts or bookmarks

**Key feedback at each step:** Content is scannable; no walls of text force sequential reading. Visitors can jump to any section.

### Journey 2: Interactive Exploration

1. Visitor reaches Hierarchy Explorer → Clicks a level node
2. Level expands to show example content → Adjacent levels dim or highlight allowed references
3. Visitor hovers another level → Arrow connections animate to show reference flow
4. Visitor uses keyboard (tab/enter) → Same behavior as mouse interaction
5. Mobile visitor taps → Touch-friendly hit targets, same content

### Journey 3: Spec Wizard

1. Visitor types a one-line product idea → Input is validated (non-empty)
2. Step 1 — Mission selection → Visitor picks from templates or writes own
3. Step 2 — Design pillars → Visitor selects 3–5 from suggested options
4. Step 3 — Key interaction → Visitor describes one core user flow
5. Output → Skeleton spec displayed, showing L1–L3 mapping
6. Visitor can restart or share (copy to clipboard)

### Journey 4: Before/After Comparison

1. Visitor reaches comparison section → Two panels visible (or toggle view on mobile)
2. Desktop: slider between panels; Mobile: toggle button
3. "Without RootSpec" panel: vague requirements, ambiguous stories
4. "With RootSpec" panel: structured hierarchy, traceable decisions, testable stories
5. No interaction required — content is readable without interaction; interaction enhances contrast

## Interaction Patterns

### Theme Toggle
- **Trigger:** Manual toggle button in header (sun/moon icon)
- **Feedback:** Instant theme switch, no flash; preference persisted in localStorage
- **Default:** System preference (`prefers-color-scheme`) on first visit

### Hierarchy Explorer Node Click
- **Trigger:** Click or Enter key on any level node
- **Feedback:** Node expands with smooth animation; reference arrows animate; other nodes dim
- **Reset:** Click same node again to collapse; clicking different node collapses current

### Wizard Step Navigation
- **Trigger:** "Next" button (disabled until current step has valid input), or Enter key
- **Feedback:** Step indicator updates; content transitions with slide animation
- **Back navigation:** "Previous" button always enabled on steps 2+
- **Validation:** Empty mission field shows inline error message, does not advance

### Before/After Toggle (Mobile)
- **Trigger:** Toggle button tap
- **Feedback:** Active panel label highlights; content swaps with fade
- **Desktop:** Drag handle between panels; panels resize proportionally

## Failure Modes

- **Wizard: no input** → "Please describe your product idea" error shown inline; advance blocked
- **Wizard: very long input** → Input capped at N characters (value in fine-tuning); counter shown
- **Broken GitHub link** → Not a runtime failure — must be caught at build time via link validation
- **Theme preference not supported** → Falls back to light theme; no error shown
- **JavaScript disabled** → Page is readable; interactive sections show static fallback content with note

## Accessibility Patterns

- All interactive elements have focus states visible at ≥ 3:1 contrast ratio
- Hierarchy Explorer nodes have `role="button"` and `aria-expanded` attributes
- Wizard steps use `aria-live` region for step announcements
- Theme toggle announces current state (`aria-label` updates dynamically)
- Before/After comparison panels have descriptive `aria-label` attributes
