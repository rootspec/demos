# Level 3: Interaction Architecture

## Core Interaction Model

The site is primarily a reading experience with three interactive components embedded in the content flow. The reading experience leads; the interactivity demonstrates. Visitors arrive with a question ("what is this?"), read through the answer, and are invited to experience the methodology firsthand before deciding whether to explore the repo.

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Interactions |
|-------|-------|----------|-----------------|
| Immediate | Single action | Seconds | Theme toggle, section expand/collapse, wizard step |
| Session | Reading journey | Minutes | Full page scroll, wizard completion, hierarchy exploration |
| Extended | Return visit | Multiple sessions | Sharing the URL, referencing specific sections, following the repo |
| Lifetime | Adoption | Ongoing | Using RootSpec on their own project |

## Primary Interaction Flows

### Flow 1: First-Time Reader (Primary)

1. Visitor arrives at the page
2. **Meta banner is immediately visible** — they understand this is a demo of the methodology before reading anything else
3. Hero orients them: what RootSpec is, the version it was built with
4. They scroll through The Problem section — recognition of a familiar pain
5. They encounter the Methodology Diagram — the five-level hierarchy becomes visual
6. How It Works section — the five skills explained with before/after context
7. They reach an interactive element (Hierarchy Explorer or Wizard) and engage
8. They read the Author's Notes — the full original text, unabridged
9. They hit the CTA — link to the GitHub repo, getting started instructions

**Trigger:** Direct link or referral
**Feedback:** Progressive disclosure as they scroll; no loading states, no spinners

### Flow 2: Hierarchy Explorer Interaction

1. Visitor sees the five-level hierarchy visualization
2. They click a level (e.g., "Philosophy") — it expands to show example content
3. Visual arrows indicate which levels each can reference (upward only)
4. Hovering a level highlights its allowed references
5. Keyboard navigation: Tab to focus levels, Enter/Space to expand, arrow keys to move between levels
6. Each level shows example content relevant to a real product (the RootSpec site itself, ideally)

**Trigger:** Curiosity about the methodology structure
**Feedback:** Expand/collapse animation ([short duration]); highlight state on hover/focus
**Edge cases:** All levels start collapsed; only one level expands at a time (accordion behavior)

### Flow 3: "Spec Your Idea" Wizard

1. Visitor enters a one-line product idea in the input field
2. Step 1 — Mission: They select from mission templates or write their own
3. Step 2 — Design Pillars: They pick [three to five] pillars from suggestions (emotional phrases)
4. Step 3 — Key Interaction: They describe one core user action in a text area
5. Output: A skeleton spec showing how their input maps to L1 (mission + pillars) and L3 (interaction)
6. Copy button allows them to take the skeleton spec with them

**Trigger:** "I want to try this with my own idea"
**Feedback:** Step indicator shows current position; previous steps are visible and editable; output appears after completing the final step
**Edge cases:** Empty input defaults to placeholder example; wizard is navigable backward to any previous step

### Flow 4: Before/After Comparison

1. Visitor encounters the split view
2. Left panel: "Without spec" — a realistic-looking vague requirements doc with ambiguous stories
3. Right panel: "With RootSpec" — same product concept rendered as a structured hierarchy with traceable decisions
4. A toggle or slider switches between the two views
5. Real content in both panels — the contrast should be substantive, not aesthetic

**Trigger:** Skepticism ("how is this different from what I already do?")
**Feedback:** Instant toggle, no animation delay; clear labeling of which state is shown

### Flow 5: Theme Toggle

1. Visitor uses the theme toggle in the header
2. Theme switches between light and dark immediately
3. Preference is stored for the session; system preference is the default on first visit

**Trigger:** Personal preference or reading environment
**Feedback:** Immediate switch ([brief duration] transition on background/text colors only; no layout shift)

## Navigation Patterns

- Header: Site name/logo + theme toggle + nav links (anchored to sections)
- Nav links: "The Problem", "How It Works", "Why I Built RootSpec", "Try It"
- "Why I Built RootSpec" nav label must match the section heading verbatim — this is the author's voice and should not be genericized
- Mobile: Navigation collapses to a compact form (hamburger or stacked); tap targets minimum [accessible touch target size]
- On mobile, the wizard and hierarchy explorer must be usable with thumb-only interaction

## Feedback Patterns

- **Expand/collapse:** Smooth height animation, [short duration]; chevron rotates to indicate state
- **Hover states:** Subtle background or border change; no color transformations that lose contrast
- **Focus states:** Visible keyboard focus ring; meets accessibility contrast minimums
- **Theme transition:** Background and text colors transition; layout elements do not shift
- **Wizard steps:** Active step is visually distinct; completed steps show a completion indicator
- **Copy action:** Button briefly changes to "Copied!" then reverts

## Failure Modes and Edge Cases

- **JavaScript disabled:** Static content (all sections, author notes, meta banner, hero) is fully readable. Interactive sections (explorer, wizard, comparison) show a graceful fallback message or static representation
- **Very narrow viewport ([minimum supported width]):** No horizontal scroll; all content reflows; the diagram scales without losing legibility
- **Keyboard-only navigation:** Tab order follows reading order; all interactive elements are reachable and operable without a mouse
- **System dark mode:** On first visit, system preference is honored; subsequent visits respect manual override
- **Long product ideas in wizard:** Text input truncates gracefully; output skeleton spec handles long mission text without breaking layout

## Verification Criteria

- Meta banner is visible above the fold on all viewports without scrolling
- Hierarchy explorer expands and collapses all five levels with keyboard navigation
- Wizard produces a spec skeleton from any non-empty product idea input
- Before/after toggle correctly switches between both panels
- Theme toggle persists for the session
- At [mobile reference width], no horizontal scrolling occurs and all tap targets are reachable
- Author's Notes section contains the full original text verbatim
