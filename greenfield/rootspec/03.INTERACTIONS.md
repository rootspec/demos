# Level 3: Interaction Architecture

**Product:** RootSpec Marketing Site
**Version:** 1.0.0
**Last Updated:** 2026-04-12

---

## Core Interaction Loops

### Loop 1: Scan and Anchor
**Scale:** Immediate (seconds)

A visitor lands and scans the page. The hero communicates what RootSpec is in a single sentence. The meta banner is immediately visible and explains the site's own origin. The version badge establishes credibility. The visitor either scrolls down (engaged) or leaves.

**Trigger:** Page load
**Action:** Passive scan
**Feedback:** Visual hierarchy (hero → meta banner → problem → interactive sections)
**Success state:** Visitor scrolls past the fold

### Loop 2: Concept Encounter
**Scale:** Immediate to session (seconds to minutes)

The visitor reaches the "How It Works" section and sees the four-skill workflow. They understand the sequence (init → spec → impl → validate) and can map it to their own development experience.

**Trigger:** Scrolling into the workflow section
**Action:** Reading and scanning the visual walkthrough
**Feedback:** Visual representation of each skill with clear before/after framing
**Success state:** Visitor understands the four commands and their purpose

### Loop 3: Hierarchy Exploration
**Scale:** Session (minutes)

The visitor reaches the Hierarchy Explorer and interacts with it. They click or tap a level to expand it and see example content. They hover or focus a level to see which levels it can reference. They discover the reference rules (arrows flow upward only) through interaction rather than reading.

**Trigger:** Arriving at the Hierarchy Explorer section
**Action:** Click/tap levels, hover/focus to see reference connections
**Feedback:** Level expands with example content; connected levels highlight; arrows animate
**Fallback:** If JavaScript is unavailable, levels are pre-expanded in a static accordion layout
**Success state:** Visitor can articulate "each level can only reference higher levels" without having been explicitly told

### Loop 4: Methodology Application (Wizard)
**Scale:** Session (minutes)

The visitor enters a one-line product idea and steps through three structured prompts:
1. Select or write a mission statement
2. Choose 3-5 design pillars from suggestions
3. Describe one key interaction

At the end, they see a skeleton spec (L1-L3) showing how their input maps to the hierarchy. The output is copyable.

**Trigger:** Entering a product idea and pressing start
**Action:** Three-step sequential wizard; each step must be completed before proceeding
**Feedback:** Progress indicator; each step shows selected values; output step reveals the skeleton spec
**Edge cases:**
- Empty product idea: prompt remains visible with placeholder; form does not submit
- User selects fewer than [min pillars] pillars: step 2 shows a minimum-count reminder before proceeding
- Long output: skeleton spec scrolls within its container; "copy" action copies full text
**Success state:** Visitor holds a skeleton spec for their own idea and recognizes the methodology applied to something they care about

### Loop 5: Before/After Comparison
**Scale:** Immediate to session (seconds to minutes)

The visitor sees two panels side-by-side (or toggled on mobile). One shows the "without spec" state; one shows the "with RootSpec" state. The content is real — not lorem ipsum. A slider or toggle lets them compare directly.

**Trigger:** Arriving at the comparison section
**Action:** Move slider or press toggle to switch views
**Feedback:** Smooth transition between states; labels clearly mark which is which
**Success state:** Visitor can identify at least one concrete difference between the two states

### Loop 6: Conversion
**Scale:** Session exit

The visitor reaches the Open Source CTA. They see a link to the GitHub repo, the getting started commands, and community links. The section is direct: here is how to begin.

**Trigger:** Arriving at the CTA section (or clicking any "get started" link)
**Action:** Click GitHub link, copy a command, or navigate to documentation
**Feedback:** External navigation to GitHub; commands are copyable
**Success state:** Visitor reaches the GitHub repo or copies a getting started command

## Theme Toggle Interaction

**Trigger:** Clicking the theme toggle button (visible in header)
**Action:** Toggle fires; preference is written to local storage; all themed elements update
**Feedback:** Immediate visual update across the entire page; no flash or partial update
**Failure mode:** If local storage is unavailable, theme toggle still works for the session but is not persisted
**Accessibility:** Toggle must have a visible label describing current state ("Switch to dark mode"); must be keyboard-operable; must announce state change to screen readers

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Sections |
|-------|-------|----------|-------------|
| Immediate | Single element or section | Seconds | Hero, Meta Banner, Theme Toggle |
| Session | Scrolling through and engaging with interactive sections | Minutes | Hierarchy Explorer, Wizard, Before/After |
| Exit | Deciding to act on what was learned | Variable | Open Source CTA, GitHub link |

---

## Failure Modes and Edge Cases

- **JavaScript disabled:** Hierarchy Explorer falls back to a static expanded layout. Wizard degrades gracefully to a non-interactive description. Before/After shows both states stacked vertically.
- **Slow connection:** Hero and meta banner are visible immediately (static HTML). Interactive sections load progressively; they do not block above-the-fold content.
- **Mobile touch:** Wizard steps use large touch targets. Hierarchy Explorer responds to tap (not hover). Before/After comparison uses a toggle button on mobile (not a slider, which is difficult on small screens).
- **Keyboard navigation:** Every interactive element is reachable via Tab. The hierarchy explorer supports arrow keys to move between levels. The wizard supports Enter to advance steps.
- **Very long product idea in wizard:** Input is accepted but output preview gracefully wraps; no truncation of user input.

## Verification Criteria

An implementation correctly satisfies this interaction architecture if:
1. A visitor can complete the Hierarchy Explorer interaction without touching a mouse
2. A visitor can complete the "Spec Your Idea" wizard on a mobile device using only touch
3. The theme toggle persists across page reloads (verified by refreshing after toggling)
4. The "without spec" and "with RootSpec" comparison panels contain real, readable content — not placeholders
5. The meta banner is visible above the fold on a [standard viewport] without scrolling
6. Every interactive section has a non-JavaScript fallback that is useful, not just empty
