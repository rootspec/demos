# Level 3: Interaction Architecture

## Core User Journey

A visitor arrives from a search, a tweet, or a colleague's recommendation. They are typically a developer or technical lead who has recently been frustrated by spec drift or unreliable AI output.

**On arrival:** The hero section delivers the tagline and one-sentence explanation. The meta banner immediately establishes that this site is itself a RootSpec demo. The visitor either dismisses it (not their problem) or leans in (this names their pain).

**Early exploration:** The visitor scrolls through the problem section and recognizes specific pain points — spec drift, philosophy-implementation gap, untraceable decisions. The how-it-works section shows the four-skill workflow. This is linear reading — no interaction required.

**Interactive engagement:** The visitor engages with interactive elements. They explore the hierarchy, run the wizard with their own product idea, or examine the before/after comparison. These are non-sequential — visitors may engage with any or none, in any order.

**Handoff:** The visitor follows the CTA to the GitHub repo, reads the README, or clicks through to the actual spec files. This is the handoff — the site's job is done.

## Primary Interaction Patterns

### Pattern 1: Passive Reading (default)
The page is navigable without any interaction. Visitors who don't engage with interactive elements still get the full value proposition through text and visual sections. No content is hidden behind interaction.

### Pattern 2: Hierarchy Explorer
**Trigger:** Visitor clicks a level in the hierarchy visualization.
**Behavior:** Level expands to show example content. Arrows visually indicate allowed reference directions (upward only). Hovering a level highlights its allowed references.
**Feedback:** Smooth expand/collapse animation. Highlighted connections on hover.
**Keyboard:** Arrow keys navigate levels; Enter/Space expand/collapse.
**Touch:** Tap to expand; visual connection highlights on tap.
**Failure mode:** If JavaScript fails, levels show a static summary. No blank panels.

### Pattern 3: Spec Wizard ("Spec Your Idea")
**Trigger:** Visitor engages with the wizard section.
**Behavior:** Three-step flow:
  1. Enter a one-line product idea
  2. Select or write a mission statement (from templates)
  3. Pick [3-5] design pillars from suggestions (or write custom)
  4. Describe one key interaction

**Output:** Skeleton spec showing how input maps to L1-L3. Displayed inline, no page navigation.
**Feedback:** Step progress indicator; output appears as visitor completes each step.
**Keyboard:** Tab through inputs; Enter to advance steps.
**Touch:** Large tap targets; scroll to output on completion.
**Failure mode:** If a step is skipped, output shows placeholder text for that level with a note to return and fill it in.
**Constraint:** Client-side only. No AI. No data sent anywhere.

### Pattern 4: Before/After Comparison
**Trigger:** Visitor views the comparison section.
**Behavior:** Toggle or slider switches between "Without spec" and "With RootSpec" views. Both panels contain real content — not lorem ipsum.
**Feedback:** Smooth transition between states. Labels clearly identify which view is active.
**Keyboard:** Toggle navigable via keyboard.
**Touch:** Tap toggle; optional swipe to switch panels.
**Failure mode:** Both views visible stacked if JavaScript fails.

### Pattern 5: Theme Toggle
**Trigger:** Visitor clicks theme toggle in header.
**Behavior:** Switches between dark and light mode. System preference is detected on first load.
**Feedback:** Immediate visual change. Preference persisted in localStorage for return visits.
**Keyboard:** Toggle is keyboard-focusable.
**Failure mode:** System preference used as fallback; no broken layout if localStorage unavailable.

## Navigation and Scrolling

The site is a single-page layout with section anchors. The header is persistent and provides jump links to major sections. Smooth scrolling is used for anchor navigation.

---

## Edge Cases and Failure Modes

| Scenario                        | Behavior                                                                 |
|---------------------------------|--------------------------------------------------------------------------|
| JavaScript disabled             | All content visible; interactive elements degrade gracefully to static   |
| Small screen (mobile)           | All sections reflow to single column; touch targets are large enough     |
| Slow network                    | Content-first render; interactive JS loads non-blocking                  |
| Screen reader                   | All interactive elements have ARIA labels and keyboard equivalents       |
| Wizard skipped / partially done | Output shows placeholders; no error state                                |
| Version badge source unavailable| Version shown as fallback string; no broken layout                       |

---

## Verification Criteria

- A first-time visitor can understand what RootSpec is from the hero and problem sections alone, without scrolling further
- The hierarchy explorer is navigable by keyboard without a mouse
- The wizard produces output for any non-empty input in step 1
- The before/after comparison never shows lorem ipsum or placeholder text
- The meta banner is visible without scrolling on all common screen sizes
- The theme toggle reflects system preference on first load
