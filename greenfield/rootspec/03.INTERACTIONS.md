# Level 3: Interaction Architecture
# RootSpec Marketing Site

---

## Overview

This site is a single-page marketing experience. Users arrive, scroll, and engage with interactive sections. There is no login, no persistence between sessions (except theme preference), and no server round-trips. The core interaction model is: **arrive → understand → explore → decide**.

---

## Interaction Loop Architecture

| Scale     | Scope                     | Duration        | Key Patterns                            |
|-----------|---------------------------|-----------------|------------------------------------------|
| Immediate | Single element interaction | Seconds          | Hover, click, toggle, type, select       |
| Session   | Full page visit            | Minutes          | Scroll, section reveal, wizard flow      |
| Extended  | Return visits              | Days/Weeks       | Theme preference (persisted), awareness  |
| Lifetime  | Framework adoption         | Ongoing          | Off-site: GitHub, docs, community        |

---

## Core Interaction Patterns

### Pattern 1: Linear Scroll with Sectioned Narrative

The page is divided into named sections. Users scroll top to bottom, encountering:
1. Meta Banner
2. Hero
3. Problem Statement
4. How It Works
5. Hierarchy Explorer (interactive)
6. Spec Wizard (interactive)
7. Before/After Comparison (interactive)
8. CTA / Footer

**Entry trigger:** Page load
**Feedback:** Section transitions animate in as they enter the viewport
**Exit:** Scroll continues naturally; no scroll hijacking

### Pattern 2: Theme Toggle

A persistent control (in the header) allows the user to switch between light and dark modes.

**Entry trigger:** User clicks the theme toggle button
**Feedback:** Immediate visual change across the full page
**Persistence:** Preference is stored in `localStorage` and applied on next visit
**Default:** System preference is read on first load

### Pattern 3: Hierarchy Explorer

An interactive visualization of the five RootSpec levels.

**Entry trigger:** User clicks or taps on a level card
**Feedback:**
- The clicked level expands to show example content
- Previously open level collapses
- Reference arrows between levels highlight the allowed reference directions
- Hovering a level highlights its allowed upstream references

**Keyboard support:** Arrow keys navigate between levels; Enter/Space expand/collapse

**Mobile:** Touch-friendly tap targets; same expand/collapse behavior without hover

### Pattern 4: Spec Your Idea Wizard

A three-step guided flow where the user maps their product idea to RootSpec L1–L3.

**Entry trigger:** User enters a product idea in the first step's input field and advances
**Steps:**
1. **What's the mission?** — User selects from mission templates or writes their own
2. **What should users feel?** — User picks [3–5] design pillar suggestions from a list
3. **Describe one key interaction** — Free text describing a core user action

**Output:** A formatted skeleton spec (L1–L3) is rendered below the wizard steps, showing how the inputs map to the hierarchy

**Feedback:** Each step shows a summary of prior choices before advancing; the output animates into view

**No AI, no API** — all output is generated from static templates on the client

**Keyboard support:** Tab through inputs; Enter to advance; Escape to reset

### Pattern 5: Before/After Comparison

A side-by-side view toggled by a slider or a tab/button pair.

**Entry trigger:** User interacts with slider or toggle control
**Feedback:** The active panel's content comes into focus; the inactive panel dims
**Content:** Real example content in both panels — "without spec" shows a vague requirements doc; "with RootSpec" shows the same subject structured as a hierarchy
**Mobile:** Stacked panels with a tab toggle rather than a side-by-side slider

## Failure Modes and Edge Cases

### Wizard incomplete input
If the user tries to advance without entering required input, the current step indicates what's missing and does not advance. No error modal — inline feedback only.

### Theme on first load
If `localStorage` is unavailable (private browsing), the system defaults to the OS preference. If the OS preference is unavailable, the default is dark mode.

### Reduced motion
All animations respect the `prefers-reduced-motion` media query. Section transitions and element reveals fall back to instant display.

### Narrow viewport on Hierarchy Explorer
On very narrow screens ([small breakpoint]), the level cards stack vertically. Reference arrows between levels are hidden; levels reference each other through numbered labels instead.

### JavaScript disabled
The page degrades gracefully — static content (hero, problem, how it works, CTA) is fully readable. Interactive sections show a minimal static fallback with a note that interactivity requires JavaScript.

## Navigation Model

- **Single page** — No routing; all sections are on one scrollable page
- **Anchor links** — Header may include anchor links to major sections for quick navigation
- **External links** — GitHub repo, SEED.md, and spec files open in new tabs (absolute URLs)
- **No internal page navigation** — All content is on the root page; the build deploys to a subpath (`/demos/greenfield/`)

---

## Verification Criteria

An interaction is correctly implemented when:
- The trigger activates the expected feedback within [a brief moment] of user action
- The feedback is visible and meaningful — users can confirm their action was received
- Keyboard navigation reaches every interactive element
- Touch targets meet minimum size standards on mobile
- No interaction requires an external API call to function
