# Level 3: Interaction Architecture

## Core Interaction Model

This is a single-page marketing and demo site. The visitor's journey is primarily read-linear (scrolling through sections) punctuated by two interactive excursions (hierarchy explorer, spec wizard) and one persistent contextual element (meta-banner). There is no login, no persistent state, and no multi-page navigation.

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Behavior |
|-------|-------|----------|--------------|
| Immediate | Single click / keypress | < 1 second | Button state, expand/collapse, toggle |
| Session | Full page visit | Minutes | Read through sections, try the wizard, explore hierarchy |
| Extended | Return visits | Across sessions | No cross-session state tracked; each visit is fresh |

## Section Flow

The page is read top-to-bottom. Section order is fixed and intentional:

1. **Meta Banner** — Persistent, above the fold, explains demo context before anything else
2. **Header** — Site identity and version badge; theme toggle
3. **Hero** — Tagline and one-sentence explanation; version badge
4. **Problem Section** — Why existing approaches fail
5. **How It Works** — Four-skill workflow walkthrough
6. **Hierarchy Explorer** — Interactive five-level visualization
7. **Spec Wizard** — "Spec Your Idea" three-step interactive tool
8. **Before/After Comparison** — Side-by-side contrast with toggle
9. **CTA Section** — Open Source CTA linking to framework repo
10. **Footer** — Attribution and build date

## Interaction Patterns

### Meta Banner

- **Trigger:** Page load
- **Behavior:** Renders at top of page, above all other content. Links to SEED.md and spec files on GitHub using absolute URLs. Does not dismiss.
- **Feedback:** None required (read-only)
- **Failure mode:** If links are broken, banner still displays with honest copy. No spinner, no loading state.

### Theme Toggle

- **Trigger:** User clicks toggle in header
- **Behavior:** Switches between light and dark mode. Default is light mode (system preference used as secondary signal, but light is the explicit default). Preference is saved in localStorage.
- **Feedback:** Immediate visual transition ([short duration] ease-out)
- **Failure mode:** If localStorage is unavailable, defaults to light mode each visit.
- **Keyboard:** Toggle is focusable and activatable via Enter/Space.

### Hierarchy Explorer

- **Trigger:** User clicks or tabs to a level card
- **Behavior:**
  - Clicking a level card expands it to show example content for that level
  - Hovering a level highlights which levels it is allowed to reference (arrows flow upward only)
  - Keyboard navigation: Tab moves between levels; Enter/Space expands/collapses; arrow keys navigate
  - Only one level is expanded at a time
- **Visual feedback:** Expand/collapse animation ([short duration] ease-out); reference arrows illuminate on hover/focus
- **Mobile:** Touch-friendly tap targets; swipe not required (tap to expand)
- **Failure mode:** If JS fails to load, the explorer renders as a static readable list (progressive enhancement)
- **Keyboard:** Full keyboard navigation required; each level is a focusable element

### Spec Wizard

- **Trigger:** User engages with the "Spec Your Idea" input
- **Behavior:**
  - Step 1: User enters a one-line product idea
  - Step 2: User selects or writes a mission statement (from templates or free text)
  - Step 3: User selects 3–5 design pillars (from suggestions or free text)
  - Step 4: User describes one key interaction
  - Output: A skeleton spec is rendered showing L1–L3 structure mapped to their inputs
- **Navigation:** Next/Back buttons step through the wizard; current step is indicated; progress is not lost on Back
- **Feedback:** Each step confirms what was captured before advancing. Output is rendered in a formatted, readable layout.
- **Mobile:** Each step is full-width; tap targets are appropriately sized
- **Failure mode:** Wizard is stateless client-side; if a step is invalid, the Next button is disabled with a brief error message. No data is sent anywhere.
- **Keyboard:** All controls (text inputs, multi-select chips, buttons) must be keyboard accessible. Tab order is logical.

### Before/After Comparison

- **Trigger:** Page scroll to section (no interaction required to see default state)
- **Behavior:** Two panels are shown side-by-side on desktop, stacked on mobile. A toggle or tab control switches between "Without spec" and "With RootSpec" views on mobile. On desktop both panels are always visible.
- **Content:** Real content, not lorem ipsum. Concrete example of a vague requirements doc vs. a structured RootSpec hierarchy.
- **Feedback:** Toggle switches immediately ([short duration])
- **Failure mode:** Content is readable without the toggle. Both panels have semantic structure.

### Version Badge

- **Trigger:** Page load
- **Behavior:** Reads RootSpec version from `.rootspec.json` at build time and renders it in the hero section and/or header. Version is static at build time.
- **Failure mode:** If version is unavailable, renders "v—" or omits the badge. Never shows an error state to the user.

### External Links (CTA, Banner)

- **Trigger:** User clicks a link
- **Behavior:** External links (GitHub repo, spec files) open in a new tab. Internal anchor links scroll to section smoothly.
- **Failure mode:** If GitHub is unreachable, the link still renders correctly. No prefetching or liveness checks.

## Failure Modes and Edge Cases

| Scenario | Behavior |
|----------|----------|
| JavaScript disabled | Meta-banner, hero, problem, workflow, CTA, footer all render as static HTML. Interactive components render a fallback message. |
| Small screen (< 375px width) | All layouts stack to single column; interactive elements remain functional |
| High contrast / forced colors | All interactive states must be distinguishable without relying solely on color |
| Very long user input in wizard | Input is truncated at [reasonable character limit]; overflow is handled gracefully |
| User opens external GitHub links | Opens in new tab; does not disrupt page state |
| User navigates away mid-wizard | No confirmation required; wizard state is discarded |

## Animation and Motion Philosophy

- All transitions: [short duration] (150–250ms) ease-out
- No parallax, no spring animations, no reveal-on-scroll effects
- Interactive state changes (hover, focus, expand) are the only animated elements
- `prefers-reduced-motion`: all animations disabled; state changes are instant

## Accessibility Requirements

- All interactive elements keyboard reachable
- Focus ring always visible when navigating by keyboard
- Color is never the sole means of conveying information
- Semantic HTML: headings in correct order, buttons for actions, links for navigation
- ARIA roles for the hierarchy explorer (tree or tablist pattern) and wizard (step indicator)
- Screen reader-compatible: all interactive state changes are announced via ARIA live regions where needed
