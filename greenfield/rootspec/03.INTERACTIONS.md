# Level 3: Interaction Architecture

**References:** 01.PHILOSOPHY.md, 02.TRUTHS.md

---

## Core Interaction Model

This is a single-page marketing site. The primary interaction model is linear scrolling discovery with optional interactive deep-dives. Visitors arrive, scan the page top-to-bottom, and optionally engage with interactive sections. No navigation to other pages, no login required.

### Interaction Scale Architecture

| Scale     | Scope                        | Duration       | Key Pattern                             |
|-----------|------------------------------|----------------|-----------------------------------------|
| Immediate | Single element interaction   | <1 second      | Click/tap → visual feedback             |
| Session   | Single page visit            | 1–10 minutes   | Scroll → discover → interact → decide   |
| Extended  | Return visits / sharing      | Days to weeks  | Site referenced after off-site research |

## Primary Interaction Loops

### Loop 1: Discovery (passive)

**Trigger:** Visitor lands on page (direct link, referral, search)
**Flow:**
1. Meta-banner is immediately visible → visitor understands demo context
2. Hero section communicates tagline and purpose in one glance
3. Problem section resonates with known pain points
4. "How It Works" walkthrough creates familiarity with the four-skill model
5. Interactive sections invite engagement

**Feedback:** Visual hierarchy and scroll animations guide attention without requiring interaction. Each section transitions smoothly into the next.

**Exit points:** Visitor clicks GitHub link, clicks "View spec →" or "View seed →" in meta-banner, or closes tab.

### Loop 2: Hierarchy Exploration (active)

**Trigger:** Visitor reaches the Hierarchy Explorer section and clicks or taps a level
**Flow:**
1. Five levels displayed in visual hierarchy (L1 at top, L5 at bottom)
2. User clicks a level → it expands to show example content and description
3. Reference arrows animate to show which levels this level can cite
4. Hovering/focusing a level highlights its allowed references
5. User clicks a different level → previous collapses, new one expands

**Feedback:**
- Smooth expand/collapse animation on level click
- Arrow highlighting shows allowed reference direction instantly
- Keyboard navigation (Tab to focus, Enter/Space to toggle) works identically to mouse

**Edge cases:**
- On mobile, arrows simplify to a directional indicator (upward-only label)
- Multiple levels cannot be open simultaneously (one active at a time)
- Initially, no level is expanded — the collapsed state shows level names only

### Loop 3: Spec Wizard (active, multi-step)

**Trigger:** Visitor reaches the "Spec Your Idea" section and begins Step 1
**Flow:**
1. **Step 1 — Mission:** User enters a one-line product idea or selects a template suggestion. Typing is free-form; suggestions are offered as clickable chips.
2. **Step 2 — Design Pillars:** User selects 3–5 design pillars from a curated list of suggestions, or writes their own. Selection state is visible (highlighted chips).
3. **Step 3 — Key Interaction:** User describes one key interaction their product enables. Free-form text input.
4. **Output:** A skeleton spec appears showing L1 (mission from Step 1), L1 (pillars from Step 2), and L3 (interaction from Step 3). The output is formatted like a real spec excerpt.

**Navigation:**
- Forward: "Next →" button, enabled only when current step has valid input
- Backward: "← Back" button returns to previous step, preserving entered values
- Restart: "Start over" resets all state to Step 1

**Feedback:**
- Active step is visually highlighted in a step indicator
- Input validation is soft — the "Next" button stays disabled until something is entered, but no error messages
- Output appears with a brief animation (fade or slide in) to signal completion

**Edge cases:**
- No data is sent anywhere — all state is in memory, lost on page refresh
- On mobile, the wizard is vertically stacked; step indicator is horizontal at the top
- If user clears all input in Step 2 (deselects all chips), "Next" re-disables

### Loop 4: Before/After Comparison (active)

**Trigger:** Visitor reaches the comparison section
**Flow:**
1. Two panels displayed side-by-side (desktop) or stacked (mobile)
2. User can toggle between "Without RootSpec" and "With RootSpec" views, OR use a drag slider (desktop) to reveal both simultaneously
3. Content in both panels is real — not lorem ipsum

**Feedback:**
- Toggle switches states instantly with a brief crossfade
- Drag slider gives immediate visual feedback as cursor moves
- On touch, swipe replaces drag; tab-accessible toggle always present

**Edge cases:**
- On mobile: slider is replaced by a toggle button (swipe gesture is secondary)
- Both panels are readable in isolation — slider/toggle is enhancement, not requirement

### Loop 5: Theme Toggle (persistent preference)

**Trigger:** User clicks the theme toggle in the header
**Flow:**
1. Toggle switches between dark and light modes
2. Preference is saved to localStorage
3. On next visit, saved preference is applied before first paint (no flash)

**Feedback:** Smooth CSS transition on color scheme change (no jarring flash). System preference is detected on first visit if no saved preference exists.

## Navigation Patterns

- **No page-level navigation** — single page, scroll-based
- **Anchor links** — the header (if present) links to major sections via smooth scroll
- **External links** — GitHub links open in new tab with appropriate rel attributes
- **Back button** — browser back from any GitHub link returns to this page (no SPA routing complications)

## Feedback and State Model

| Action                          | Immediate Feedback           | Persistent State         |
|---------------------------------|------------------------------|--------------------------|
| Click hierarchy level           | Expand animation             | None (resets on refresh) |
| Complete wizard step            | Step indicator advances      | None (resets on refresh) |
| Toggle comparison               | Panel crossfade              | None (resets on refresh) |
| Toggle theme                    | Color scheme transition      | localStorage preference  |
| Click external link             | New tab opens                | None                     |

---

## Failure Modes and Edge Cases

- **No JavaScript:** Core content (hero, problem, how-it-works) must be readable. Interactive components degrade gracefully — showing static fallback content or a minimal representation.
- **Slow connection:** Static content loads first; interactive components hydrate after. No spinner or loading state should block page comprehension.
- **Very small screens (<320px):** Layout may compress but must not overflow or produce horizontal scroll.
- **Keyboard-only navigation:** All interactive elements (hierarchy levels, wizard steps, comparison toggle, theme toggle) are reachable and operable via keyboard in logical tab order.
- **High-contrast mode:** Text must remain readable against background in both dark and light themes at any contrast preference.

---

## Verification Criteria

1. A user who does not interact with any interactive element still receives the full core message
2. The hierarchy explorer can be fully navigated using only Tab, Enter, and Space keys
3. The spec wizard produces a real-looking spec skeleton from any non-empty input
4. The before/after comparison communicates its contrast without the slider (toggle alone is sufficient)
5. The theme toggle preference persists across page refreshes
