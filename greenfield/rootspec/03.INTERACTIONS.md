# Level 3: Interaction Architecture

## Interaction Loop Architecture

| Scale     | Scope                    | Duration     | Key Patterns                        |
|-----------|--------------------------|--------------|-------------------------------------|
| Immediate | Single element interaction | Seconds    | Click, hover, toggle, expand        |
| Session   | Full site visit          | Minutes      | Scroll narrative, try interactives  |
| Extended  | Return visit / sharing   | Days/Weeks   | Share link, revisit for reference   |
| Lifetime  | Adoption                 | Ongoing      | Use RootSpec, refer others          |

## Core Journey: First Visit

The site is a single downward narrative. The visitor scrolls through a structured argument:

1. **Arrival** — Hero section delivers the tagline and one-sentence explanation. Immediate clarity about what this is. Version badge visible.
2. **Recognition** — Meta banner reveals the site is itself a RootSpec demo. Links to spec and seed files. This is the credibility hook.
3. **Problem** — Names the specific frustrations: spec drift, philosophy-implementation gap, dead documentation, unreliable AI output. Visitor feels seen.
4. **Solution** — How It Works section walks through the four skills (init → spec → impl → validate) with clear before/after visual. The methodology becomes concrete.
5. **Experience** — Interactive features let the visitor try the methodology hands-on (hierarchy explorer, spec wizard, before/after comparison).
6. **Action** — Open source CTA with GitHub link and getting started instructions.

### Scroll Behavior

The narrative is designed for continuous scrolling. Each section transitions smoothly into the next. Section entry animations provide rhythm but never block content. The visitor should never feel lost — a persistent indicator shows where they are in the narrative.

## Interactive Pattern: Hierarchy Explorer

**Trigger:** Visitor reaches the hierarchy section or clicks to explore.

**Interaction loop:**
1. Five levels displayed in a visual hierarchy (vertical stack or tree)
2. Click a level to expand — shows example content for that level
3. Visual connections (lines or arrows) show reference rules — arrows flow upward only
4. Hover a level to highlight its allowed references (which levels it can see)
5. Only one level expanded at a time — clicking another collapses the current one

**Feedback:** Expansion is animated. Reference arrows illuminate on hover. The constraint (no downward references) is visually obvious — you can SEE the one-way flow.

**Failure mode:** If JavaScript fails, show all five levels statically expanded with a text description of reference rules.

## Interactive Pattern: Spec Wizard

**Trigger:** Visitor reaches the wizard section or clicks "Spec Your Idea."

**Interaction loop:**
1. **Step 1: Mission** — Text input: "What's your product idea?" Plus template suggestions to pick from
2. **Step 2: Design Pillars** — Pick 3-5 emotional pillars from suggestions, or write custom ones. Suggestions adapt based on the idea entered.
3. **Step 3: Key Interaction** — Describe one core user interaction in structured form (who, what trigger, what feedback)

**Output:** A skeleton spec card showing how their input maps to L1-L3. The output is visual and immediate — the visitor sees structured spec content generated from their three answers.

**Constraints:** Client-side only, no AI. Uses templates and structured mapping. The goal is to demonstrate the methodology is approachable, not to produce a production spec.

**Failure mode:** If JavaScript fails, show a pre-filled example spec card with a note that the interactive version requires JavaScript.

## Interactive Pattern: Before/After Comparison

**Trigger:** Visitor reaches the comparison section.

**Interaction loop:**
1. Two panels side by side (or stacked on mobile)
2. Slider or toggle switches between "Without spec" and "With RootSpec"
3. **Without:** Vague requirements doc, ambiguous user stories, untraceable decisions — real content, not lorem ipsum
4. **With:** Structured hierarchy, testable stories, every feature traces to a design pillar — matching real content

**Feedback:** Transition between states is smooth. The contrast is self-evident — the "with" panel is visibly more structured and traceable.

## Interaction Pattern: Theme Toggle

**Trigger:** Visitor clicks theme toggle or system preference changes.

**Behavior:**
- Defaults to system preference (prefers-color-scheme)
- Manual toggle overrides system preference for the session
- Toggle is always visible and accessible
- Transition between themes is smooth, not jarring

## Interaction Pattern: Version Badge

**Behavior:** Current RootSpec version (e.g., v6.2.1) displayed in the hero or header. Static display — not interactive. Updated from a single constant.

## Edge Cases and Failure Modes

- **JavaScript disabled:** Core content (text, images) renders. Interactive features show static fallbacks with a note.
- **Slow connection:** Content loads progressively. Interactive features initialize after core content is visible.
- **Small screens:** All interactive features adapt to touch. Hierarchy explorer uses vertical accordion. Wizard uses full-width steps. Before/after uses stacked panels with toggle.
- **Screen readers:** All interactive elements have ARIA labels. Hierarchy relationships described textually. Wizard steps announced on transition.
- **Keyboard navigation:** Tab through all interactive elements. Enter/Space to activate. Escape to collapse. Arrow keys within the hierarchy explorer.

## Verification Criteria

- A visitor can understand RootSpec's core value proposition without interacting with any interactive feature (content-first)
- The hierarchy explorer makes the five-level structure and reference rules intuitive within [brief interaction]
- The spec wizard produces a recognizable spec skeleton from three inputs
- The before/after comparison creates an immediate "aha" about the value of structure
- Theme toggle works without page reload or layout shift
- All interactive features are usable via keyboard alone
