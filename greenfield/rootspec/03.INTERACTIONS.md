## Interaction Loop Architecture

| Scale     | Scope                         | Duration     | Key Patterns                              |
|-----------|-------------------------------|--------------|-------------------------------------------|
| Immediate | Single section interaction    | Seconds      | Scroll, click, expand, toggle             |
| Session   | Full page read + exploration  | Minutes      | Progressive disclosure, wizard completion |
| Extended  | Return visits, sharing        | Days/weeks   | Bookmarking, sharing spec links           |
| Lifetime  | Adoption and advocacy         | Ongoing      | Referral, community, repo engagement      |

---

## Core Interaction Patterns

### Pattern 1: Linear Scroll with Anchored Sections

The primary interaction is reading. Visitors scroll through sections in order. Each section is self-contained and legible without the others, but the sections build an argument from top to bottom:

1. Meta-banner (trust frame)
2. Hero (what/why)
3. Problem section (resonance)
4. How it works (solution)
5. Hierarchy explorer (tangible demo)
6. Spec wizard (personal application)
7. Before/after comparison (proof)
8. Open source CTA (conversion)
9. Footer (attribution)

The visitor does not need to complete the scroll. Each section is designed to be a stopping point where they've received value.

### Pattern 2: Progressive Disclosure (Hierarchy Explorer)

The hierarchy explorer reveals complexity incrementally. The five levels are visible at a glance. Clicking a level expands it to show example content and its allowed references. Arrows flow upward only, visualizing the dependency inversion principle.

**Trigger:** Visitor clicks a level name or label
**Feedback:** Level expands; example content appears; reference arrows highlight
**Edge cases:**
- Keyboard navigation (arrow keys or tab) navigates between levels
- Touch: tap to expand, tap again to collapse
- All five levels can be simultaneously expanded on large screens; mobile defaults to one-at-a-time

### Pattern 3: Guided Form Flow (Spec Wizard)

The wizard is a three-step guided form. Each step builds on the previous. The output is a skeleton spec.

**Step 1:** Product idea (one-line text input)
**Step 2:** Mission selection (template options + write-your-own)
**Step 3:** Design pillars (multi-select from suggestions)

**Output:** A rendered skeleton showing how their input maps to L1-L3. The output is display-only — no save, no export.

**Trigger:** Visitor enters their product idea and clicks Next
**Feedback:** Progress indicator advances; previous step summary appears above
**Edge cases:**
- Empty step 1: validation message, cannot proceed
- Write-your-own mission: freeform text input appears
- Fewer than [minimum pillars] selected: soft warning, not a hard block
- Back navigation: previous step restores filled values

### Pattern 4: Binary Comparison (Before/After)

The before/after comparison presents two states side by side (or toggled on mobile). Visitors can switch between "without spec" and "with RootSpec" views. Real content in both panels.

**Trigger:** Visitor clicks toggle or moves slider
**Feedback:** Content transitions [quickly]; active panel is visually distinct
**Edge cases:**
- Mobile: toggle button replaces slider
- Keyboard: toggle is keyboard-accessible

### Pattern 5: Theme Toggle

A manual light/dark theme toggle in the header. System preference is detected on first load. The preference is persisted for return visits.

**Trigger:** Visitor clicks the theme toggle icon
**Feedback:** Page transitions to the alternate theme [quickly]
**Edge cases:**
- System preference changes while page is open: no automatic update (manual toggle only after initial load)
- Preference persisted in localStorage

## Section Interaction Details

### Meta Banner

Always visible at the top of the page. Contains links to the SEED.md and spec files on GitHub. The banner is informational — no interactive state beyond the links.

**Failure mode:** If GitHub links are unreachable, the banner text still communicates the demo context. Links are not required for the banner's primary purpose.

### Hero

Displays the RootSpec version badge (read from `.rootspec.json` at build time). Contains the primary tagline and one-sentence explanation. No interactive elements beyond the nav toggle.

### Navigation

Optional sticky header with the version badge and theme toggle. Section navigation via anchor links.

### Footer

Identifies the site builder (Claude, built by Anthropic), the build date, and the framework version. No interactive elements.

## Failure Modes

| Failure | User Experience | Recovery |
|---------|----------------|----------|
| JavaScript disabled | Wizard and explorer are non-functional | Static content remains readable; informational sections work fully |
| Slow connection | Content loads progressively; hero is text-only | No spinners; content appears as it loads |
| Mobile touch | Slider-based comparison becomes toggle-based | Touch targets are [adequately sized] |
| Keyboard-only | All interactive elements are tab-navigable | Focus indicators are visible; escape collapses expanded elements |
| GitHub links unreachable | Banner links are dead | Banner text still frames the demo context |

## Verification Criteria

An implementation satisfies this interaction architecture when:
- A visitor can reach the CTA without interacting with any interactive element (pure scroll path)
- The hierarchy explorer communicates the five-level structure without external explanation
- The wizard produces a skeleton spec from any one-line product idea
- The before/after comparison shows real content (no lorem ipsum) in both panels
- The theme toggle persists preference across page reload
- All interactive elements are operable by keyboard
- The meta-banner is the first content element a visitor reads
