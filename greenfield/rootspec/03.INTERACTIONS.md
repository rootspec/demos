# Level 3: Interaction Architecture

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Interaction |
|-------|-------|----------|----------------|
| Immediate | Single element interaction | Seconds | Click/tap to expand, toggle, or step through wizard |
| Session | Full page reading and exploration | Minutes | Read → explore hierarchy → try wizard → click CTA |
| Extended | Return visit or shared link | Days | Share a specific section; return after trying the framework |

## Core User Journeys

### Journey 1: The Skeptical Reader

**Trigger:** Developer arrives via search, social share, or colleague recommendation. They are skeptical of "another methodology."

1. **Land** — Meta-banner is the first thing seen. It says: "this site was built from a ~100-line description using four commands." The reader understands immediately this is a demonstration, not a pitch.
2. **Orient** — Hero communicates the core proposition in one read. The problem section validates the reader's existing frustration (spec drift, AI output without validation).
3. **Understand** — The hierarchy explorer lets them click through the five levels and feel the structure. Visual connections show upward-only references. This is the methodology made tangible.
4. **Evaluate** — The comparison section shows the before/after contrast with real content. The methodology diagram shows how the spec surrounds the development cycle and gates output.
5. **Act** — CTA section links to the GitHub repo and npm package. The spec wizard offers a low-friction first taste.

**Success signal:** Reader opens the GitHub repo or runs `/rs-init` in their own project.

### Journey 2: The Experimenter

**Trigger:** A visitor wants to understand what the spec output actually looks like before committing.

1. **Arrive** → read meta-banner and hero
2. **Find wizard** — The "Spec Your Idea" section invites them to enter a one-line product idea
3. **Step through** — Three steps: mission, design pillars, key interaction. Each step produces visible structured output
4. **See result** — A skeleton spec mapping their input to L1-L3 renders inline. The methodology feels real and approachable
5. **Continue** — Links to the GitHub repo and docs are the natural next step

**Success signal:** Reader completes all three wizard steps and views the generated skeleton.

### Journey 3: The Team Lead Evaluating Tooling

**Trigger:** A technical lead researching alternatives to Jira + Confluence for spec management.

1. **Arrive** → meta-banner confirms this is a demo, lending credibility
2. **Read the problem section** — Scope drift, unread docs, philosophy-implementation gap resonate
3. **Examine how it works** — The four-skill workflow is concrete and actionable
4. **Review the comparison** — Before/after shows the specific difference in structure and traceability
5. **Assess fit** — The CTA section provides the path to the framework, docs, and community

**Success signal:** Reader clicks through to the GitHub repo or shares the link with their team.

## Interaction Patterns

### Meta-Banner
- Always visible, above all content on every device
- Static text with two links: spec files and seed file (both absolute GitHub URLs)
- No interaction beyond link clicks
- Does not dismiss or collapse

### Theme Toggle
- **Trigger:** User clicks theme icon in header
- **Response:** Page transitions between light and dark mode in [short transition duration]
- **State:** Preference persists in localStorage; system preference is detected on first visit
- **Feedback:** Icon changes to reflect current mode

### Version Badge
- Displays current RootSpec version (read from `.rootspec.json` at build time)
- Visible in hero section and/or site header
- Static, no interaction

### Hierarchy Explorer
- **Entry:** Five level cards displayed in a vertical or stacked layout
- **Expand:** Click/tap a level to expand it and see example content and description
- **Reference lines:** Visual connections show which levels each can reference (upward only); hovering a level highlights its allowed references
- **Collapse:** Click again to collapse; only one level is expanded at a time (or all can coexist — implement whichever feels more tool-like)
- **Keyboard:** Arrow keys navigate between levels; Enter/Space expands/collapses; focus ring visible
- **Mobile:** Touch-friendly tap targets; no hover-only interactions

### Spec Wizard
- **Entry:** Single text input — "Enter a one-line product idea"
- **Step 1:** Mission selection — user picks from templates or writes their own
- **Step 2:** Design pillar selection — user picks 3-5 from suggestions
- **Step 3:** Key interaction description — free text
- **Output:** Skeleton spec rendered inline, showing L1-L3 structure populated from their input
- **Navigation:** Back/next buttons; keyboard navigable; no form submission or network calls
- **Mobile:** Single-column layout, large tap targets

### Comparison View
- **Layout:** Side-by-side (desktop) or stacked toggle (mobile)
- **Content:** Real content in both panels — not lorem ipsum
  - Without spec: vague requirements doc, ambiguous stories, untraceable decisions
  - With RootSpec: structured hierarchy, testable stories, features tracing to design pillars
- **Interaction:** Toggle switch or slider to switch between views on mobile; both visible simultaneously on desktop

### Methodology Diagram
- SVG — no raster images
- Shows: spec surrounding the development cycle; validation gate allowing only valid solutions through
- Considered linework — not a Visio export or clip-art illustration
- Static on load; no animation required

### CTA Section
- Primary links: GitHub repo, npm package, getting-started docs
- Secondary: Community links if available
- No email capture, no account creation, no waitlist

## Failure Modes and Edge Cases

- **JavaScript disabled:** Core content (meta-banner, hero, problem, how it works, CTA) is fully readable. Interactive components (hierarchy explorer, wizard) degrade gracefully with a static fallback message.
- **Slow network:** Text content loads first; interactive JS components load progressively
- **Small screens (< 360px):** Content remains readable; interactive elements stack vertically
- **Keyboard-only navigation:** All interactive elements reachable and operable via keyboard alone
- **Base path misconfiguration:** If assets resolve to the wrong path, the site fails visibly (404s) rather than silently. This is detectable and fixable.

## Verification Criteria

- Meta-banner is the first visible element on all viewport sizes
- Theme toggle changes the page appearance and persists across reload
- All five hierarchy levels are explorable without reading any documentation
- The wizard produces a skeleton spec from a one-line input through three steps
- The comparison view shows real, substantive content in both states
- All interactive elements are operable by keyboard alone
- All links use absolute GitHub URLs, not relative paths
- The version badge reflects the actual value in `.rootspec.json`
