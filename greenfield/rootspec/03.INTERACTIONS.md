# Level 3: Interaction Architecture

**Product:** RootSpec Marketing Site
**Level:** L3 — HOW Users and Product Interact
**References:** L1-2 + External

---

## Core User Journey

### First Contact (Immediate Loop)

A visitor arrives — likely from a link shared by a developer colleague or a search for spec tooling alternatives.

1. The **meta banner** is the first thing they read: an honest statement that this site was generated from a ~100-line description using four commands
2. They see the **hero**: tagline, one-sentence explanation, version badge, and a visual of what RootSpec is
3. They understand the premise within [brief moment] and decide whether to continue reading

**Trigger:** Arrival at the site
**Feedback:** Immediate comprehension of what the site is and what it's honestly claiming

### Conceptual Understanding (Session Loop)

The visitor reads or scrolls through sequential sections that build their understanding:

1. **Problem section** — Validates their pain with current approaches
2. **How It Works** — The four skills (init, spec, impl, validate) as a workflow
3. **Interactive: Hierarchy Explorer** — Hands-on exploration of the five levels
4. **Interactive: Before/After Comparison** — Contrast with and without a spec
5. **Interactive: Spec Wizard** — Trying the methodology on their own idea

**Trigger:** Curiosity after understanding the premise
**Feedback:** Progressive comprehension; each section resolves a specific question the previous section raised

### Action (Transition Loop)

The visitor reaches a decision point:

- They follow the CTA to the GitHub repo for RootSpec
- They share the link with a colleague
- They return later to start their own spec

**Trigger:** Sufficient understanding to act
**Feedback:** Clear, singular CTA — no competing actions on the page

## Hierarchy Explorer Interaction

The hierarchy explorer is the primary teaching mechanism for the five-level model.

**Pattern:** Expandable tree with directional relationship visualization

1. User sees all five levels displayed simultaneously as a vertical list or diagram
2. User clicks (or tabs to) any level — it expands to show example content for that level
3. Arrows or connectors show which levels each can reference (upward only)
4. Hovering or focusing a level highlights its allowed reference targets
5. User can collapse any level by clicking again or pressing Escape

**Keyboard behavior:** Arrow keys navigate between levels; Enter/Space toggles expansion; Escape collapses
**Touch behavior:** Tap to expand/collapse; no hover states required on touch devices
**Feedback:** Smooth expand/collapse animation; reference highlights appear within [brief delay]

**Failure modes:**
- JavaScript disabled: show all levels in a static expanded state
- Very small screen: levels stack vertically; arrows become simplified indicators

## Spec Wizard Interaction

The spec wizard demonstrates that starting a RootSpec spec is approachable.

**Pattern:** Linear three-step form with structured choices

**Step 1 — Mission:** User enters a one-line product idea (free text, [short length] max)
**Step 2 — Design Pillars:** User selects [pillar_count_min] to [pillar_count_max] emotional pillars from a list of suggestions, or writes their own
**Step 3 — Key Interaction:** User describes one core user interaction in free text

**Output:** A skeleton spec preview showing how their inputs map to L1 (mission, pillars) and L3 (interaction). Each output field is labeled with its spec level.

**Navigation:** Back/Next buttons; step indicator shows current position; final step has a "See your skeleton spec" action
**Keyboard behavior:** Tab navigates between inputs; Enter submits each step
**Feedback:** Skeleton spec appears below (or replaces) the wizard on completion

**Constraints:**
- No API calls — all output is template-based, generated client-side
- No data is persisted — refreshing resets the wizard
- The output is clearly labeled as a skeleton, not a complete spec

**Failure modes:**
- Empty required field: highlight field and show inline message before advancing
- Very long input: truncate gracefully in the output preview

## Before/After Comparison Interaction

**Pattern:** Side-by-side toggle or slider comparison

- **Without spec panel:** Shows a vague requirements doc fragment, ambiguous story, no traceability
- **With RootSpec panel:** Shows the structured equivalent — spec level label, testable criterion, pillar reference

**Toggle mode (mobile):** A binary switch between "Without" and "With" views
**Slider mode (desktop, optional):** A drag handle reveals more of one panel or the other
**Content:** Real examples from the RootSpec methodology — not lorem ipsum

---

## Theme Toggle Interaction

**Pattern:** System preference detection with manual override

- On first visit, theme matches system preference (`prefers-color-scheme`)
- A visible toggle button (in the header or accessible corner) allows manual switch
- Manual preference persists via `localStorage` for the session and future visits

**Feedback:** Theme transitions with a [very brief] crossfade — not an instant flash
**Keyboard behavior:** Toggle is reachable and activatable via keyboard
**Failure mode:** If system preference is unavailable, default to light theme

---

## Navigation and Layout Behavior

- Single-page layout — all sections on one scrollable page
- No routing, no page loads, no navigation that leaves the page
- Section anchors may be present for deep linking from the meta banner (e.g., `#hierarchy-explorer`)
- The meta banner is always at the top, above the fold on all screen sizes

## Edge Cases and Failure Modes

| Scenario                        | Expected Behavior                                              |
|---------------------------------|----------------------------------------------------------------|
| JavaScript disabled             | Meta banner, hero, problem, how-it-works render as static HTML |
| Very narrow viewport (< [min_width]) | All sections reflow to single-column; touch-friendly targets |
| User skips wizard steps         | Cannot advance without completing required field              |
| `.rootspec.json` missing at build | Build warns and shows a fallback version indicator           |
| External GitHub links unreachable | Links still render correctly; no client-side link checking  |
