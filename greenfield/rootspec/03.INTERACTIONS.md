# Interactions (L3)

*References: [[01.PHILOSOPHY]] [[02.TRUTHS]]*

## Core User Journey

**First 30 seconds**: User lands on hero, sees tagline and one-sentence explanation, notices meta-banner explaining this is a live RootSpec demo with links to spec/seed files.

**Minutes 1-2**: User scrolls through problem section, recognizes pain points from their experience with spec drift and unreliable AI output.

**Minutes 2-3**: User views "How It Works" walkthrough, sees visual representation of init → spec → impl → validate workflow.

**Minutes 3-5**: User interacts with hierarchy explorer, clicking levels to see example content and understanding reference flow (arrows pointing upward only).

**Minutes 5-7**: User tries spec wizard with their own product idea, walks through mission → design pillars → key interaction, sees skeleton spec output.

**Minutes 7-10**: User explores before/after comparison with slider/toggle, compares vague requirements vs structured hierarchy.

**Exit decision**: User either clicks GitHub link for repository access or leaves with clear understanding of RootSpec methodology.

## Interaction Patterns

### Primary Actions
- **Explore hierarchy**: Click/tap levels in interactive diagram to expand content and highlight allowed references
- **Wizard progression**: Step through 3-stage product ideation with structured templates
- **Compare approaches**: Toggle or slide between "without spec" vs "with RootSpec" examples
- **Access source**: Click meta-banner links to view spec files and SEED.md in repository

### Navigation
- **Scroll-based progression**: Natural top-to-bottom flow through sections
- **Anchor jumping**: In-page links for quick section access
- **Theme switching**: Manual toggle or automatic system preference detection
- **External linking**: GitHub repository, getting started resources

### Feedback Mechanisms
- **Visual state**: Clear indication of active/selected items in hierarchy explorer
- **Progress indication**: Wizard shows current step and completion status
- **Hover affordances**: Interactive elements show hover states on desktop
- **Focus management**: Keyboard navigation with visible focus indicators

## Interaction Flows

### Hierarchy Explorer
1. **Initial state**: All 5 levels visible with clear labels (Philosophy, Truths, Interactions, Systems, Implementation)
2. **Level selection**: Click/tap expands level to show example content, highlights allowed reference arrows
3. **Reference visualization**: Arrows flow upward only, highlighting which levels each can reference
4. **Multiple selection**: Can expand multiple levels simultaneously for comparison
5. **Reset**: Clear action to collapse all levels and return to overview

### Spec Wizard
1. **Product input**: Single text field for one-line product idea
2. **Mission clarification**: Choose from template missions or write custom
3. **Design pillars**: Select 3-5 from suggested emotional phrases or add custom
4. **Key interaction**: Describe one primary user workflow
5. **Output generation**: Show skeleton spec mapping input to L1-L3 levels
6. **Next steps**: Link to full RootSpec documentation for implementation

### Before/After Comparison
1. **Baseline view**: Show "without spec" panel with vague requirements example
2. **Toggle mechanism**: Slider or button to switch between states
3. **Contrast state**: Show "with RootSpec" panel with structured hierarchy
4. **Synchronized content**: Both panels address same product concept
5. **Highlight differences**: Visual emphasis on traceability and structure

## Error States

**Explorer malfunction**: Level won't expand → Show reload prompt with explanation
**Wizard incomplete**: User tries to advance without required input → Highlight missing fields with helpful prompts
**Comparison stuck**: Toggle mechanism unresponsive → Provide manual state buttons
**External link failure**: GitHub repository unreachable → Show alternative getting-started content
**Mobile layout issues**: Touch targets too small → Ensure minimum 44px tap targets throughout
**Keyboard trap**: Focus gets stuck in interactive element → Implement proper focus management with escape routes

## Accessibility Requirements

**Keyboard navigation**: All interactive elements reachable via Tab, Enter, Space, arrow keys
**Screen reader support**: Proper ARIA labels, landmarks, and live regions for dynamic content
**Visual indicators**: Clear focus outlines, sufficient color contrast, no color-only information
**Touch accessibility**: Minimum 44px tap targets, proper spacing between interactive elements
**Motion sensitivity**: Respect prefers-reduced-motion for users sensitive to animation
**Progressive enhancement**: Core functionality works without JavaScript, enhanced with scripting
