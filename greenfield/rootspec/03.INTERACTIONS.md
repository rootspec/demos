# Level 3: Interaction Architecture
# RootSpec Marketing Site

## Core Interaction Model

The site is a single long-scrolling page with distinct sections. The primary interaction model is **progressive disclosure**: visitors scan down, each section answering the next natural question. Interactive components interrupt the scroll with hands-on engagement. Navigation is section-anchor based.

## Interaction Loop Architecture

| Scale     | Scope                           | Duration   | Key Pattern                              |
|-----------|---------------------------------|------------|------------------------------------------|
| Immediate | Single click / hover            | Seconds    | Visual feedback, expand/collapse         |
| Session   | Full page read or wizard use    | 2-10 min   | Progressive understanding, state held    |
| Extended  | Return visit after trying tool  | Days       | Deeper engagement via GitHub/docs        |
| Lifetime  | Adoption into workflow          | Ongoing    | Spec methodology becomes part of process |

## Section Flow (Top to Bottom)

### 1. Meta Banner
**Trigger:** Page load
**Interaction:** Static read, with two links (View spec →, View seed →)
**Feedback:** Links open in new tab to GitHub
**Constraint:** Banner must be visible without scrolling on all viewports; cannot be dismissed

### 2. Hero
**Trigger:** Scroll past banner
**Interaction:** Read tagline + one-sentence explanation; observe version badge
**Feedback:** Version badge reads from build-time config — a live artifact
**Secondary:** CTA button scrolls to "How It Works" section

### 3. The Problem
**Trigger:** Continued scroll
**Interaction:** Read — no interactive elements; purely communicative
**Feedback:** None required; content is the message
**Goal:** Visitor recognizes their own situation in the described pain points

### 4. How It Works
**Trigger:** Continued scroll
**Interaction:** Read the four-skill walkthrough (init → spec → impl → validate)
**Feedback:** Visual indicators show progression through skills
**Goal:** Visitor understands the process shape before seeing the detail

### 5. Hierarchy Explorer (Interactive)
**Trigger:** Scroll into section or CTA from How It Works
**Interaction:**
- Click a level to expand it and see example content
- Hover a level to highlight allowed reference directions
- All five levels visible simultaneously; one expanded at a time
- Tab/keyboard navigation moves between levels
**Feedback:**
- Clicked level expands with smooth animation
- Reference arrows animate to show allowed/forbidden directions
- Previously expanded level collapses (accordion pattern)
**Failure modes:**
- If JavaScript is disabled, all levels show collapsed with static descriptions
- Touch devices: tap replaces hover; swipe within expanded content supported

### 6. Spec Your Idea Wizard (Interactive)
**Trigger:** CTA "Try it yourself" or scroll into section
**Interaction flow:**
1. Step 1 — Enter one-line product idea (free text input, max [short text length])
2. Step 2 — Select mission style (3-4 template options or write your own)
3. Step 3 — Pick design pillars (checkbox grid of 8-10 suggestions, pick 3-5)
4. Step 4 — Describe one key interaction (free text input)
5. Output — Skeleton spec preview showing L1-L3 structure with their inputs mapped in

**Feedback at each step:**
- "Next" button activates once minimum required input is provided
- Back navigation preserves prior inputs
- Output panel shows formatted skeleton, scrollable, copyable
**Failure modes:**
- Empty required fields: next button stays disabled, field border changes to error state
- Very long input: truncate display in output preview; preserve original in state
**State management:** Client-side only; no persistence across sessions; wizard resets on page reload

### 7. Before/After Comparison
**Trigger:** Continued scroll
**Interaction:** Toggle or slider between "Without spec" and "With RootSpec" views
**Feedback:** Smooth transition between two panels; both show real document content
**Keyboard:** Toggle via Space bar or Enter when focused
**Failure mode:** If toggle fails, both panels show stacked vertically

### 8. Open Source CTA
**Trigger:** Continued scroll
**Interaction:** Button/link to GitHub repo; secondary link to getting started
**Feedback:** External links open in new tab
**Goal:** Provide the next action for visitors who are ready to try the tool

### 9. Footer
**Trigger:** Scroll to bottom
**Interaction:** Static; contains attribution, build date, builder identity
**Content:** Builder name, build date, link to GitHub

## Global Interaction Patterns

### Theme Toggle
**Trigger:** User clicks toggle in header or hero
**Initial state:** Detected from OS preference via `prefers-color-scheme`
**Feedback:** Immediate visual switch between dark and light modes; preference stored in `localStorage`
**Keyboard:** Toggle via Enter/Space when focused

### Scroll Behavior
**Smooth scroll** on anchor links to sections
**Sticky header** with navigation anchors becomes visible after hero scrolls out of view
**Active section indicator** highlights current section in sticky nav

### Animation Philosophy
- All section-entry animations trigger once on first scroll into view (not on scroll-out)
- Interactive component animations respond immediately (no delay on user-initiated actions)
- Theme transitions apply to all color values simultaneously with a [short duration] fade
- No animations on critical text content — readability is never sacrificed for motion

## Failure Mode Catalog

| Scenario | Response |
|----------|----------|
| JavaScript disabled | Meta banner, hero, problem, how-it-works, before/after, and CTA sections fully readable; interactive sections show static fallback descriptions |
| Narrow viewport (mobile) | All layouts stack vertically; wizard uses full-width steps; hierarchy explorer stacks levels; before/after shows tabs instead of side-by-side |
| Very slow network | Static HTML and CSS loads first; JavaScript-dependent components show loading state |
| Version badge config missing | Badge shows "version unknown" rather than erroring; does not break layout |

## Verification Criteria for Implementers

- A visitor can read and understand the five levels without clicking anything
- The hierarchy explorer must convey reference directionality (arrows only go up)
- The wizard output must look like a real spec skeleton, not a toy
- The before/after must use real, distinct content in both panels — no lorem ipsum
- The meta banner must be visible without scrolling on viewport widths from [mobile minimum] to [desktop maximum]
- All interactive elements must be reachable and operable via keyboard alone
- Theme preference must persist across page refresh
