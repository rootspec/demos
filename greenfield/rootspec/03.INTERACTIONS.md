# Level 3: Interaction Architecture — HOW (Conceptual)

*References: 01.PHILOSOPHY.md, 02.TRUTHS.md*

## Interaction Loop Architecture

| Scale     | Scope                   | Duration       | Key Patterns                          |
|-----------|-------------------------|----------------|---------------------------------------|
| Immediate | Single interaction      | Seconds        | Scroll, click, hover, expand/collapse |
| Session   | Single visit            | Minutes        | Read → explore → try wizard → CTA     |
| Extended  | Return visits           | Days/Weeks     | Bookmark, share, reference             |
| Lifetime  | Adoption decision       | Ongoing        | Evaluate → pilot → embed in workflow  |

## Core User Journeys

### Journey 1: Discovery (Primary)

A developer lands on the site having heard about RootSpec or clicked a link. They're skeptical.

1. **Landing** — Hero and meta-banner immediately orient them: what this is, what it was built with
2. **Problem resonance** — The Problem section triggers recognition: "yes, this is exactly the pain I have"
3. **Mechanism understanding** — The How It Works section maps the four-skill workflow
4. **Conceptual exploration** — The Hierarchy Explorer makes the five-level model tangible
5. **Validation through play** — The Spec Wizard lets them apply the methodology to their own idea
6. **Decision point** — CTA to GitHub; they understand enough to evaluate it for their team

**Key trigger:** Recognition that their current spec problems are structural, not organizational
**Key feedback:** "This matches how I already think about products"
**Success signal:** Visitor opens the GitHub repo or bookmarks the site

### Journey 2: Validation (Secondary)

A developer has already heard of RootSpec and comes to validate their understanding or share with a colleague.

1. **Landing** — Version badge confirms they're on the current version
2. **Meta-banner** — They show a colleague: "see, this site was built from this pipeline"
3. **Specific section** — They navigate directly to the Hierarchy Explorer or Comparison
4. **Share** — Copy URL, share with team

**Key trigger:** Trust signal — the spec and source are publicly visible
**Key feedback:** Seeing the actual spec that generated this site

## Interaction Patterns

### Pattern 1: Progressive Disclosure

Sections are self-contained. Visitors don't need to scroll the full page to understand any single concept. Each section answers one question completely before introducing the next.

- Problem section: "What's broken with current approaches?"
- How It Works: "How does RootSpec fix it?"
- Hierarchy Explorer: "What exactly are the five levels?"
- Wizard: "How would I use this for my product?"

### Pattern 2: Demonstrate-Don't-Describe

Interactive elements exist to demonstrate methodology, not to describe it. The Hierarchy Explorer shows the reference hierarchy visually. The Wizard walks through spec creation. The Comparison shows before/after with real content.

Failure mode: Interactive features become visually impressive but methodologically shallow. Every interactive feature must answer: "what does this teach about RootSpec?"

### Pattern 3: Honest Signal

The meta-banner is always visible on load. It sets expectations before the visitor forms opinions. Rough edges become evidence of methodology power ("look what a sparse seed produces") rather than evidence of carelessness.

The banner must not minimize or apologize. It must contextualize.

### Pattern 4: Zero Friction CTA

The call to action is a GitHub link. No sign-up, no trial, no pricing. The methodology is open and the code is public. Friction between "I understand this" and "I'm trying this" must be minimal.

## Hierarchy Explorer Interaction

**Entry:** Visitor sees five labeled levels displayed vertically or in a visual hierarchy
**Expand:** Clicking a level reveals example content for that level
**Reference visualization:** Arrows or highlights show which other levels each can reference (upward only)
**Hover:** Hovering a level highlights its allowed reference directions
**Keyboard:** Tab through levels; Enter/Space to expand/collapse; arrow keys to navigate

**Failure modes:**
- Visitor doesn't understand why arrows only go upward → tooltip or inline explanation required
- Too much content in each expanded level overwhelms → keep examples concise
- Mobile: tap to expand, no hover states required

## Spec Wizard Interaction

**Entry:** Visitor sees a single input: "Describe your product idea in one line"
**Step 1 — Mission:** Select from templates or write custom mission statement. Template options should be diverse (productivity tool, developer tool, consumer app, B2B SaaS).
**Step 2 — Design Pillars:** Grid of [N] suggested pillar options. Visitor selects 3-5. Custom entry available.
**Step 3 — Interaction:** Describe one key user interaction in plain text.
**Output:** Skeleton spec showing L1-L3 structure with their inputs mapped to the correct levels.

**Feedback during wizard:**
- Progress indicator shows current step number and total step count
- Previous step always accessible (no lock-in)
- Output appears immediately after Step 3; no loading state needed (client-side only)

**Failure modes:**
- Visitor submits empty/trivial input → show placeholder output with generic example, don't error
- Visitor skips a step → allow skipping with placeholder values in output
- Output looks too impressive → must be clearly labeled as a skeleton, not a complete spec

## Before/After Comparison Interaction

**Entry:** Two panels side by side (desktop) or toggle (mobile)
**Without-spec panel:** Shows vague requirements doc — prose, ambiguous stories, no traceability
**With-spec panel:** Shows structured L1-L5 hierarchy — testable stories, pillar references, system boundaries
**Toggle/slider:** Smooth transition between states
**Content:** Real representative content, not lorem ipsum. Based on a fictional "TaskManager" product for concrete familiarity.

## Theme Toggle

**System detection:** On first load, use system preference (`prefers-color-scheme`)
**Manual override:** Persistent toggle in header; persists across sessions via `localStorage`
**Transition:** Smooth CSS transition on theme change (no flash)
**No flash on load:** Apply theme class before first paint using inline script in `<head>`

## Responsive Behavior

- **Mobile first:** All interactive features must work via touch
- **Hierarchy Explorer:** Stacked vertically on mobile; horizontal layout optional on desktop
- **Spec Wizard:** Single-column flow on all screen sizes; no horizontal scrolling
- **Comparison:** Toggle mode on mobile (not slider); panels stack or swap on button press
- **Navigation:** Minimal header, no hamburger menu required if sections are simple anchor links

## Edge Cases and Failure Modes

| Scenario | Response |
|----------|----------|
| JavaScript disabled | All static content visible; interactive features show graceful fallback message |
| Very slow connection | Critical content (hero, problem, how-it-works) loads first; interactive sections lazy-load |
| Screen reader | All interactive elements have ARIA labels; hierarchy explorer navigable via keyboard |
| User skips wizard steps | Skeleton output generates with placeholder values for skipped steps |
| No system theme preference | Default to light theme |

## Verification Criteria

An implementation correctly captures this interaction architecture if:

1. A developer can understand RootSpec's core value proposition without scrolling past the hero + problem sections
2. The Hierarchy Explorer communicates the upward-only reference rule without requiring additional explanation
3. The Spec Wizard produces output that a visitor can read and recognize as "this maps to something I've thought about before"
4. The meta-banner sets expectations before the visitor has time to form negative opinions about rough edges
5. All interactive features work via keyboard navigation
6. Theme preference persists across page reloads
