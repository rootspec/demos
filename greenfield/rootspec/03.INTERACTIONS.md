# Level 3: Interaction Architecture

**Product:** RootSpec Marketing Site
**Version:** 1.0.0
**Status:** Active

---

## Core Interaction Model

This is primarily a reading site. Most visitors arrive, read, and leave — often to the GitHub repo. The interaction architecture must support progressive engagement: visitors can skim, read selectively, or dive deep depending on motivation. Interactive elements interrupt the reading flow only when they add genuine value.

---

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Patterns |
|-------|-------|----------|--------------|
| Immediate | Single element interaction (toggle, expand, click) | Seconds | Theme toggle, hierarchy level expansion, wizard step navigation |
| Session | Complete page visit — reading, exploring, deciding | Minutes | Banner → Hero → Problem → How It Works → Explore → Wizard → CTA |
| Extended | Return visit or referral from GitHub | Minutes to hours | Revisiting interactive elements with new context after reading docs |

---

## Primary Interaction Flows

### Flow 1: First Arrival (Standard Reading Path)

**Trigger:** Visitor arrives at the URL, likely via link from GitHub or social share.

**Flow:**
1. Meta banner is the first visible element — visitor reads what the site is and why it looks the way it does
2. Hero section establishes the product name, tagline, and version badge
3. Problem section resonates (or doesn't) with developer experience
4. "How It Works" section provides a walkthrough of the four skills
5. Hierarchy explorer invites active engagement with the core concept
6. Spec Wizard invites the visitor to try the methodology with their own idea
7. Before/After comparison provides a concrete illustration of the benefit
8. Author's Notes section provides the intellectual argument at depth
9. Open Source CTA provides the path to action (GitHub)

**Outcome:** Visitor understands what RootSpec is, has experienced the hierarchy interactively, and has a path to the GitHub repo.

**Failure modes:**
- Visitor skips the meta banner and forms incorrect expectations → accept; cannot force reading order
- Visitor loses interest before the interactive elements → the problem/how-it-works sections must earn attention before the interactive demos appear
- Visitor misunderstands the wizard output as AI-generated → the section framing must be explicit that it is template-based

---

### Flow 2: Hierarchy Explorer

**Trigger:** Visitor scrolls to or navigates to the hierarchy explorer section and sees the five-level visualization.

**Flow:**
1. Default state shows all five levels labeled, with brief descriptions visible
2. Visitor clicks or taps a level — it expands to show example content and allowed references
3. Hovering a level highlights the levels it can reference (arrows flow upward only)
4. Keyboard navigation (Tab to focus, Enter/Space to expand) is available throughout
5. Visitor can expand multiple levels or collapse them

**Feedback:** Selected level visually distinguishes itself; reference relationships shown with directional indicators. Transitions are quick ([brief transition duration]) and functional — not decorative.

**Failure modes:**
- Touch targets too small on mobile → each level must have sufficient tap area
- Reference arrows confusing on small screens → fallback to text-only description of reference rules on narrow viewports

---

### Flow 3: Spec Wizard

**Trigger:** Visitor reaches the wizard section, which invites them to enter a one-line product idea.

**Flow:**
1. **Step 1 (Mission):** Visitor enters a product idea. Prompted to select or write a mission statement from template options. Options are short, varied, and include a free-text fallback.
2. **Step 2 (Pillars):** Visitor selects [min pillars count] to [max pillars count] design pillars from a curated set of suggestions. Custom pillar input is available.
3. **Step 3 (Interaction):** Visitor describes one key interaction — what triggers it, what happens, what the user feels.
4. **Output:** A rendered skeleton spec showing how the visitor's input maps to L1 (Mission + Pillars), L2 (one derived truth), and L3 (one interaction pattern). The output uses the actual RootSpec format with level labels.

**Feedback:**
- Active step is visually distinct
- Previous steps are visible but non-editable once advanced
- Output appears after Step 3 is completed; visitor can navigate back to revise
- No submission — all state is in-memory

**Failure modes:**
- Visitor enters empty fields → advance button disabled until required fields are filled; clear affordance that input is needed
- Visitor advances to output quickly without engaging with steps → accept; the output still has value
- Visitor expects AI-generated content → section framing must clearly say "template-based"

---

### Flow 4: Before/After Comparison

**Trigger:** Visitor reaches the comparison section.

**Flow:**
1. Default view shows either side-by-side panels (wide screens) or a toggle (narrow screens)
2. "Without spec" panel shows: vague requirements document, ambiguous story, untraceable decision
3. "With RootSpec" panel shows: structured hierarchy entry, testable story, feature tracing to a design pillar
4. Content in both panels is real (not lorem ipsum) and corresponds to the same fictional feature
5. On narrow screens, a toggle switches between panels; toggle state persists during the session

**Feedback:** Panels are visually distinct — "without" panel styled to suggest document entropy, "with" panel styled to suggest structure and order. No animation required; the content is the point.

**Failure modes:**
- Side-by-side too dense on narrow screens → toggle layout is the fallback
- Content in panels seems fabricated → use plausible developer domain content (e.g., "user notifications feature")

---

### Flow 5: Theme Toggle

**Trigger:** Visitor clicks the theme toggle in the header.

**Flow:**
1. System preference is detected on initial load; light mode is the default if no preference is set
2. Toggle switches between light and dark themes
3. Selected theme is stored in `localStorage` and respected on return visits

**Feedback:** Transition between themes is smooth ([brief transition duration]). Toggle icon reflects current state (sun/moon or equivalent).

**Failure modes:**
- Flash of wrong theme on load → theme is applied as early as possible, before first paint
- Toggle not discoverable → placed in a consistent, expected location (header right)

---

## Multi-Scale Coordination

The page is a single scrollable document. No internal routing between pages. All sections are present in the DOM on load; no lazy-loaded sections (interactive islands may lazy-load their JS bundles).

Navigation anchors in the header allow direct linking to sections. These must be stable — the URL hash should reflect the current section as the user scrolls, where feasible.

---

## Verification Criteria

An implementation satisfies this interaction architecture when:

1. The meta banner is visible above the fold on all standard viewport sizes
2. The hierarchy explorer correctly shows reference relationships (L1 has no allowed references; L5 can reference all levels) via interactive state
3. The spec wizard produces a rendered skeleton spec in the RootSpec format with the visitor's actual input reflected
4. The before/after comparison presents real, non-placeholder content in both panels
5. The theme toggle works and persists across page reloads
6. All interactive elements are operable by keyboard alone (no mouse-only interactions)
7. The GitHub links in the meta banner and CTA section resolve to the correct repository URLs
8. The page does not make any external API calls (verified by network inspection)
