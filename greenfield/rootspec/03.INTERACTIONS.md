# Level 3: Interaction Architecture

References: [01.PHILOSOPHY.md, 02.TRUTHS.md]

## Core Interaction Model

The site has two distinct visitor modes:

1. **Reader** — Arrives skeptical, reads linearly from top to bottom, seeks to understand the methodology through prose and visual explanation
2. **Explorer** — Arrives curious, skips to interactive sections, wants to try the methodology firsthand before reading about it

Both modes must be supported. The page layout and flow accommodate linear reading while making interactive sections discoverable and inviting to explorers.

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Patterns |
|-------|-------|----------|-------------|
| Immediate | Single interaction | Milliseconds | Click response, toggle feedback, step completion |
| Session | Page visit | Minutes | Reading comprehension, wizard completion, explorer navigation |
| Extended | Return visit | Days | n/a — single-page site, no persistent state |
| Lifetime | Methodology adoption | Ongoing | n/a — conversion to the framework happens off-site |

## Primary Interaction Flows

### Flow 1: Linear Reader

**Trigger:** Visitor arrives at the page (any entry point)

1. Meta banner appears immediately — visitor understands this is a demo
2. Hero section communicates what RootSpec is and why it exists
3. Problem section validates the visitor's pain (recognition moment)
4. How It Works section explains the four-command workflow
5. Interactive sections provide depth for those who want it
6. Author's Notes provide philosophical grounding for those who read
7. CTA section provides clear path to the GitHub repo

**Feedback:** Smooth scrolling, section transitions at [brief duration]; no jarring jumps or disorienting effects

**Exit point:** CTA section links to GitHub repo and documentation

### Flow 2: Hierarchy Explorer

**Trigger:** Visitor clicks on or tabs to the Hierarchy Explorer component

1. Five levels are displayed as interactive elements
2. Visitor clicks a level (or presses Enter/Space) to expand it
3. Expanded level shows example content for that level
4. Visual connections indicate which levels each level can reference
5. Hovering a level highlights its allowed upstream references
6. Visitor can collapse a level by clicking it again or pressing Escape

**Feedback:** Level expands/collapses with [quick transition]; reference arrows highlight in [accent color]; keyboard focus is visible and logical

**Error state:** If no level is selected, show an invitation to "Click a level to explore"

**Mobile adaptation:** Touch-friendly tap targets; reference arrows simplified or replaced with text indicators on small screens

### Flow 3: Spec Your Idea Wizard

**Trigger:** Visitor clicks "Start" or "Try it" in the wizard section

**Step 1 — Mission:**
- Visitor enters a one-line product idea (free text input)
- Optionally selects from mission templates (pre-written examples)
- "Next" advances to Step 2

**Step 2 — Design Pillars:**
- Visitor selects 3-5 design pillars from a set of suggestions (emotional phrases)
- Or writes their own pillar (custom input)
- "Next" advances to Step 3 (enabled only when 3+ pillars selected)

**Step 3 — Key Interaction:**
- Visitor describes one core user interaction in plain text
- "Generate Spec" triggers output display

**Output display:**
- Skeleton spec appears showing how visitor's inputs map to L1-L3
- Each spec section is labeled with its level (L1: Philosophy, L2: Truths, L3: Interactions)
- Visitor sees their own words reflected in structured spec format

**Back navigation:** Visitor can return to any previous step

**Reset:** Visitor can start over from Step 1

**Feedback:** Step indicator shows current position; "Next" button is disabled until current step's minimum input is satisfied; output appears with [quick transition]

**Error state:** Empty text fields show inline validation message before advancing; no silent failures

**Mobile adaptation:** Full-screen step view on small screens; large touch targets for pillar selection

### Flow 4: Before/After Comparison

**Trigger:** Visitor arrives at the comparison section

**Toggle variant:** Single toggle switches between "Without RootSpec" and "With RootSpec" views
**Slider variant (optional enhancement):** Drag slider reveals the "With RootSpec" panel progressively

**Without RootSpec panel:** Shows vague requirements doc, ambiguous user stories, untraceable decisions — real content, not lorem ipsum

**With RootSpec panel:** Shows structured hierarchy, testable stories, features that trace to design pillars — real content demonstrating actual spec output

**Feedback:** Panel switch has [brief transition]; current mode is visually distinct; toggle state is clear

**Mobile adaptation:** Stacked vertically on small screens; toggle at top for easy access

### Flow 5: Theme Toggle

**Trigger:** Visitor clicks the theme toggle control in the header

1. System preference is detected on load and applied as initial theme
2. Visitor can override by clicking the toggle
3. Theme preference persists for the session (does not require storage beyond session)

**Feedback:** Theme transition applies to all elements at once in [brief duration]; no flash of unstyled content

### Flow 6: Meta Banner

**Trigger:** Page load — banner is always visible, non-dismissable

- Explains this site is a RootSpec demo built from a sparse seed
- Links to SEED.md and spec files in the GitHub repo (absolute URLs, not relative paths)
- Frames that rough edges are the result of minimal human guidance
- Banner remains visible as visitor scrolls (sticky or above-fold placement)

**Feedback:** Banner does not distract from reading; it informs without competing

## Failure Modes

- **Wizard incomplete data:** If visitor attempts to advance with insufficient input, inline validation message appears; "Next" button remains disabled; no data is lost
- **Interactive element not loading (JS disabled):** Page content is readable without JS; interactive sections degrade gracefully to static explanations
- **Small screen overlap:** Interactive elements that cannot fit on small screens collapse to simplified text-based alternatives
- **Invalid URL in meta banner:** Since links point to absolute GitHub URLs, broken links would indicate a repository structure change; no runtime fallback possible, but links should be tested in CI

## Keyboard Navigation

- **Tab order** follows reading order top to bottom
- **Hierarchy Explorer:** Tab to reach the component, Tab through levels, Enter/Space to expand/collapse, Escape to collapse current
- **Wizard:** Tab through form fields and buttons, Enter to submit each step, Escape to return to previous step
- **Before/After toggle:** Tab to reach toggle, Enter/Space to switch panels
- **Theme toggle:** Tab to reach, Enter/Space to toggle

## Verification Criteria for Implementations

- Meta banner must appear above the fold on all common viewport sizes without scrolling
- Hierarchy Explorer must be navigable without a mouse; all states reachable via keyboard
- Wizard must produce visible spec output that uses the visitor's own words
- Before/After comparison must contain real, substantive content in both panels
- Theme toggle must not cause a flash of unstyled content on load
- All GitHub links in the meta banner must use absolute URLs (not relative paths)
- Interactive elements must be touch-friendly on mobile (minimum [comfortable touch target size])
