# Level 3: Interaction Architecture

## Interaction Loop Architecture

| Scale     | Scope                        | Duration   | Key Interaction                                      |
|-----------|------------------------------|------------|------------------------------------------------------|
| Immediate | Single section / component   | Seconds    | Read prose, expand hierarchy level, take wizard step |
| Session   | Page scroll, explore features| Minutes    | Understand methodology, try interactive features     |
| Extended  | Return visit, share with team| Days/Weeks | Reference spec, share wizard output, cite Author's Notes |
| Lifetime  | Adopt methodology            | Ongoing    | Use RootSpec on real projects                        |

## Core Interaction Patterns

### Pattern 1: Linear Reading with Anchored Navigation

**Trigger:** Visitor lands on the page.

**Flow:**
1. Meta-banner is the first thing visible — establishes the demo context before any content
2. Visitor reads hero: tagline, one-sentence explanation, version badge
3. Visitor scrolls through sections in order: Problem → How It Works → Hierarchy Explorer → Spec Wizard → Comparison → Author's Notes → CTA
4. Navigation anchors allow jumping to any section

**Feedback:** Section transitions are quick (functional, not decorative). Active section highlighted in navigation.

**Failure mode:** If visitor skips the meta-banner, they may evaluate the site as a conventional marketing page. The banner must appear above the fold and before any other content.

### Pattern 2: Hierarchy Explorer — Expand and Connect

**Trigger:** Visitor reaches the Hierarchy Explorer section.

**Flow:**
1. Five hierarchy levels are shown in a visual layout (L1 at top, L5 at bottom)
2. Visitor clicks a level to expand it — sees the level's name, purpose, and example content
3. Hovering a level highlights the levels it can reference (arrows flow upward only)
4. Visitor navigates between levels via keyboard (Tab to focus, Enter/Space to expand)
5. Mobile: touch targets are large enough for thumb interaction; expand/collapse on tap

**Feedback:** Expanded level is visually distinguished. Reference arrows animate on hover/focus. Collapse happens when another level is expanded (one at a time) or on click of the same level.

**Failure mode:** If arrows are confusing, fallback is text description of reference rules below the diagram. If JavaScript fails, the component renders in a collapsed-but-readable default state.

### Pattern 3: Spec Wizard — Structured Input to Skeleton Output

**Trigger:** Visitor reaches the "Spec Your Idea" section.

**Flow:**
1. Step 1 — Visitor enters a one-line product idea in a text field
2. Step 2 — Visitor selects or writes their product's mission (template options or free text)
3. Step 3 — Visitor picks 3-5 design pillars from a list of suggestions
4. Step 4 — Visitor describes one key interaction
5. Output — A skeleton spec (L1-L3 preview) is shown, mapping visitor input to the hierarchy levels

**Feedback:** Progress indicator shows current step (Step N of N). Each completed step is summarized above the current step. Output appears immediately on completion with no loading state (client-side only).

**Failure mode:** If visitor skips a required field, inline validation prompts completion before advancing. Wizard state is not persisted; refreshing resets to Step 1. No external calls are made.

### Pattern 4: Before/After Comparison

**Trigger:** Visitor reaches the Comparison section.

**Flow:**
1. Two panels are shown side by side (desktop) or stacked (mobile)
2. Toggle or slider allows switching between "Without RootSpec" and "With RootSpec" views
3. Each panel contains real content — not lorem ipsum — showing a vague requirement doc vs. a structured hierarchy
4. Visitor toggles between states to see the contrast

**Feedback:** Toggle state is visually clear. Panel content changes without page reload.

**Failure mode:** On narrow screens, panels stack vertically and the toggle becomes a tab control.

### Pattern 5: Theme Toggle

**Trigger:** Visitor clicks the theme toggle in the header.

**Flow:**
1. System preference is detected on load and applied immediately (no flash of wrong theme)
2. Visitor can toggle between light and dark modes manually
3. Preference is persisted in localStorage and respected on return visits

**Feedback:** Toggle state reflects current mode. Transition between themes is instant (no animation — avoid jarring color shifts).

**Failure mode:** If localStorage is unavailable, system preference is used on every load. If system preference is unavailable, light mode is the default.

## Verification Criteria for Implementers

- Meta-banner is visible on initial load without scrolling on any viewport width
- Theme preference is applied before first paint (no flash of wrong theme)
- Hierarchy Explorer is keyboard-navigable without a mouse
- Wizard output is shown without any network request
- Before/After comparison works on mobile with touch input
- All internal links resolve correctly under the `/demos/greenfield/` base path
- Author's Notes section reproduces the author's text verbatim — no truncation, no paraphrasing
- Version badge reads from `.rootspec.json`, not a hardcoded string

## Edge Cases and Failure Modes

- **JavaScript disabled:** Page must still be readable. Interactive components (Explorer, Wizard) display a static fallback or a graceful "requires JavaScript" message. Core content (Problem, How It Works, Author's Notes, CTA) is accessible without JS.
- **Slow connection:** No images to load except SVG (inline or linked). No web fonts that block rendering beyond a reasonable fallback. Page is usable before fonts finish loading.
- **Mobile/touch:** All interactive features use touch-friendly tap targets. No hover-only interactions; hover states are enhancements, not requirements.
- **Print:** Page is printable. Key sections (Author's Notes, How It Works) render cleanly without decorative elements.
