# Level 3: Interaction Architecture

**References:** `01.PHILOSOPHY.md`, `02.TRUTHS.md`

---

## Core Interaction Model

This is a marketing and demonstration site. Visitors arrive skeptical or curious and leave either converted or unconvinced. The interaction model prioritizes progressive disclosure: immediate hook, conceptual clarity, tactile proof, clear path to action.

Visitors do not "use" this site — they experience it. Every interactive element must reward engagement rather than demand it.

---

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Pattern |
|-------|-------|----------|-------------|
| Immediate | Section scroll, element hover | Seconds | Visual feedback, smooth transitions |
| Session | Exploring interactive features | Minutes | Progressive disclosure, no dead ends |
| Extended | Returning after trying RootSpec | Days | Version badge shows evolution |
| Lifetime | Referencing the site as evidence | Ongoing | Links to spec and seed remain stable |

## Primary User Journey

### Entry → Hook

1. Visitor arrives at the page
2. Meta banner is immediately visible — establishes the site's nature honestly
3. Hero section delivers tagline and one-sentence explanation
4. Version badge confirms this is a live, versioned artifact

### Hook → Understanding

5. Problem section validates the visitor's pain (spec drift, AI without wisdom, Google Doc specs)
6. "How It Works" section walks through four skills: init → spec → impl → validate
7. Diagram shows the methodology as a filter: spec surrounds the development cycle, only valid solutions pass

### Understanding → Proof

8. Hierarchy Explorer invites click/tap — visitor expands levels and sees example content
9. Visual connections show which levels reference which (arrows flow upward only)
10. Hovering/focusing a level highlights its allowed references
11. Before/After Comparison shows real content (not lorem ipsum) with slider or toggle

### Proof → Participation

12. Spec Wizard invites visitor to try the methodology on their own idea
13. Three-step flow: mission → design pillars → key interaction
14. Output is a skeleton spec showing how input maps to L1-L3
15. No sign-up, no AI — pure structured templates

### Participation → Action

16. Open Source CTA links to GitHub repo with getting-started instructions
17. Meta banner links to seed and spec files for full pipeline audit

## Interaction Patterns

### Meta Banner

- Persistent — visible on page load without scrolling
- Contains two links: [View the spec →] and [View the seed →]
- Links are absolute GitHub URLs (not relative — would break static build prerenderer)
- Tone is honest: "this is what you get from a sparse seed with no iteration"

### Hero Section

- Tagline: immediate visual clarity about what RootSpec is
- Version badge: read from `.rootspec.json` at build/render time, displayed prominently
- No interaction required — delivers value on sight

### Hierarchy Explorer

- Five levels displayed as expandable/collapsible cards or accordion panels
- Click or keyboard activation expands a level to reveal example content
- Visual lines or arrows show allowed reference directions (upward only)
- Hovering or focusing a level highlights which levels it can reference
- Touch-friendly: tap targets sized for mobile interaction
- Keyboard accessible: focus management, Enter/Space to activate, arrow keys to navigate

### Spec Wizard

- Three sequential steps — cannot skip or reorder
- Step 1: Enter one-line product idea (text input)
- Step 2: Select or write mission statement (template options available)
- Step 3: Pick 3-5 design pillars from suggestions (or write custom)
- Output panel: skeleton spec showing L1-L3 structure with visitor's input
- Back navigation available between steps
- No submission — output is displayed client-side only
- Touch-friendly throughout

### Before/After Comparison

- Two panels: "Without RootSpec" (vague doc) and "With RootSpec" (structured hierarchy)
- Toggle or slider mechanism to switch between views
- Real content in both panels — visitors can read and compare
- Accessible: can be operated via keyboard, not only mouse drag

### Theme Toggle

- System preference detected on load
- Manual toggle available and visible
- Preference persists across sessions (localStorage or similar client-side mechanism)
- Smooth transition between modes — not a hard flash

### Smooth Animations

- Section transitions on scroll: fade-in or slide-up as sections enter viewport
- Interactive elements respond to hover/focus with visual feedback
- Transitions are meaningful — they signal state change, not decoration
- All animations respect `prefers-reduced-motion`

## Failure Modes and Edge Cases

| Scenario | Behavior |
|----------|----------|
| JavaScript disabled | Static content remains accessible; interactive features degrade gracefully |
| `.rootspec.json` unavailable at build time | Version badge falls back to "unknown" rather than breaking build |
| User skips Spec Wizard steps | Step navigation prevents forward progression without completing current step |
| Visitor on slow connection | No external API calls means no loading failures; all assets from same origin |
| Touch device on Hierarchy Explorer | Tap replaces hover; focus states replace hover highlights |
| System preference for reduced motion | Animations disabled; transitions use opacity only if needed |

---

## Verification Criteria

For implementers:

- Meta banner is visible without scrolling on all viewport sizes
- Hierarchy Explorer can be fully operated via keyboard (no mouse required)
- Spec Wizard produces skeleton spec output for any valid input combination
- Before/After Comparison is accessible via keyboard (not only mouse drag/hover)
- Theme toggle persists across page reload
- Version displayed matches `version` field in `.rootspec.json`
- All interactive features work on mobile (minimum supported viewport width)
- No network requests are made after initial page load
