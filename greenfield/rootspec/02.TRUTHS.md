# Level 2: Stable Truths

**Product:** RootSpec Marketing Site
**Version:** 1.0.0
**Status:** Active

---

## Design Philosophy

### Truth: The site is the argument, not a container for it

The aesthetic and editorial quality of this site IS part of the case for RootSpec. A sloppy or generic site proves that spec-driven AI development doesn't work. A considered, intentional site proves it does. Every design decision — typography, whitespace, interaction behavior — must reinforce the thesis, not simply avoid undermining it.

### Truth: Show rather than explain, wherever possible

Visitors should be able to experience the RootSpec hierarchy before they fully understand it. Interactive elements provide conceptual handholds that prose cannot. However, the interactions must be genuine demonstrations of methodology, not decorative widgets — they must produce real structural output that reflects the actual spec format.

### Truth: Editorial credibility comes from restraint

The most damaging thing this site can do is look like a generic AI-generated SaaS page. The counter-strategy is not to add more decoration — it's to remove defaults. Fewer colors, more whitespace, typography-first hierarchy, no gradient blobs, no floating emoji. The site reads as editorial (long-form, considered) not commercial (conversion-optimized, animated).

### Truth: Light mode is the default; the reader chooses dark

Editorial sites read better in light mode. Defaulting to dark signals AI-generated output, not authorial intention. The dark mode toggle exists for user preference, not as a design statement.

### Truth: Complexity lives in content, not chrome

Interactive elements and the SVG diagram provide the visual interest. The page frame — navigation, typography, spacing — should not compete. The hierarchy explorer and wizard are the centrepieces; the surrounding chrome should recede.

## Architectural Commitments

### Truth: Static site, no server

This is a static marketing site. No server-side rendering beyond build-time HTML generation. No backend API. No database. All interactive state is ephemeral, client-side only. This is a commitment for simplicity, reliability, and cost.

### Truth: Client-side interactivity for the three demos

The hierarchy explorer, spec wizard, and before/after comparison are client-side components. No external API calls. No user accounts. No data persistence after the session ends. This enforces the "no external API calls" principle from L1.

### Truth: Base path stability across environments

The site deploys to `/demos/greenfield/`. This base path is the same in development, preview, and production builds. It is never derived from an environment variable that differs between environments. All asset paths and internal links resolve under this subpath.

### Truth: Version is read from the spec, not hardcoded

The RootSpec version number visible on the site is read at build time from `.rootspec.json`. It is never a hardcoded string in the template. This demonstrates the methodology's principle that authoritative values live in one place.

## Trade-Offs

- **We choose editorial depth over conversion optimization.** Visitors who want to understand the methodology will stay; visitors looking for a quick signup flow will not find one. That is correct behavior.
- **We choose honest meta-commentary over a seamless "product site" illusion.** The meta banner breaks the fourth wall deliberately. A visitor who misses the banner and evaluates this as a conventional marketing site will misunderstand the whole point.
- **We choose minimalism over visual richness.** The diagram and interactive components are the designed artifacts; everything else is recessive. This risks looking sparse to some visitors. That is acceptable.
- **We choose light mode default over dark.** Some developers prefer dark. The toggle exists. But the default experience must be editorial, not cinematic.
- **We choose client-side demos over AI-powered features.** A live AI spec wizard would be more impressive but would require API keys, rate limits, and cost. Template-based client-side output demonstrates the methodology approachably without adding infrastructure.
- **We choose verbatim over edited author voice.** The Author's Notes section reproduces the full, unedited text. Some prose may feel unpolished or longer than a typical marketing site would allow. That is the point — this is the author's thinking, not a copywriter's sanitized version.

## Success Criteria

The site succeeds when:

1. A developer who lands on the page understands the core RootSpec value proposition without reading the entire page
2. The meta banner is seen and understood before any marketing content — visitors know they are looking at a demo, not a polished product site
3. At least one interactive element (hierarchy explorer or wizard) is engaged with before the visitor exits
4. The Author's Notes section is present, complete, and verbatim
5. The GitHub links (framework repo and seed/spec files) are functional and resolve to correct absolute URLs
6. The page is functional across common viewport sizes and works on touch devices
7. The RootSpec version number is visible and matches the `version` field in `.rootspec.json`
8. The page meets basic accessibility standards — keyboard navigation works, color contrast is sufficient, interactive elements have appropriate ARIA labels

---

## Constraints

- **Framework:** Astro with TypeScript; Tailwind CSS for utility styles; React for interactive island components
- **Deployment target:** GitHub Pages at `/demos/greenfield/`
- **No external fonts** — Fonts loaded from CDN introduce dependency risk and privacy surface; use system-safe web fonts or self-host via npm packages
- **No tracking or analytics** — No third-party scripts. This is a principle, not a limitation.
- **Accessibility:** Keyboard navigation required for hierarchy explorer and wizard; WCAG AA color contrast minimum
- **Performance:** Static HTML delivered fast; interactive islands should not block page load
