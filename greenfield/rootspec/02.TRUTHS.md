# L2: Truths

## Design Philosophy

Specification is fundamentally about validation, not generation. The goal is not to automate human decision-making but to preserve and validate human intent as it flows through the development process.

## Trade-offs

### We choose validation over automation
The spec exists to prove what was built matches what was intended, not to replace human judgment with AI. Tools should amplify human wisdom, not substitute for it.

### We choose methodology over tooling
RootSpec works with any framework, any AI, any team size. The five-level hierarchy is framework-agnostic because good principles transcend implementation details.

### We choose structure over flexibility  
Strict hierarchy prevents scope drift and circular dependencies. Users can't reference lower levels from higher ones, which feels restrictive but creates clarity.

### We choose transparency over polish
This demo site acknowledges its rough edges upfront. Users see exactly what minimal human guidance produces, not a carefully crafted marketing experience.

### We choose demonstration over explanation
Interactive elements and real examples prove the methodology works. Users experience the hierarchy rather than reading about it.

## Success Criteria

### Primary Success: Specification Adoption
- Developers use RootSpec for new complex projects
- Teams reference the spec during implementation discussions  
- AI-assisted development produces more reliable, intentional output

### Secondary Success: Methodology Understanding
- Visitors understand the five-level hierarchy without reading documentation
- Interactive elements demonstrate core concepts effectively
- Users can articulate why philosophy should drive implementation

### Validation Metrics
- Time from landing page to hierarchy comprehension
- Completion rate of interactive "Spec Your Idea" wizard
- GitHub stars and community adoption of RootSpec framework

## Constraints

### Technical Constraints
- Client-side only - no external API calls or server dependencies
- Mobile-responsive with touch-friendly interactions
- Keyboard accessible for all interactive elements
- Dark/light theme support with system preference detection

### Content Constraints  
- All examples must be real, not lorem ipsum
- Interactive elements must work without AI or external services
- Site must load and function completely offline after first visit
- Version number displayed prominently and easy to update

### Scope Constraints
- Focus on methodology explanation, not implementation tutorials
- Target technical leads and developers, not general audiences
- Demonstrate RootSpec principles rather than teaching specification theory
- Show working examples rather than abstract concepts
