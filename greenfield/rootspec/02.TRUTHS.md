# Level 2: Stable Truths
# RootSpec Marketing Site

---

## Truth: The site is the proof

The most compelling argument for RootSpec is that this site was built with it. Every design decision must reinforce this meta-narrative. We choose **demonstration over description** — if we can show the methodology working, we don't need to explain it as theory.

**Trade-off:** We choose transparency over polish. Rough edges are documented and explained, not hidden. A perfect-looking site built without a visible process would undermine the message.

---

## Truth: Philosophy precedes all implementation choices

Every visual decision, interactive element, and content choice must trace to a Design Pillar from Level 1. Features that cannot be justified by a Pillar are rejected.

**Trade-off:** We choose methodical scope over feature richness. A smaller, coherent feature set that fully supports the Pillars is better than a larger set that dilutes the message.

---

## Truth: Interactivity is explanation, not decoration

Interactive elements exist to teach, not to impress. Each interactive section must leave the user with a clearer understanding of RootSpec than if they had read equivalent text.

**Trade-off:** We choose educational depth over visual spectacle. Animations and transitions are purposeful; they guide attention, not distract it.

---

## Truth: Content is primary; design is substrate

Typography, color, and layout serve content legibility. No design choice should require the user to struggle to read, navigate, or understand.

**Trade-off:** We choose content clarity over visual novelty. A subtle, restrained design that foregrounds the message is the right design.

---

## Truth: The site runs entirely client-side

No runtime API calls, no server-rendered personalization, no external data dependencies. All interactivity (wizard, hierarchy explorer, comparison) is implemented in the browser with static data.

**Trade-off:** We choose reliability over dynamism. The site works offline, loads fast, and cannot fail due to a third-party service outage. AI-assisted features are out of scope for this demo.

---

## Truth: Dark/light theming follows user intent

Theme preference is read from the system and can be overridden manually. The choice persists across sessions. Both themes must meet accessibility contrast standards.

**Trade-off:** We choose user autonomy over design uniformity. We maintain two fully specified theme variants rather than forcing a single palette.

---

## Truth: Version is a first-class signal

The RootSpec version used to build this site is displayed prominently. This is a demo of a framework; visitors should know exactly which version they're seeing demonstrated.

**Trade-off:** We choose specificity over timelessness. The version badge may date the demo, but that specificity is more honest than presenting a vague "latest."

---

## Truth: Mobile-first responsive layout

All layouts and interactive features are designed for the smallest screen first, then enhanced for larger viewports. Touch-friendly targets, readable type at base sizes, and functional interactions on phones are non-negotiable.

**Trade-off:** We choose universal access over desktop optimization. A developer reading this on a phone during a commute must have the same quality experience as one at a workstation.

---

## Success Criteria

The site succeeds when:

- A developer can explain what RootSpec is after [a short visit]
- A developer can find the spec, seed, and GitHub repo from the landing page
- The meta banner immediately conveys that this site is a live demo of the methodology
- Interactive elements work without error on desktop and mobile
- The site loads and is usable in [brief duration] on a typical connection
- Theme toggle, keyboard navigation, and screen reader compatibility meet WCAG AA standards
