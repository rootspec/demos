# Level 2: Stable Truths

**Product:** RootSpec Marketing Site
**Level:** L2 — WHAT Strategy
**References:** L1 + External

---

## Design Philosophy

### Truth: The demo is the argument

The most persuasive explanation of RootSpec is a site built with RootSpec. Every section exists to show the methodology working, not to describe it abstractly. Visitors who understand the site understand the product.

This supports the **Earned Trust** pillar (L1): claims are verified by the artifact itself.

### Truth: Honesty outperforms polish

The meta banner and its candid framing — "this is what you get from a sparse seed with no iteration" — are more credible than a professionally designed marketing site. The rough edges, properly contextualized, become proof of the methodology's capability floor.

This supports the **Intellectual Honesty** pillar (L1).

### Truth: Interaction teaches better than description

The hierarchy explorer and spec wizard are not nice-to-have enhancements — they are the primary explanatory mechanism. A developer who spends three minutes with the hierarchy explorer learns more than one who reads the entire page.

This supports the **Methodological Clarity** pillar (L1).

### Truth: Simplicity over completeness

We choose a focused, comprehensible site over a comprehensive feature set. Every section must justify its presence by teaching something visitors cannot learn from the other sections.

This supports the **Confident Accessibility** pillar (L1).

## Trade-offs

- **We choose client-side interactivity over server-side personalization** — no backend, no API, no state that can fail in production
- **We choose candor about limitations over minimizing them** — visitors see what a sparse seed produces, not an optimistic best case
- **We choose direct prose over animated marketing copy** — developer credibility over conversion optimization
- **We choose a live version badge over a static marketing claim** — the version shown is always what was used, verified at build time
- **We choose a single-page layout over deep navigation** — the whole story is visible in one scroll; nothing important is buried

## Constraints We Accept

- No external API calls — all interactions are client-side only
- No build-time data fetching beyond reading `.rootspec.json` for the version
- Dark/light theme must follow system preference by default with manual override
- All interactive features must be keyboard accessible and touch-friendly
- The meta banner must be permanently visible — it cannot be dismissed or hidden

## Definition of Success

A visitor who arrives skeptical of specification methodology should leave understanding:
1. What problem RootSpec solves and who has that problem
2. How the five-level hierarchy works conceptually
3. That the methodology is approachable — they could start a spec for their own product
4. Where to go to learn more or get started

A visitor who already uses specification tooling should leave understanding:
1. What makes RootSpec different from their current approach
2. That the hierarchical traceability model solves problems their current tool does not

## Architectural Commitments

- Built with Astro as the static site framework, React for interactive islands
- Tailwind CSS for styling — utility-first with no custom CSS methodology imposed
- The RootSpec version is read from `.rootspec.json` at build time; it is never hardcoded
- GitHub repository links point to the `main` branch at `https://github.com/rootspec/demos/tree/main/greenfield`
