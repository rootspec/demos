# Level 2: Stable Truths

**References:** 01.PHILOSOPHY.md
**Purpose:** Philosophically committed choices about how this site works

---

## Truth: The site is the proof

The most important design decision is that this site does not merely describe RootSpec — it demonstrates it. The specification, the implementation, and the tests are all public artifacts. The site's existence as a working product, built with the methodology, is the primary argument.

*Supports:* Proof Over Promise pillar (01.PHILOSOPHY.md)

---

## Truth: Transparency over polish

When rough edges exist — imperfect layout, minimal iteration, sparse visual detail — they must be acknowledged, not hidden. The meta-banner is the most critical element on the page because it reframes rough edges as honest artifacts of a constrained seed, not failures of craft. A demo that hides its limitations is a lie; a demo that names them is a proof.

*Supports:* Intellectual Honesty pillar (01.PHILOSOPHY.md)

---

## Truth: Typography carries the argument

Visual decoration competes with the content. This site's aesthetic strategy is to let typography do the work: high-quality serif body text signals essay-quality writing; clean sans-serif handles navigation and UI labels; monospace identifies commands and code. Whitespace creates rhythm. No gradients, no glassmorphism, no hero imagery.

*Supports:* Considered Craft pillar (01.PHILOSOPHY.md)

---

## Truth: Light mode is the default

Dark mode is a toggle, not the default. Editorial sites read better in light mode. Defaulting to dark mode is a signal that a site was generated by an AI optimizing for training data patterns. This site must not look like that.

*Supports:* Considered Craft, Intellectual Honesty pillars (01.PHILOSOPHY.md)

---

## Truth: Interactivity demonstrates, not decorates

Interactive elements exist to make abstract concepts tangible — the hierarchy explorer makes the five levels explorable; the wizard lets visitors apply the methodology to their own idea. Interactions must feel like tools, not animations. No springy physics, no parallax, no decorative motion. Transitions are quick and functional.

*Supports:* Proof Over Promise, Approachable Depth pillars (01.PHILOSOPHY.md)

---

## Truth: All functionality is client-side

No external API calls. No server-side rendering dependencies beyond static site generation. No user data collected or transmitted. The site works fully offline after initial load. This is a commitment to simplicity, privacy, and reliability.

*Supports:* Considered Craft pillar (01.PHILOSOPHY.md); aligns with the methodology's preference for constraints that clarify intent

---

## Truth: One page, complete context

All RootSpec messaging — what it is, why it exists, how to start — must be present on this single page. There is no separate "real" marketing site. The open source CTA is the primary conversion target, alongside the wizard. The site must function as both a demo and a complete product page.

*Supports:* Intellectual Honesty, Proof Over Promise pillars (01.PHILOSOPHY.md)

---

## Trade-offs

- **We choose editorial quality over feature completeness.** A single well-crafted section is worth more than three mediocre ones. Fewer sections, done well, is the right call.
- **We choose honest rough edges over polished fabrication.** The meta-banner exists precisely because iteration was constrained. Hiding the constraint would undermine the thesis.
- **We choose static generation over dynamic personalization.** No user accounts, no tracking, no API calls. The constraint is a feature — it proves the methodology works without infrastructure.
- **We choose light mode over dark mode.** Dark mode is available as a toggle, but the default must signal intentionality.
- **We choose depth over breadth in interactivity.** Two well-crafted interactive elements (hierarchy explorer, wizard) are better than five shallow ones.

---

## Success Criteria

This site succeeds when:
- A skeptical developer can understand what RootSpec is and why it exists within [a short reading time] of landing
- The meta-banner is the first thing a visitor reads after the hero
- The hierarchy explorer communicates the five-level hierarchy without requiring the visitor to read the docs
- The wizard produces a recognizable spec skeleton that feels applicable to the visitor's own work
- The spec files are publicly inspectable and match what the site describes
- The site does not look like a Vercel template, a Series A startup site, or a generic AI-generated dev landing page
