## Design Philosophy

### Truth: Demonstration over description

Every claim about the methodology must be backed by a live example on this site itself. The site is proof-of-concept by design. Visitors should not have to take our word for it — the spec, the seed, the code, and the tests are all publicly accessible. This truth supports the Earned Credibility pillar from L1.

### Truth: Simplicity over completeness

The site does not attempt to teach the full RootSpec methodology — it communicates the core concept clearly and invites further exploration. Every section must justify its existence by contributing to visitor understanding or demonstrating methodology. Complexity that doesn't serve Immediate Clarity is cut. This truth supports the Approachable Depth pillar from L1.

### Truth: Client-side interactivity only

All interactive features run entirely in the browser with no external API calls, no server-side rendering of dynamic data, and no authentication. This is a deliberate constraint: it demonstrates that the methodology does not require complex infrastructure to explain, and it eliminates a category of failure modes that would undermine Earned Credibility.

### Truth: Honest framing is non-negotiable

The site was built with minimal human guidance — a sparse seed, four commands, no design mockups. Rough edges are the result of that process, not negligence. The meta-banner communicates this directly. Hiding or softening this framing would contradict the core message that specification guides implementation. Transparency in the demo serves Developer Respect from L1.

### Truth: Version is a first-class signal

The RootSpec version number is a meaningful data point for developers evaluating the framework. Displaying it prominently — sourced from `.rootspec.json` at build time — is not decorative. It tells visitors exactly what version produced this site and reinforces that the framework is versioned, maintained, and evolving.

## Architectural Commitments

### Static-first generation

The site is built as a static site (Astro framework) with islands of client-side interactivity. Static generation means the core content loads instantly without JavaScript. Interactive components hydrate only when needed. This matches the constraint of no external API calls and maximizes reliability.

### Component isolation

Each section of the page is a self-contained component. Interactive components (Hierarchy Explorer, Spec Wizard, Comparison toggle) own their state internally and do not depend on shared global state. This simplifies implementation and prevents cascading failures.

### Theme system: system preference + manual override

Dark/light mode is determined by the user's system preference by default, with a manual toggle that persists via localStorage. This is the baseline expectation for any developer-facing site and supports the Developer Respect pillar.

### Mobile-first responsive design

All layouts are designed for mobile first and scale to desktop. Interactive features use touch-friendly targets. The hierarchy explorer and spec wizard must be fully functional on touch screens, not degraded fallbacks.

## Trade-offs

- **We choose demonstration over explanation** — The spec wizard shows methodology over describing it, even if some nuance is lost in simplification.
- **We choose static generation over dynamic personalization** — No user accounts, no saved state between sessions, no server-side customization. Simplicity over capability.
- **We choose honest roughness over polished fiction** — The site reflects what a sparse seed produces, not an idealized vision. Authenticity over impression management.
- **We choose depth-on-demand over flat content** — Sections like the hierarchy explorer progressively reveal detail rather than front-loading everything. Scannability over completeness.
- **We choose client-side interactivity over AI-generated content** — The spec wizard uses structured templates, not live AI calls. Reliability over dynamism.

---

## Definition of Success

The site succeeds if:
1. A developer understands the core RootSpec concept (philosophy-guides-implementation) without reading documentation
2. A developer can locate the spec, the seed, and the GitHub repository from the site
3. The interactive features work on both desktop and mobile without error
4. The site accurately represents the RootSpec version it was built with
5. A visitor who runs four RootSpec commands could recreate a similar result

---

## Accepted Constraints

- No backend infrastructure — static hosting only
- No authentication or user accounts
- No external API dependencies at runtime
- All content must be accurate and traceable to the spec or seed
- Version number must be sourced from `.rootspec.json`, not hardcoded
