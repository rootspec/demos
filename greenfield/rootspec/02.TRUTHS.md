# Level 2: Truths

## Design Philosophy

**Purpose-aligned specification over feature completeness**: We prioritize demonstrating methodology over building comprehensive marketing features. Every site element exists to teach or validate RootSpec principles.

## Strategic Trade-offs

### Transparency over Polish
We choose **honest demonstration** over **marketing perfection**.
- This site openly acknowledges it was generated from minimal human guidance
- Rough edges are disclosed as natural outcomes of sparse seed requirements
- The meta-banner explicitly states the site's generated nature

### Interactive Learning over Static Content  
We choose **hands-on exploration** over **passive consumption**.
- Multiple interactive features (hierarchy explorer, spec wizard) over text-heavy explanations
- Visual demonstrations of methodology over theoretical descriptions
- Client-side interactivity over external API dependencies

### Methodology Demonstration over Product Sales
We choose **educational value** over **conversion optimization**.
- Show the RootSpec process in action rather than just describing benefits
- Link to actual spec files and source code for transparency
- Focus on teaching the approach rather than hiding complexity

### Developer-Focused over Marketing-Optimized
We choose **technical authenticity** over **broad appeal**.
- Tone matches thoughtful developer explaining a methodology they believe in
- Technical accuracy over simplification
- Practical examples over buzzword marketing

## Success Metrics

### Primary Success
- **Comprehension Rate**: Visitors understand the five-level hierarchy after exploring the site
- **Methodology Clarity**: Developers can articulate how RootSpec differs from traditional spec approaches
- **Practical Interest**: Technical leads investigate further based on interactive demonstrations

### Secondary Success  
- **Framework Demonstration**: Site serves as proof-of-concept for RootSpec's own methodology
- **Community Discovery**: Open source community finds and contributes to the project
- **AI Development Validation**: Shows reliable AI-assisted development when guided by solid specifications

## Constraints Accepted

### Technical Constraints
- **Client-side only**: No backend dependencies or external API calls
- **Astro framework**: Static site generation with minimal JavaScript
- **Version coupling**: Site version must track and display RootSpec framework version
- **Accessibility standards**: Full keyboard navigation and screen reader compatibility

### Content Constraints
- **Self-documenting**: Site must reference its own spec and seed files
- **Framework agnostic messaging**: Don't oversell specific tooling integrations
- **Honest limitations**: Acknowledge what RootSpec doesn't solve
- **Developer attribution**: Clear identification of site builder and build date

### Resource Constraints
- **Single-pass generation**: Built from one specification cycle without iteration
- **Minimal maintenance**: Site structure should not require frequent updates
- **Documentation parity**: Any feature claims must be demonstrable in the site itself
