# Level 2: Stable Truths — WHAT Strategy

> References: 01.PHILOSOPHY.md

## Design Philosophy

### Truth: Dependency inversion is the architecture
The hierarchy is not a convention — it is a constraint enforced by validation. Philosophy (L1) must never be modified to accommodate implementation decisions (L4–L5). When conflict arises, lower levels change, never higher.

This truth operationalizes the "Wisdom Over Knowledge" pillar: the spec preserves human intent precisely because that intent is written at a level that cannot be casually overwritten.

### Truth: Specification is declaration, not procedure
A spec written in RootSpec describes desired state — what the product should be and feel like — not the steps to achieve it. Implementation details (frameworks, algorithms, data structures) belong at L4 and below. L1–L3 are framework-agnostic by design.

This truth operationalizes "Clarity Without Ceremony": a readable L1 requires no technical knowledge because it contains none.

### Truth: Validation is the product's immune system
The spec is only valuable if it can be violated. A spec that always passes validation has no teeth. Every validation check exists to surface a real divergence between intent and implementation. The four skills form a lifecycle: init → spec → impl → validate, and validate is not optional.

This truth operationalizes "Traceable Trust": trust is earned through repeated validation, not asserted upfront.

### Truth: The spec bridges human and AI
AI can generate implementations from spec; AI cannot generate philosophy from code. The spec is the contractual boundary between human wisdom and machine execution. RootSpec's value proposition is making that boundary explicit, version-controlled, and testable.

This truth operationalizes "Human-AI Partnership": the spec is the handoff artifact, not a byproduct.

## Architectural Commitments

- **Five levels, no more, no less** — The hierarchy is fixed. Products do not add levels or skip levels. L4 may have multiple files (one per system), but the five-level structure is inviolable.
- **Natural language L1–L3, structured YAML L5** — Expressiveness at the philosophy layer; machine-readability at the implementation layer. L4 is structured markdown (human-readable with clear schema).
- **Four commands as the complete workflow** — `/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate`. No hidden steps, no sprawling toolchain. Complexity is in the spec, not in the tool.
- **Client-side and offline-first** — The spec is files in version control. No external services required to read, write, or validate. Internet access is optional, never required for core workflow.

## Trade-offs

- **We choose validation over permissiveness** — The tool surfaces problems rather than silently accepting ambiguous specs. A spec that passes validation has earned that status.
- **We choose natural language over configuration-first** — YAML and structured formats appear only at L5. L1–L3 are prose because product philosophy should be readable by non-engineers.
- **We choose explicit hierarchy over flat organization** — Five ordered levels require more upfront discipline than a flat document. The payoff is that changes can be localized: L4 can change without touching L1.
- **We choose methodology over magic** — RootSpec does not generate features or infer product vision from a codebase. It surfaces and validates intent that humans supply. This is slower than full AI automation and more reliable.
- **We choose CLI over GUI** — The tool lives in the terminal and in version control. This trades discoverability for composability; RootSpec fits into existing dev workflows rather than demanding its own interface.

## Definition of Success

### For the product itself
A team using RootSpec can answer "why does this feature exist?" by pointing to a specific design pillar and tracing it through L2 commitments to L3 interaction patterns to L4 systems to L5 stories. The answer is in the spec, not in someone's memory.

### For the marketing site
A visitor who lands on this site and reads for [N] minutes understands:
1. What problem RootSpec solves
2. How the hierarchy works (conceptually, not just as a list)
3. That this site itself was built using RootSpec (and can see the evidence)
4. Where to start (one clear call to action)

## Constraints We Accept

- The methodology requires upfront investment — you must think before you build. Users who want to skip straight to code are not the target audience.
- Validation is heuristic, not formal proof — the tool can surface likely violations, not guarantee philosophical correctness.
- The spec does not run or deploy anything — it is validation infrastructure, not a build system. Integration with CI/CD is the team's responsibility.
- Natural language specs can be ambiguous — RootSpec reduces ambiguity through structure, but cannot eliminate it. Human judgment remains essential at every level.
