# Level 2: Stable Truths
# RootSpec Marketing Site

## Truth: Philosophy must precede implementation, even in a marketing site

We follow the RootSpec hierarchy in building the site that explains RootSpec. The product spec (L1-L5) is written before any code exists. This is not decoration — it is a live demonstration that the methodology scales from complex enterprise software down to a single-page marketing site.

**Trade-off:** We choose rigorous spec-first process over fast, intuition-driven development.

## Truth: The site is the demo, not just documentation about the demo

The primary persuasive device is the site's own existence as a RootSpec output. Explaining RootSpec by following RootSpec is more convincing than any prose we could write. Every design decision serves this meta-narrative.

**Trade-off:** We choose transparent self-demonstration over conventional marketing polish.

## Truth: Static generation over runtime complexity

The site is a static marketing site. All interactive features are client-side only, with no server state, no API calls, and no authentication. This keeps the demo clean, fast, and auditable by anyone who views source.

**Trade-off:** We choose simplicity and auditability over dynamic personalization.

## Truth: Real content over idealized content

Every section of the site contains actual, specific content — not lorem ipsum, not placeholder wireframes. The before/after comparison shows real spec artifacts. The wizard produces real skeleton specs. The version badge shows the actual version from `.rootspec.json`.

**Trade-off:** We choose authentic demonstration over controlled, idealized presentation.

## Truth: Accessibility is structural, not additive

Dark/light theme with system detection, keyboard navigation for all interactive components, touch-friendly controls, and responsive layout are foundational requirements — not features to add at the end. They reflect the Design Pillar "Accessible Depth" from L1.

**Trade-off:** We choose inclusive design from the start over retrofitting accessibility later.

## Truth: The meta-banner is non-negotiable

Every visitor must understand that this site was generated from a sparse ~100-line seed with no iteration, no manual design, and no manual code. This context reframes rough edges as evidence rather than failures. The banner cannot be dismissed, minimized, or hidden below the fold on any device.

**Trade-off:** We choose honest context-setting over a smooth first impression.

## Truth: The version badge is a proof point, not decoration

The RootSpec version number displayed on the site is read from `.rootspec.json` at build time. It cannot be hardcoded. This demonstrates that the site is a living artifact of the framework, not a static marketing brochure.

**Trade-off:** We choose build-time accuracy over designer-controlled visual control.

## Definition of Success

The site succeeds when:
- A skeptical developer understands the core hierarchy within 60 seconds of landing
- A technical lead can use the interactive wizard to map their own product idea in under 5 minutes
- A visitor can locate the actual spec files and seed document without searching
- The site itself passes its own validation tests — it is not exempt from the methodology it promotes
- The meta-banner is the first thing a visitor reads after the hero — not hidden, not skipped

## Constraints We Accept

- **No external API calls** — All interactivity is client-side; no backend, no AI integration in the wizard
- **No cookies or tracking** — The site collects no user data
- **No JavaScript required for core content** — The spec explanation, hierarchy, and CTA all work without JavaScript; interactive features gracefully degrade
- **Minimal dependencies** — The stack should be explainable in one sentence; no dependency on proprietary platforms
- **Framework-agnostic philosophy** — The site itself demonstrates RootSpec, not a specific framework choice
