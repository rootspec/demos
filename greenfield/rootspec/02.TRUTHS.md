# Level 2: Stable Truths

## Design Philosophy

### Truth: Typography is the primary design medium
The site communicates through writing. Visual design serves the writing; the writing does not serve the layout. A serif body face signals essay-quality content. A monospace face for commands signals precision. The type system is the brand.

**Reference:** L1 Design Pillar "Approachable Rigor" — rigor expressed through clarity of prose, not visual complexity.

### Truth: The aesthetic is the argument
A polished, considered site demonstrates that the methodology produces real results. Generic AI-generated aesthetics (dark backgrounds, gradient blobs, oversized hero text) actively undermine the thesis. The site must look like it was made by someone who cared.

**Reference:** L1 Mission — keeping intent intact requires caring enough to be specific.

### Truth: Light mode default, dark mode as preference
Editorial sites read in light mode. Defaulting to dark is a tell that the site was generated by an AI defaulting to its training distribution. System preference detection plus manual toggle respects user autonomy without making assumptions.

**Reference:** L1 Design Pillar "Human Philosophy, AI Execution" — human preference governs, AI doesn't override.

### Truth: Complexity from composition, not imposition
Simple rules (five-level hierarchy, upward-only references) combine to create rich, enforceable structure. The site explains this by demonstrating it, not by listing features. Interactive elements show rather than tell.

**Reference:** L1 Design Pillar "Approachable Rigor."

## Architectural Commitments

### Commitment: Static-first, client-side interactive
The site is a static marketing site with client-side interactive components. No server-side state, no API calls, no authentication. All interactive features (hierarchy explorer, spec wizard, comparison view) are self-contained and work without network access after initial load.

**Trade-off:** We choose zero runtime infrastructure over personalization or dynamic content. The site cannot adapt to users, but it requires no ops.

### Commitment: Framework-native data sourcing
The RootSpec version number is read from `.rootspec.json` at build time — not hardcoded. The meta-banner links to actual files in the GitHub repository. The site consumes its own spec artifacts.

**Trade-off:** We choose self-referential correctness over convenience. Hardcoding is faster but creates the kind of drift the methodology exists to prevent.

### Commitment: Base path is canonical and consistent
The site deploys to `/demos/greenfield/`. This base path is configured once and used identically in dev, preview, and production builds. No environment-specific switching. Cypress runs against the dev server at the configured subpath.

**Trade-off:** We choose predictability over environment-specific flexibility. Switching base paths between environments creates exactly the kind of invisible inconsistency the spec should prevent.

### Commitment: Responsive with no feature degradation
All interactive features — hierarchy explorer, spec wizard, comparison view — work on mobile via touch. The site is not "mobile-friendly as an afterthought"; interactive elements are designed touch-first and enhanced for pointer devices.

**Trade-off:** We choose interaction parity over desktop-exclusive polish. Some animations may be simpler on mobile.

### Commitment: Keyboard accessible throughout
The hierarchy explorer and spec wizard are fully navigable by keyboard. Focus management, ARIA labels, and semantic HTML are not retrofitted — they are part of the initial implementation.

**Trade-off:** We choose universal access over visual-only interaction design.

## Success Criteria

- A visitor understands what RootSpec is and why it exists within [short reading duration] of arriving
- A visitor can explore the five-level hierarchy interactively without reading documentation first
- A visitor can produce a skeleton spec from a one-line product idea using the wizard
- The meta-banner is visible before the fold on every device and screen size
- The site demonstrates through its own existence that the methodology produces real results
- Every asset URL and internal link resolves correctly under the `/demos/greenfield/` base path

## Constraints

- No external API calls — all features are client-side
- No AI integration — the wizard uses structured templates, not LLM calls
- No authentication, accounts, or server-side state
- The site cannot look like a SaaS landing page, a startup pitch, or a Vercel template
- The meta-banner is not optional and must appear above the fold
