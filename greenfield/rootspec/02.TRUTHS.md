# Level 2: Stable Truths

References: [01.PHILOSOPHY.md]

## Truth: The spec is the source of truth, not the code

The specification is not a description of the code — the code is an artifact produced by the spec. When specification and implementation diverge, the specification wins. Code is regenerable; validated intent is not.

This means the spec must be kept alive, not archived. Every feature addition, every pivot, every trade-off decision must be reflected in the spec before implementation begins.

*Supports: Validated Intent, Philosophy First*

## Truth: Hierarchy is the mechanism of stability

The five-level hierarchy is not organizational preference — it is the technical constraint that makes the spec stable under change. Philosophy (L1) does not change when implementation details (L5) change. Strategy (L2) does not break when interaction patterns (L3) evolve.

Any specification system that allows lower-level details to contaminate higher-level principles will exhibit drift. The reference hierarchy prevents this structurally, not procedurally.

*Supports: Philosophy First, Validated Intent*

## Truth: Validation gates are the primary output

The value of RootSpec is not the spec document — it is the validation gate. A spec that cannot validate an implementation is a PRD. A spec that can validate an implementation is a build target.

We choose executable validation over document completeness. A shorter spec that can be run is worth more than a comprehensive spec that cannot be tested.

*Supports: Validated Intent, Honest Progress*

## Truth: AI executes; humans supply philosophy

AI tools are extraordinarily capable at generating implementations that satisfy stated constraints. They are not capable of supplying the unstated assumptions, implicit priorities, and philosophical commitments that make software coherent over time. That is the human's job.

RootSpec is designed around this division: the interview and spec authoring process extracts human wisdom; the implementation and validation pipeline is handled by AI. Neither works well doing the other's job.

*Supports: Philosophy First, Approachable Rigor*

## Truth: Natural language is the specification medium

The specification must be readable by the humans who hold the intent — product owners, founders, domain experts who may not be engineers. A specification written in config files or code is only accessible to engineers, which means the people who can catch philosophical drift are excluded from the review process.

We choose natural language specification over machine-friendly config. AI can parse natural language; humans can read and challenge it. This is a deliberate trade-off: we accept some ambiguity in natural language in exchange for broad accessibility.

*Supports: Approachable Rigor, Philosophy First*

## Truth: Interactive features are proof of methodology

The RootSpec marketing site is itself a RootSpec demo. Every interactive element on the site should demonstrate that the methodology produces real, working software — not just compelling arguments. The site's existence is the argument.

Interactive features must be real implementations, not mockups. If an interactive element fails to demonstrate the methodology, it undermines the methodology's credibility.

*Supports: Honest Progress, Validated Intent*

## Truth: Visual quality is part of the argument

A marketing site for a methodology that claims to produce quality software must itself be high-quality. A generic AI-generated site look actively undermines the thesis. The design of this site is not decoration — it is evidence.

We choose editorial quality over generic web aesthetics. Generous whitespace, considered typography, and restrained palette signal that someone thought carefully. These choices support the claim that the methodology produces considered, intentional results.

*Supports: Honest Progress, Approachable Rigor*

## Definition of Success

The RootSpec marketing site succeeds when:

- A developer who has been burned by spec drift reads the site and recognizes their problem described accurately
- A developer who has struggled with AI-generated code recognizes the validation gap RootSpec addresses
- A visitor who arrived skeptical can explain the methodology to a colleague after reading the page
- The "Spec Your Idea" wizard gives a visitor a concrete sense of what RootSpec output looks like
- The Hierarchy Explorer makes the five-level structure tangible without requiring documentation reading
- The meta-banner is the first thing visitors see and clearly frames the demo context

## Trade-offs

- **Rigor over speed** — The methodology takes more upfront investment than writing code directly; we accept this because the long-term cost of drift exceeds the upfront cost of spec
- **Natural language over machine precision** — Some ambiguity is acceptable in exchange for human readability; we mitigate this through structured validation, not through replacing natural language
- **Tool-agnostic over opinionated** — Not prescribing a specific stack means the methodology must be general enough to work with any tooling; we accept this complexity to avoid locking users in
- **Light mode default over dark mode** — Editorial sites read better in light mode; defaulting to dark mode would signal AI-generated aesthetics; we choose light mode as the editorial default
- **Client-side only over server features** — All interactive features run client-side, eliminating the need for authentication, APIs, or backend infrastructure; this limits feature depth but dramatically reduces complexity and deployment risk
