# L2: Stable Truths

## Design Philosophy

Simplicity over completeness. This is a marketing and demonstration site, not a documentation portal. Every section must earn its place by either explaining the methodology or demonstrating it. Nothing exists for decoration.

## Trade-offs

- **We choose honest roughness over polished fiction.** A sparse seed produces an imperfect result. We show that result rather than masking it. (Supports: Credibility through honesty)
- **We choose demonstration over description.** Interactive elements that let visitors experience the methodology are more valuable than paragraphs that explain it. (Supports: Tangible over abstract)
- **We choose client-side simplicity over server-side power.** No backend, no API calls. Everything runs in the browser. This constrains features but eliminates reliability risk. (Supports: Clarity over cleverness)
- **We choose focused content over exhaustive coverage.** Five sections, each tight. We resist adding more because the bar is: does this help a burned developer understand RootSpec and want to try it? (Supports: Confident but not preachy)
- **We choose system preference defaults with manual override for theming.** Dark/light theme follows the user's OS by default, with a visible toggle. We do not force a single theme. (Supports: Approachable depth)

## Success Criteria

- A developer unfamiliar with RootSpec can understand what it is and why it exists within 60 seconds of landing on the page.
- The meta banner is immediately visible and clearly explains the demo context — including links to the spec and seed files.
- The hierarchy explorer communicates the five-level dependency structure without any supporting text required.
- The spec wizard produces a recognizable skeleton spec from three inputs.
- The before/after comparison makes the value of structured specification viscerally obvious.
- The RootSpec version is visible without scrolling on desktop.
- All interactive features are functional on mobile without layout breakage.
- No external network requests are made after initial page load.

## Constraints We Accept

- No authentication, no data persistence, no user accounts.
- No server-side rendering beyond static site generation at build time.
- No third-party analytics or tracking scripts.
- The build must complete without manual intervention after the four RootSpec commands run.
- Visual polish is secondary to functional correctness — this is a demo, not a product launch.

## What We Are Not Building

- A full documentation site (that comes later, separately).
- A live coding environment or playground.
- A multi-page application with routing.
- A community or forum platform.
- A sign-up or waitlist system.
