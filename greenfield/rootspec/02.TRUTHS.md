## Design Philosophy

### Truth: Philosophy is infrastructure, not documentation

The spec hierarchy is not a document to be written and filed — it is a living constraint system that governs every downstream decision. This shapes everything: the methodology enforces top-down dependency, not bottom-up rationalization.

*Supports: Intellectual Clarity, Methodological Trust*

### Truth: Demonstration beats explanation

The most persuasive argument for RootSpec is this site. Every interactive element, every section, and the meta-banner together constitute proof that the methodology is real, usable, and honest. Copy and marketing language take a back seat to showing the work.

*Supports: Approachable Rigor, Earned Confidence*

### Truth: Structured templates over open-ended AI

Interactive features are fully client-side and template-driven. No external API calls. The wizard and hierarchy explorer demonstrate structured thinking — the kind RootSpec enforces — rather than AI generation. This prevents the irony of a spec-validation tool depending on unvalidated AI output at runtime.

*Supports: Methodological Trust, Earned Confidence*

## Trade-offs

### We choose validation over generation

RootSpec validates alignment between philosophy and code; it does not generate code from scratch. Users who want a code generator should look elsewhere.

*Trade-off: RootSpec over AI-first generation tools*

### We choose honest roughness over polished approximation

The meta-banner explicitly acknowledges this site's rough edges as the product of minimal human guidance. We do not hide or apologize for them — they are the demonstration.

*Trade-off: Authenticity over polish*

### We choose hierarchy over flexibility

Every spec level has a fixed place and fixed reference rules. There is no "custom levels" or "skip L2" mode. The rigidity is the point — it prevents circular dependencies.

*Trade-off: Structural integrity over configurability*

### We choose client-side simplicity over dynamic personalization

No server-side rendering of user-specific content, no external API calls, no authentication. The site is a static demonstration, not a SaaS product.

*Trade-off: Simplicity and reliability over dynamic personalization*

### We choose natural language over configuration-first

Specs are written in human-readable markdown, not YAML-first config or code annotations. YAML is used only in L5 (user stories and fine-tuning) where structure provides testability.

*Trade-off: Readability and approachability over machine-parseability at higher levels*

## Success Definition

The site succeeds when:

- A developer who lands on the page understands what RootSpec is within [brief reading time] without clicking anything
- A visitor who uses the Hierarchy Explorer or Spec Wizard leaves with a mental model of the five-level hierarchy they could explain to a colleague
- The meta-banner makes a visitor curious about the seed file rather than skeptical of the site's quality
- The site demonstrates RootSpec's own methodology — the spec, implementation, and tests are all traceable to this hierarchy

## Constraints

- **Static deployment** — The site deploys to GitHub Pages at the subpath `/demos/greenfield/`. All asset URLs and internal links must resolve correctly from that subpath.
- **No external API calls** — All interactivity is client-side only.
- **Framework version visibility** — The RootSpec version (read from `.rootspec.json`) must be visible in the hero and/or header.
- **Responsive and accessible** — All interactive features work on mobile and are keyboard-navigable.
- **Dark/light theme** — System preference detection with manual toggle.
