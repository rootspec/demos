# Level 2: Stable Truths

## Design Philosophy

### Truth: Philosophy guides implementation — never vice versa
The five-level hierarchy exists to enforce one direction of dependency. L1 (why) must never be altered to accommodate L5 (how much). When the spec seems to conflict with code reality, the code is wrong — not the philosophy.

*Supports: Clarity Over Completeness, Philosophy Stability (L1)*

### Truth: Human wisdom + AI execution, not AI autonomy
AI systems have knowledge without wisdom. They can generate valid code that violates product intent. RootSpec's role is to encode human wisdom (philosophy, trade-offs, interactions) into a structured document the AI must satisfy — not merely consult.

*Supports: Validation as First Principle, Philosophy Stability (L1)*

### Truth: Validation is a conversation, not a gate
The spec identifies gaps and inconsistencies. It does not block merges or shame developers. Friction belongs in the spec review, not the deployment pipeline.

*Supports: Approachability Without Sacrifice (L1)*

### Truth: Natural language is the spec — not code
Specifications written in YAML-first or code-first formats create a different kind of lock-in: tooling becomes the authority. RootSpec uses natural language at every level except testable acceptance criteria.

*Supports: Approachability Without Sacrifice, Clarity Over Completeness (L1)*

## Architectural Commitments

### Commitment: Static site, client-side only
The marketing site has no server-side state, no database, no user accounts. All interactivity runs in the browser. This eliminates an entire class of infrastructure, security, and latency concerns.

*Trade-off: We choose simplicity and deployability over personalization and analytics.*

### Commitment: Framework version is surfaced, not hidden
The site reads the RootSpec version from `.rootspec.json` at build time and displays it. The version is a demonstration artifact — this site is evidence of what the framework produces.

*Trade-off: We choose transparency over polish — rough edges are acknowledged, not hidden.*

### Commitment: Interactive features replace marketing copy
Rather than describing the hierarchy and wizard in prose, we implement them as interactive components. The demo IS the explanation.

*Trade-off: We choose experience over brevity — visitors spend more time but leave with deeper understanding.*

### Commitment: No external API calls
All interactive features (Hierarchy Explorer, Spec Wizard, Comparison) run entirely client-side with structured templates. No third-party dependencies for core functionality.

*Trade-off: We choose reliability and privacy over dynamic personalization.*

### Commitment: The site is a RootSpec demo
The meta banner is not optional. Every visitor must understand this site was produced from a sparse seed file using four commands. This honesty is itself a product claim.

*Trade-off: We choose credibility over polish — acknowledging limitations builds more trust than hiding them.*

## Definition of Success

The site succeeds when a developer who has never heard of RootSpec:

1. Understands the core concept (philosophy-guided hierarchy) within [N] seconds of landing
2. Completes the Spec Wizard and sees their idea mapped to L1-L3
3. Leaves with a concrete next step (run `/rs-init` on their project)
4. Trusts that rough edges are the honest result of minimal guidance, not carelessness

## Constraints We Accept

- **Subpath deployment constraint** — All URLs must resolve correctly under `/demos/greenfield/` base path
- **Static generation constraint** — No server-side rendering; all data must be available at build time
- **Accessibility constraint** — All interactive components must be keyboard-navigable; touch targets must meet mobile standards
- **Honesty constraint** — We do not claim the output is polished; the meta banner exists specifically to prevent misinterpretation
