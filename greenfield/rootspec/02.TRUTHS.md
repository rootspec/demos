# L2 — TRUTHS
*WHAT strategy*

## Design Philosophy

**Show, don't tell** — Interactive demonstration over verbose explanation. The methodology becomes tangible through hands-on exploration, not academic theory.

## Trade-offs

- **Transparency over perfection** — We choose honest revelation of limitations over polished marketing facade
- **Demonstration over documentation** — We choose working examples over comprehensive explanations  
- **Interaction over information** — We choose hands-on exploration over passive consumption
- **Simplicity over completeness** — We choose focused core concepts over exhaustive feature coverage
- **Client-side over server-side** — We choose zero external dependencies over dynamic personalization
- **Framework showcase over framework hiding** — We choose visible Astro+React+Tailwind over abstract presentation

## Success Criteria

**Primary:** Visitors understand RootSpec solves real problems they have experienced, can articulate the methodology's value proposition, and know how to start using it.

**Secondary:** The site demonstrates its own creation story credibly — visitors believe this quality is achievable through the RootSpec pipeline with minimal human iteration.

**Tertiary:** Technical leads recognize this as a practical methodology for their teams, not just an interesting concept.

## Constraints

**Performance:** All interactions must run client-side with no external API calls — the site must work offline after initial load.

**Accessibility:** Full keyboard navigation and screen reader support for all interactive features — the methodology must be inclusive.

**Responsiveness:** All features must work on mobile with touch-friendly interactions — developers work across devices.

**Maintenance:** Single source configuration for version numbers and framework details — the site must stay current without manual updates across multiple files.
