# Level 2: Stable Truths

## Truth: The site is the argument

The most persuasive demonstration of RootSpec's value is the site itself — built from a seed document using the four skills, with the spec and code publicly visible. No copy can substitute for this proof. Every design and content decision must protect and amplify this argument, not distract from it.

*Supports: Proof Over Promise pillar (01.PHILOSOPHY.md)*

## Truth: Validation gates everything

In the RootSpec methodology, validation is not a phase — it is the continuous thread connecting intent to output. The site must communicate this clearly: the spec validates the code, the user stories are the tests, and generation is only as good as the verification that follows it. This message must not be buried in implementation detail.

*Supports: Mission (01.PHILOSOPHY.md)*

## Truth: Philosophy-first hierarchy is the core differentiator

What makes RootSpec different from TDD, BDD, and AI-assisted coding tools is the strict five-level hierarchy where each level can only reference higher levels. This prevents drift at the source. The site's information architecture must make this hierarchy tangible — not described abstractly but experienced interactively.

*Supports: Clarity Over Cleverness pillar (01.PHILOSOPHY.md)*

## Truth: Light mode is the default; typography is the primary design tool

We choose editorial legibility over developer-aesthetic convention. Dark-mode-first signals AI generation defaults; light-mode-first with quality typography signals intentional authorship. The design conviction is itself part of the argument.

*Trade-off: We accept that some visitors will prefer dark mode. We provide the toggle but do not default to it.*

*Supports: Editorial Restraint pillar (01.PHILOSOPHY.md)*

## Truth: Interactive elements demonstrate, they do not decorate

The hierarchy explorer and spec wizard exist to make the methodology concrete — they are working tools, not marketing widgets. They must function correctly on all devices, be keyboard accessible, and produce real output that visitors can reason about. Animations are functional, not expressive.

*Trade-off: We accept more implementation complexity in exchange for demonstrating the methodology firsthand.*

*Supports: Proof Over Promise, Approachability Without Condescension pillars (01.PHILOSOPHY.md)*

## Truth: No external API calls; everything runs client-side

The site has no backend. The wizard, hierarchy explorer, and all interactive features operate entirely in the browser. This eliminates infrastructure complexity, protects visitor privacy, and ensures the site works identically in all environments.

*Trade-off: We accept that the wizard cannot call a real AI. Structured templates produce a skeleton spec — useful and accurate, but not AI-generated output.*

## Truth: The four skills are the product's interface

`/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate` — these four commands are how developers interact with RootSpec. The site must present them as a coherent workflow, not a list of features. The sequence matters: init establishes the scaffold, spec captures intent, impl executes, validate confirms.

*Supports: Clarity Over Cleverness pillar (01.PHILOSOPHY.md)*

## Truth: Honest about rough edges

The meta-banner must communicate clearly that this site was produced from minimal guidance — a sparse seed, four commands, no manual code. Rough edges are evidence of the methodology's current state, not carelessness. Transparency here is more credible than polish that obscures the process.

*Supports: Proof Over Promise, inviolable principles (01.PHILOSOPHY.md)*

## Definition of Success

- A developer who arrives skeptical of yet another methodology leaves understanding what RootSpec actually does
- The "Spec Your Idea" wizard is completed by a meaningful proportion of visitors
- The meta-banner links are clicked — visitors inspect the spec and seed
- The GitHub repository receives stars and `/rs-init` installs from visitors who arrived via this site

## Constraints

- Static site: no server-side rendering of dynamic content, no database, no authentication
- Deployed to GitHub Pages at `/demos/greenfield/` — all asset URLs and internal links must resolve under this base path
- Version displayed on site must be read from `.rootspec.json` at build time — never hardcoded
- All GitHub links must use absolute URLs to the `rootspec/demos` repository; relative links will break the static prerenderer
