# Level 2: Stable Truths — WHAT Strategy

*References: 01.PHILOSOPHY.md*

## Design Philosophies

### Truth: Hierarchy is the core mechanism, not a nice-to-have

The five-level hierarchy is what makes RootSpec useful. Each level can only reference higher levels — never lower. This is not a convention or a preference; it's what prevents philosophy from becoming contaminated by implementation details.

Without this constraint, every other spec tool degrades into the same mess: tightly coupled decisions, cascading changes that corrupt foundational intent, and a document graveyard.

### Truth: Declarative over prescriptive

Specifications describe WHAT and WHY, not HOW. Implementation strategy belongs at L4; philosophy belongs at L1. When a spec describes implementation details at the wrong level, it becomes fragile — any refactor breaks the spec.

The spec must be a stable abstraction above the code, not a shadow of it.

### Truth: Executable specs, not aspirational ones

A spec that cannot be validated against code is decoration. RootSpec specs generate real tests, catch real violations, and block real regressions. This is the difference between "we have a spec" and "our spec has power."

The four-skill workflow (`/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate`) produces artifacts that execute, not artifacts that describe.

### Truth: Natural language first, structure second

Developers don't think in YAML. They think in intent. The specification language prioritizes prose and hierarchy over rigid schemas. Structure emerges from dialogue, not from filling in forms.

This lowers the barrier to adoption: any thoughtful developer can write a RootSpec spec without learning a new grammar.

## Trade-offs

- **We choose validation over speed** — Running `/rs-validate` adds friction, but that friction catches real problems before they compound
- **We choose hierarchy over flexibility** — Not every spec fits cleanly into five levels; some teams will push back. We accept that friction because the hierarchy is what prevents spec drift.
- **We choose human intent over AI autonomy** — AI executes; humans define purpose. This limits what AI can do unilaterally, but preserves alignment.
- **We choose correctness over completeness** — A partial spec that is correct is more valuable than a complete spec that is superficial.
- **We choose methodology over tooling** — RootSpec is a thinking framework first. The CLI, the YAML, the tests are scaffolding. Developers who understand the methodology can apply it without any of the tooling.

## Definition of Success

RootSpec succeeds when:
- A developer can onboard onto a complex codebase by reading the spec from L1 downward
- An AI-generated feature can be accepted or rejected based on spec validation, not gut feel
- Philosophy documents remain stable across multiple implementation cycles
- Teams stop fighting about "what were we thinking when we built this" because the spec answered it in advance

## Constraints We Accept

- **Requires upfront investment** — Writing a good L1-L2 takes time. Teams that want to skip to implementation will resist this. RootSpec is not for throwaway prototypes.
- **Not a code generator** — RootSpec validates intent and guides implementation. It does not write application code directly. Teams must pair it with AI coding tools or standard development.
- **Spec maintenance is real work** — When products change significantly, specs need updating. RootSpec makes this explicit and structured, but does not eliminate it.
- **Opinionated on hierarchy** — Teams that prefer flat documentation or fluid spec approaches will find RootSpec constraining. That constraint is the product.

## Site-Specific Truths

### Truth: The marketing site is itself a RootSpec demo

Every implementation decision on this site must be traceable to this spec. The site exists as proof-of-concept: a complete product built from a sparse seed with no manual design mockups and no iteration beyond the automated pipeline.

Rough edges are honest signal, not failures. The meta-banner must communicate this clearly so visitors understand what they're seeing.

### Truth: Interactivity demonstrates methodology

Static marketing copy can describe RootSpec. Interactive features let visitors experience it. The hierarchy explorer and spec wizard exist to make the methodology tangible, not to be impressive demos.

If an interactive feature doesn't deepen understanding of RootSpec, it doesn't belong on this site.
