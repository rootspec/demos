**Product:** RootSpec Marketing Site

**Last Updated:** 2026-04-12

## Design Philosophy

### Truth: The site is the proof, not the pitch
This site does not market RootSpec by describing its benefits — it demonstrates them. Every section exists to instantiate the methodology, not to make claims about it. The meta banner, the interactive sections, the visible version number, and the spec files in the GitHub repo are all evidence. If the methodology works, the site proves it. If the site has rough edges, those edges are honest evidence of what a sparse seed produces.

*References: [Earned Clarity pillar, Transparent Provenance pillar — 01.PHILOSOPHY.md]*

### Truth: Show before tell
Visitors encounter interactive demonstrations before they read explanations. The hierarchy explorer precedes a full description of RootSpec's levels. The "Spec Your Idea" wizard lets visitors apply the methodology before studying it. Passive reading is a last resort.

*References: [Tangible Methodology pillar — 01.PHILOSOPHY.md]*

### Truth: Client-side completeness
All interactive features run in the browser with no external API calls. This is both a technical commitment and a philosophical one — it keeps the demo self-contained and prevents the site from becoming unavailable due to backend failures. Trust is earned through reliability.

*References: [Inviolable principle: No external API calls — 01.PHILOSOPHY.md]*

### Truth: Honest about limitations
The meta banner is mandatory, prominent, and direct. It tells visitors that this site was generated from a sparse seed and links to the raw inputs. Acknowledging limitations builds credibility; hiding them destroys it.

*References: [Transparent Provenance pillar, Inviolable principle: Never obscure rough edges — 01.PHILOSOPHY.md]*

## Architectural Commitments

### We choose progressive disclosure over comprehensive upfront explanation
The site introduces each concept at the moment it's useful — not in a single wall of text at the top. Visitors who scan will find the interactive sections; visitors who read will find the deep explanation behind each one.

**Choosing this over:** front-loading all methodology content in a single "How It Works" section

### We choose static generation over dynamic delivery
Content is pre-rendered where possible. The marketing site has no authentication, no user accounts, and no server-side personalization. This simplifies deployment, maximizes performance, and eliminates entire categories of failure.

**Choosing this over:** server-rendered personalized content, CMS-driven copy

### We choose system preference + manual override for theming
Dark/light mode follows the visitor's system preference by default. A visible toggle allows manual override. The preference is persisted locally. This respects user intent without requiring account creation.

**Choosing this over:** forcing a single theme, requiring sign-in to persist preference

### We choose structured templates over AI for the wizard
The "Spec Your Idea" wizard generates a skeleton spec from the visitor's inputs using structured templates — no AI call required. This keeps the demo self-contained and demonstrates that the methodology is approachable without requiring AI assistance.

**Choosing this over:** live AI generation via API, which would require API keys and external dependencies

### We choose SVG diagrams over external image assets
Visual representations of the RootSpec methodology (the hierarchy, the spec-as-validation-gate) are built as inline SVG where possible. This ensures diagrams render correctly regardless of CDN availability and can be themed with CSS.

**Choosing this over:** raster image exports, external diagramming tool embeds

## Definition of Success

This site succeeds when a developer who has never heard of RootSpec can:
1. Understand the five-level hierarchy and why the reference rules matter
2. Experience running three steps of the "Spec Your Idea" wizard and recognize the methodology in their own output
3. Find the GitHub repo and the getting started instructions without searching
4. Feel that the methodology is for developers like them — not for enterprise processes or academic frameworks

Secondary success: the site itself, as a demo, accurately represents what a sparse-seed RootSpec run produces. Its quality is evidence, not marketing.

## Constraints Accepted

- **Minimal human iteration:** The site is generated from a sparse seed with no design mockups or iterative review. Visual polish is bounded by what the methodology produces from minimal guidance.
- **No backend:** All interactivity must work client-side. Features that require a server (authentication, AI calls, analytics) are out of scope.
- **Version-pinned:** The displayed RootSpec version must match the `.rootspec.json` version field at build time. This is a verifiable claim.
- **Accessibility baseline:** All interactive sections must be keyboard-navigable and screen-reader-compatible. This is not negotiable (see Respectful Intelligence pillar).
- **Mobile parity:** Interactive features must be touch-friendly. A visitor on a phone should experience the full methodology demo.
