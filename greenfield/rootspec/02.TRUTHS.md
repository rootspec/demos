# Level 2: Stable Truths

## Design Philosophy

### Truth: The spec is the source of truth, not a byproduct of development

Specifications in most workflows are written first and then abandoned. In RootSpec, the spec is the living center of the workflow — it generates user stories, gates implementation, and is updated when legitimate product pivots occur. Code and tests exist to prove the spec is satisfied, not the other way around.

### Truth: Philosophy must be stable; implementation must be flexible

The five-level hierarchy enforces a one-way dependency. The highest levels (why, what experience) should rarely change. The lowest levels (how, how much) change constantly. This asymmetry is intentional — it prevents philosophical drift without preventing technical evolution.

### Truth: Validation is not a phase — it is the architecture

Traditional development treats validation as a step near the end. RootSpec treats it as the structural constraint that everything else is built around. Every user story is a validation claim. Every implementation is a proof. Nothing ships without passing the spec.

### Truth: AI executes; humans supply wisdom

AI systems are capable of extraordinary generation but have no authentic intent. RootSpec uses AI to do what it is good at (generating user stories, implementing code, running tests) while keeping humans in control of what AI cannot provide (philosophy, trade-offs, design pillars). The spec is the contract between human wisdom and machine execution.

### Truth: Complexity must be earned, not imposed

The methodology must not require more process than the product justifies. A small product with a one-line seed should produce a lightweight spec. A complex multi-team product should produce a rich one. The workflow scales with the problem, not with the team's appetite for documentation.

## Architectural Commitments

### The five-level hierarchy is non-negotiable

Every concern lives at exactly one level. L1 cannot reference L3. L4 cannot reference L5. Violations are errors, not warnings. This constraint is what gives the spec its structural integrity — without it, the hierarchy is just a filing system.

### The site is itself a demonstration of the methodology

The RootSpec marketing site is built using RootSpec. The meta-banner is not a novelty; it is the primary proof-of-concept. Every visitor should be able to trace the site's design decisions back to this spec. The site's roughness (where it exists) is honest evidence of what a sparse seed produces — not carelessness.

### Static output, client-side interactivity

The site renders as a static build deployed to GitHub Pages. Interactive features (hierarchy explorer, spec wizard, before/after comparison) run entirely client-side with no external API calls. This constraint enforces simplicity and eliminates operational cost.

### Light mode as default, dark mode as toggle

Editorial reading is the primary mode of the site. Light mode serves long-form reading better than dark mode. Dark mode is available for users who prefer it, detected from system preferences with a manual override. Defaulting to dark mode would signal AI-generated output — the opposite of the intended aesthetic.

### Typography is the primary design element

The site's visual quality argument rests on typography, not decoration. High-quality serif for body copy (essay-quality signal), clean sans-serif for UI elements, monospace for code and command examples. No gradients, no glassmorphism, no decorative effects that compete with the content.

## Trade-offs

- **We choose honest roughness over polished fiction.** A sparse seed produces a sparse site. Where the demo shows limitations of minimal guidance, we label them rather than papering over them.
- **We choose editorial depth over SaaS landing page breadth.** The site reads like a thoughtful essay, not a feature matrix. Users who want to understand get more; users who want a quick pitch get less. This is correct.
- **We choose client-side simplicity over server-side capability.** No API calls, no personalization, no dynamic content. What you see is what everyone sees. This eliminates operational risk and keeps the demo honest.
- **We choose five commands over infinite configurability.** The CLI surface is intentionally small. Advanced users who want more control have the spec itself — they can read it, fork it, extend it. The tool doesn't try to accommodate every workflow variation.
- **We choose visible validation over invisible quality.** The meta-banner, the spec links, the version badge — these are not decorative. They are proof that the methodology produces traceable, auditable results.

## Definition of Success

The site succeeds when:
- A developer visits, reads the author's notes, and thinks "this is exactly the problem I have"
- A technical lead clicks "Spec Your Idea" and gets a skeleton spec that feels like a useful starting point
- A team considering AI-assisted development understands why validation matters more than generation
- The GitHub repo receives stars from developers who are adopting the methodology, not just bookmarking it

The site fails when:
- Visitors think it's a generic marketing site for another AI tool
- The meta-banner is missed because it's buried or visually weak
- The author's voice is diluted or paraphrased in the "Why I Built RootSpec" section
- The interactive elements feel like toys rather than tools

## Constraints

- **No external API calls** — all interactivity is client-side; no keys to manage, no rate limits, no privacy concerns
- **No numeric values in L1-L4** — actual timing, sizing, and tuning values live in L5 fine-tuning only
- **Base path `/demos/greenfield/`** — all URLs, assets, and links resolve under this subpath in dev, preview, and production
- **GitHub Pages deployment** — static output only; no server-side rendering, no edge functions
- **Absolute GitHub URLs** — all links to SEED.md and spec files must use absolute URLs to the GitHub repo; relative links break the static build's prerenderer
