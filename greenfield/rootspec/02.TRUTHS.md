# Level 2: Stable Truths

> References: L1 (01.PHILOSOPHY.md)

## Design Philosophy

Simplicity over completeness. Every section earns its place by demonstrating the methodology — not by adding marketing copy. Content that doesn't teach or show is cut.

## Trade-offs

- **We choose interactivity over comprehensive documentation** — Three focused interactive tools teach more than ten pages of prose
- **We choose static generation over server-side rendering** — No backend simplifies deployment, reduces attack surface, and proves the "no external API" pillar
- **We choose honest transparency over polished marketing** — The meta-banner explicitly names what this is and what it isn't; visitors get the real story
- **We choose dark/light theme system preference over forced dark mode** — Respects developer environment preferences without imposing aesthetics
- **We choose Astro + React islands over a full SPA** — Static-first with interactive islands gives fast load, good accessibility, and correct behavior for interactive components
- **We choose absolute GitHub URLs over relative spec links** — Static prerenderer cannot resolve relative paths at build time; correctness over convenience
- **We choose subpath deployment over root deployment** — All asset URLs and internal links must resolve under `/demos/greenfield/` for GitHub Pages hosting

## Success Criteria

- A developer with no prior RootSpec knowledge understands the core concept within 90 seconds
- All three interactive features (Hierarchy Explorer, Spec Wizard, Before/After Comparison) work on mobile touch devices
- The site passes keyboard navigation for all interactive elements (WCAG AA)
- Zero broken links — all GitHub URLs resolve to actual files in the repo
- The meta-banner clearly communicates what the site is and links to the spec and seed
- Page load time under N seconds on a standard connection (value defined in fine-tuning)
- The RootSpec version displayed matches the version in `.rootspec.json`

## Constraints We Accept

- **No AI at runtime** — Interactive tools use templates and structured logic; no LLM calls from the browser
- **No user accounts or persistence** — Wizard state is ephemeral; nothing is saved between sessions
- **Rough-edge tolerance** — A sparse seed produces a sparse result; the site doesn't need to look like a funded product
- **GitHub Pages subpath** — All URLs must account for the `/demos/greenfield/` base path
- **Build-time version injection** — The RootSpec version is read at build time from `.rootspec.json`, not dynamically fetched

## Key Messages (canonical wording)

- "When code and specs can be generated trivially, the real value is validation and proof."
- "AI has knowledge but not wisdom. Humans supply philosophy; AI executes implementation."
- "Four skills drive all workflows: `/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate`"
- "Not a code generator — extracts and validates human intent"
- "Not a gatekeeper — identifies problems, doesn't block work"
