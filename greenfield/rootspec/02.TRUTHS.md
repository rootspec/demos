# Level 2: Stable Truths

**References:** `01.PHILOSOPHY.md`

---

## Truth 1: Show over tell

The site must demonstrate RootSpec rather than describe it. Every section that explains a concept should also let the visitor experience it. Explanation without experience produces skepticism; experience without explanation produces confusion. Both are needed, in that order.

*Supports Pillars: Earned Clarity, Demonstrable Proof*

---

## Truth 2: The medium is the message

This site was built using RootSpec. That fact is not incidental — it is the primary evidence. The meta banner, the version badge, and the links to actual spec files are not decorations; they are proof of concept. Hiding or downplaying how the site was built would contradict the product's core claim.

*Supports Pillars: Demonstrable Proof, Confident Credibility*

---

## Truth 3: Honest over polished

A site that acknowledges its own limitations is more credible than one that hides them. The meta banner must be direct: this was generated from a sparse seed with minimal human guidance. Rough edges are expected and should be attributed honestly. Visitors who discover authenticity trust the methodology more, not less.

*Supports Pillars: Confident Credibility*

---

## Truth 4: Depth without gatekeeping

The interactive features (Hierarchy Explorer, Spec Wizard, Before/After Comparison) must be approachable on first contact — no sign-up, no setup, no prerequisites. Visitors should be able to experience the methodology within seconds of arriving. Depth is available for those who seek it; it is never required to get value.

*Supports Pillars: Approachable Depth, Earned Clarity*

---

## Truth 5: Client-side only architecture

All interactivity runs entirely in the browser. No external API calls, no server-side processing, no authentication. This is both a technical constraint (no external dependencies at runtime) and a philosophical commitment (the methodology works without infrastructure). Structured templates, not AI, power the Spec Wizard.

*Supports Pillar: Approachable Depth*

---

## Truth 6: System preference respects user intent

Dark/light theme follows the visitor's system preference by default. Manual override is available and persists. This is not a cosmetic feature — it reflects a core respect for user autonomy that the methodology also demands.

*Supports Pillar: Confident Credibility*

---

## Truth 7: Version is a first-class signal

The RootSpec version the site was built with is displayed prominently. Version is not metadata — it is a statement of which iteration of the methodology produced this outcome. Visitors comparing versions over time can observe how the methodology itself evolves.

*Supports Pillar: Demonstrable Proof*

---

## Trade-offs

| We choose | Over | Rationale |
|-----------|------|-----------|
| Honest rough edges | Polished marketing surface | Credibility requires honesty about process |
| Client-side simplicity | Server-side power | No infrastructure = no external dependencies, no point of failure |
| Structured templates (Spec Wizard) | AI generation | Demonstrates methodology is approachable without AI |
| Sparse seed + four commands | Long design process | The seed-to-site pipeline is itself the product demo |
| Developer voice | Corporate copywriting | Audience is developers who distrust marketing language |

---

## Success Criteria

The site succeeds when:
- A skeptical developer understands what RootSpec does in under [brief reading time]
- A curious developer can try the Spec Wizard and produce a skeleton spec for their own idea without reading documentation
- A technical lead can follow the meta banner links and audit the full pipeline (seed → spec → impl → validate)
- The site's own construction serves as credible evidence of the methodology's claims

---

## Constraints

- **No external API calls at runtime** — enforced by architecture, not discipline
- **No AI in the interactive features** — the wizard uses structured templates only
- **Deployed at a subpath** — all asset URLs and internal links must resolve correctly under `/demos/greenfield/`
- **Responsive at all breakpoints** — all interactive features must work on mobile with touch input
- **Keyboard accessible** — Hierarchy Explorer and Spec Wizard navigable without a mouse
