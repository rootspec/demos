# L2 — Truths

## Design Philosophy

**Declarative over imperative** — Describe what the product should be, not how to build it. Let the spec guide implementation decisions rather than prescribing exact technical approaches.

**Demonstration over explanation** — Show RootSpec working rather than describing how it works. Interactive features prove the methodology's value better than marketing copy.

**Transparency over perfection** — Acknowledge this site's limitations as a demo built from minimal guidance. Rough edges demonstrate realistic expectations for automated specification.

**Structure over flexibility** — Follow the five-level hierarchy strictly. Constraints enable creativity by providing clear boundaries.

## Trade-offs

We choose **methodology demonstration** over feature completeness — Interactive hierarchy explorer and spec wizard are more valuable than comprehensive documentation.

We choose **client-side simplicity** over backend sophistication — All interactive features run in the browser rather than requiring server infrastructure.

We choose **honest limitations** over polished marketing — Show what minimal human guidance produces rather than hide the tool's constraints.

We choose **educational depth** over surface appeal — Technical leads need to understand the methodology, not just feel impressed.

We choose **structural clarity** over visual complexity — Clean information hierarchy matters more than sophisticated animations or graphics.

We choose **keyboard accessibility** over mouse-only interactions — Professional developers expect full keyboard navigation.

We choose **system theme detection** over brand-specific styling — Respect user preferences rather than force a particular aesthetic.

## Success Criteria

**Primary Success** — Technical leads understand how RootSpec prevents specification drift and philosophy-implementation gaps. They can explain the five-level hierarchy after one site visit.

**Engagement Success** — Visitors spend time with interactive features (hierarchy explorer, spec wizard) rather than just reading text. Average session includes meaningful interaction with the methodology.

**Conversion Success** — Motivated visitors successfully navigate to the GitHub repo and getting started instructions. Clear path from understanding to action.

**Demonstration Success** — The site itself proves RootSpec's value by being built with RootSpec. Meta banner communicates this clearly without defensiveness.

**Educational Success** — Visitors can distinguish RootSpec from code generators, design systems, and traditional specification approaches. They understand it's about validation and intent extraction.

## Constraints

**No external APIs** — All functionality must work offline and client-side only. No backend dependencies or third-party service calls.

**Framework limitations** — Built with Astro for static site generation. Interactive features use minimal JavaScript with progressive enhancement.

**Content constraints** — Real examples in all sections, no lorem ipsum. Before/after comparisons use actual content that demonstrates the methodology.

**Accessibility requirements** — Must meet WCAG 2.1 AA standards. All interactive features accessible via keyboard navigation and screen readers.

**Performance budget** — Initial page load under 3 seconds on average mobile connection. Interactive features respond within 100ms.

**Maintenance simplicity** — RootSpec version and configuration managed through single config file. Easy updates without code changes.

**Responsive requirements** — All features fully functional on mobile devices. Touch-friendly interface design for mobile interactions.
