# **L2: Truths**

## Design Philosophy

**Demonstration Over Documentation** — The site teaches through experience rather than explanation. Interactive features let visitors feel the methodology rather than just read about it.

**Hierarchy Over Flexibility** — Strict level dependencies prevent architectural debt. This constraint enables confidence in long-term product evolution.

**Validation Over Generation** — Focus on ensuring AI-produced content serves human intent, not on automating human creativity away.

**Client-Side Over Server-Side** — All interactive features run locally to demonstrate that RootSpec methodology doesn't require complex infrastructure.

## Strategic Trade-offs

**Philosophy-First Over Feature-First** — We choose foundational clarity over rapid feature delivery. Better to have fewer features that clearly serve the mission than many features that dilute focus.

**Transparency Over Marketing Polish** — We choose honest acknowledgment of limitations over glossy presentation. The meta banner admits rough edges as evidence of minimal guidance.

**Structured Intent Over Creative Freedom** — We choose the constraint of five-level hierarchy over the flexibility of ad-hoc specification. Constraints enable rather than limit good design.

**Natural Language Over Configuration** — We choose human-readable specifications over YAML/JSON configuration files. Accessibility to non-technical stakeholders matters more than tooling convenience.

**Evidence Over Claims** — We choose showing the actual spec and seed files over describing theoretical benefits. Proof matters more than persuasion.

## Success Criteria

### Primary Success Metrics

**Comprehension Rate** — 80% of visitors can explain the five-level hierarchy after interacting with the hierarchy explorer.

**Conviction Rate** — 60% of visitors who complete the "Spec Your Idea" wizard report increased confidence in specification-driven development.

**Discovery Rate** — 90% of visitors find and view either the source spec or seed files through the meta banner links.

### Secondary Success Metrics

**Engagement Depth** — Average time on interactive features exceeds 3 minutes.

**Technical Clarity** — Developer-focused visitors can identify which level their current project problems occur at.

**Methodology Adoption** — Visitor conversion to running `/rs-init` in their own projects (tracked via GitHub repo analytics).

### Leading Indicators

**Bounce Rate** — Less than 40% of visitors leave without scrolling past hero section.

**Interactive Feature Usage** — More than 70% of visitors engage with at least one interactive element.

**Meta Banner Click-Through** — More than 30% of visitors follow links to view spec or seed files.

## Acceptance Constraints

### Performance Constraints

**Load Performance** — Initial page load under 2 seconds on 3G connections.

**Interaction Performance** — Interactive features respond within 100ms of user input.

**Animation Performance** — All transitions maintain 60fps on modern devices.

### Content Constraints

**Accuracy Constraint** — All technical claims about RootSpec methodology must be verifiable against the actual framework documentation.

**Currency Constraint** — RootSpec version displayed on site must match the framework version used to build the specification.

**Honesty Constraint** — The meta banner must accurately reflect the minimal human input used to create the site.

### Technical Constraints

**Accessibility Constraint** — All interactive features must work with keyboard navigation and screen readers.

**Compatibility Constraint** — Full functionality on browsers from the last 2 years.

**Independence Constraint** — No external API calls required for core functionality.

## Non-Negotiable Commitments

**Hierarchy Integrity** — The site structure and content must exemplify the RootSpec methodology it promotes. No exceptions for marketing convenience.

**Open Source Transparency** — Source code, specification files, and build process must be publicly viewable and clearly linked.

**Version Accountability** — The RootSpec framework version used must be prominently displayed and traceable.

**Demonstration Completeness** — Every major RootSpec concept mentioned must be demonstrated through the site's actual implementation, not just described.
