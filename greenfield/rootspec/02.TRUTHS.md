# Level 2: Truths

## Design Philosophy

**Declarative over imperative**: RootSpec extracts human intent rather than generating code templates. The specification describes what should exist, not how to build it.

**Validation over gatekeeping**: The framework identifies misalignment between philosophy and implementation but never blocks legitimate development work.

**Structure over flexibility**: Rigid hierarchy prevents circular dependencies and maintains philosophical consistency even as implementation evolves.

## Strategic Trade-offs

### We Choose Structured Methodology Over Flexible Documentation
- **Rationale**: Supports **Purposeful Constraints** pillar by preventing specification drift
- **Cost**: Learning curve for teams used to informal documentation
- **Benefit**: Specifications stay current and traceable to core principles

### We Choose Natural Language Over Configuration Files
- **Rationale**: Supports **Incremental Mastery** and **Philosophical Clarity** pillars
- **Cost**: Cannot auto-generate from existing YAML/JSON configs
- **Benefit**: Accessible to non-technical stakeholders and easier human review

### We Choose Client-Side Interactivity Over Server-Side Complexity
- **Rationale**: Supports **Effortless Traceability** and **Validation Confidence** pillars
- **Cost**: No personalization or data persistence across sessions
- **Benefit**: Fast, reliable demonstrations that work anywhere

### We Choose Methodology Demonstration Over Feature Completeness
- **Rationale**: This site showcases RootSpec principles in practice
- **Cost**: Some rough edges from minimal iteration on generated code
- **Benefit**: Honest representation of what the methodology produces
