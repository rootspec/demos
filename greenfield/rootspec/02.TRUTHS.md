# Level 2: Stable Truths

## Design Philosophy

### Truth: The spec is the source of truth, not a mirror of it
Most documentation describes code that already exists. RootSpec inverts this: the specification precedes and governs the code. Implementation is the process of making code match the spec, not making the spec match the code.

*References: L1 — Intent Is the Asset pillar*

### Truth: Hierarchy prevents drift
The five-level dependency hierarchy (L1 → L5, each level only referencing higher levels) is the core mechanism for keeping philosophy stable when implementation changes. Circular dependencies are architecturally impossible by design. This is not a style choice — it is an enforcement mechanism.

*References: L1 — Philosophy Before Implementation pillar*

### Truth: Validation is the whole job
Generating code is now trivial. Proof that the code satisfies intent is the hard, valuable part. The methodology's payoff comes at validation: user stories are the tests; test failure means spec-code drift; passing means the intent was delivered.

*References: L1 — Validation Is Everything pillar*

### Truth: Natural language specification over configuration-first
The spec is written in prose and structured markdown, not YAML-first config or formal notation. This keeps the barrier low and the spec legible to anyone on the team — including stakeholders who are not engineers.

*References: L1 — Approachable Without Being Simplified pillar*

## Architectural Commitments

### Commitment: Four commands as the complete workflow
All RootSpec interactions are expressed through four skills: `/rs-init`, `/rs-spec`, `/rs-impl`, `/rs-validate`. The methodology is learnable in a single sitting and memorable without documentation.

### Commitment: AI executes; humans supply philosophy
AI has knowledge but not wisdom. Humans supply the philosophy at L1-L2; AI executes implementation at L4-L5. The spec is the bridge — it carries human intent into the domain where AI operates, so AI output is constrained and verifiable.

### Commitment: Static site, client-side interactivity only
The marketing site requires no server, no API, no authentication. All interactive features run in the browser. This reduces operational complexity to zero and demonstrates that the methodology works for content-first products.

### Commitment: Light mode default, editorial aesthetic
The site defaults to light mode. The primary design elements are typography and whitespace — not gradients, glassmorphism, or decorative effects. The aesthetic signals "a place where someone thinks" rather than "a Series A startup landing page."

### Commitment: Base path consistency
The site deploys under `/demos/greenfield/`. This base path is the same in dev, preview, and production. No environment-based path switching.

## Trade-offs

- **We choose proof over speed** — Validation takes time; shipping without validation is faster but produces drift. We accept slower deployment cycles in exchange for verified intent delivery.
- **We choose hierarchy over flexibility** — Strict level dependencies mean some ideas must be refactored to fit the hierarchy. We accept this constraint because it prevents the circular dependencies that cause drift.
- **We choose prose over configuration** — Natural language is harder to parse programmatically but more legible to humans. We choose legibility; tooling adapts.
- **We choose editorial aesthetics over conversion-optimized landing page patterns** — A polished, considered design communicates credibility. Familiar SaaS landing page patterns communicate "AI-generated default." We choose the former even if it sacrifices some conversion-rate optimization.
- **We choose client-side-only interactivity** — No server, no API. This limits some features (no AI-powered wizard) but eliminates operational complexity and keeps the demo honest about what a static spec-driven build can produce.

## Definition of Success

- A developer who finds the site understands what RootSpec is and why it exists within the first scroll
- The meta-banner communicates immediately that the site itself is the demo — visitors understand they are seeing the methodology in action, not just reading about it
- The interactive features (Hierarchy Explorer, Spec Wizard) make the methodology tangible — visitors experience it, not just read about it
- The Author's Notes section carries the author's voice intact — no paraphrasing, no condensing
- The site passes its own spec validation: user stories are testable and pass against the live implementation

## Constraints We Accept

- No external API calls — all interactivity is client-side templating and state
- No images unless SVG — the RootSpec diagram must be SVG, not a raster asset
- No JavaScript framework lock-in at L1-L3 — Astro + React is an L4 decision, not a philosophical one
- Version number displayed prominently — read from `.rootspec.json` at build time, not hardcoded
- Author's Notes reproduced verbatim — the section exists to carry the author's voice, not to summarize it
