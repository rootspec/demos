## Interaction Loop Architecture

| Scale     | Scope                  | Duration   | Key Patterns                          |
|-----------|------------------------|------------|---------------------------------------|
| Immediate | Single post action     | Seconds    | Like toggle, bookmark toggle          |
| Session   | Browse → discover      | Minutes    | Feed scroll, search, explore, follow  |
| Visit     | Full demo evaluation   | 5-10 min   | Meta banner → spec links → interact   |
| Lifetime  | Return visitor         | Occasional | No persistent state; starts fresh     |

## Core User Journeys

### Journey 1: Home Feed Browse

**Trigger:** User lands on `/`

1. User sees the meta banner explaining the demo context
2. User sees a paginated feed of posts (10 at a time) with author avatars, handles, timestamps, and engagement counts
3. User clicks "Load more" to fetch the next page of posts
4. User clicks a post to navigate to post detail
5. User clicks an author handle to navigate to their profile
6. User likes or bookmarks a post — count updates immediately (client-side toggle)

**Feedback loop:** Like/bookmark button state changes immediately. Count increments/decrements. No loading state required — client-side only.

**Failure mode:** If post data is missing, show an empty state ("No posts yet") rather than an error.

---

### Journey 2: Post Detail & Thread

**Trigger:** User clicks a post anywhere in the app

1. User sees the full post with author info, content, and engagement counts
2. If the post is a reply, the parent post is shown above with a visual thread connector
3. Replies to this post are listed below
4. User can like the post from the detail view
5. User can click any author handle to navigate to their profile

**Failure mode:** If post ID doesn't exist, show "Post not found" with a link back to home.

---

### Journey 3: Profile View

**Trigger:** User clicks a handle anywhere in the app

1. User sees the profile: avatar, display name, handle, bio, follower/following counts
2. User sees a "Follow" or "Unfollow" button reflecting current follow state
3. User clicks Follow — button changes to Unfollow, follower count increments (client-side)
4. User sees all posts by this user, most recent first
5. User can click any post to go to post detail

**Failure mode:** If handle doesn't match any user, show "User not found."

---

### Journey 4: Search

**Trigger:** User navigates to `/search`

1. User sees a search input and empty results area
2. User types a query — results update as they type (no submit button required)
3. Results show matching posts with author attribution; minimum query length of [min_query_length] characters
4. User can filter results by tag using tag chips below the search input
5. User clicks a result to navigate to post detail

**Failure mode:** No results shows "No posts matching [query]" — not an error, just an empty state.

---

### Journey 5: Explore

**Trigger:** User navigates to `/explore`

1. User sees trending tags with post counts
2. User sees suggested users to follow (all users, sorted by follower count)
3. User sees popular posts (highest engagement)
4. User clicks a tag to filter posts by that tag (updates post list below the tag chips)
5. User clicks a user to navigate to their profile

**Active tag state:** The selected tag is visually highlighted. Clicking the same tag again deactivates the filter and shows all popular posts.

---

### Journey 6: Compose a Post

**Trigger:** User clicks the compose button (persistent in layout)

1. A text area appears (inline or modal — implementation decides)
2. User types post content — character count shown, maximum [max_post_length] characters
3. User submits — post appears at the top of the home feed with client-side state
4. New post persists in feed for the session but is not saved across reloads

**Failure mode:** Empty posts are rejected with an inline validation message. Posts over [max_post_length] characters cannot be submitted.

---

### Journey 7: Theme Toggle

**Trigger:** User clicks the theme toggle in the nav

1. System preference is detected on first load and applied immediately
2. User clicks toggle — theme switches between dark and light
3. Selection is stored in localStorage and persists across page reloads
4. All surfaces (nav, feed, cards, explore) respond to the active theme

---

## Interaction States

### Post Engagement States
- **Like:** Unliked (default) → Liked (toggled). Count reflects current state.
- **Bookmark:** Unbookmarked (default) → Bookmarked (toggled). Count does not increment (bookmarks are personal, not public).

### Follow States
- **Follow:** Not following (default) → Following (toggled). Follower count on profile reflects state.
- Following state persists for the session only.

### Theme States
- **Light:** Default if no system preference detected
- **Dark:** Applied when system preference is `prefers-color-scheme: dark` or user has toggled manually
- Manual toggle overrides system preference for the session; localStorage persists the override

---

## Navigation Model

- Top navigation bar is persistent across all routes
- Logo links to home feed
- Nav items: Home, Explore, Search
- Compose button accessible from navigation
- All navigation is client-side routing (SvelteKit); no full page reloads

---

## Meta Banner Interaction

The banner is persistent and non-dismissible. It contains:
- Explanation of the demo context (scaffold → RootSpec → this app)
- Link to original scaffold commit on GitHub
- Link to spec files (rootspec/ directory)
- Link to SEED.md

Banner lives below the nav, above the main content on every page.
