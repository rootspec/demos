## Interaction Loop Architecture

| Scale     | Scope                       | Duration     | Key Patterns                          |
|-----------|-----------------------------|--------------|---------------------------------------|
| Immediate | Single action               | Seconds      | Toggle like, follow, compose, search  |
| Session   | Exploring multiple surfaces | Minutes      | Browse feed → profile → post → search |
| Extended  | N/A (no persistence)        | —            | State resets on each page load        |
| Lifetime  | Demo evaluation             | One visit    | Arrive → explore → understand → link  |

---

## Core Interaction Loops

### Loop 1: Feed Browsing

**Trigger:** User visits `/` (Home Feed)

**Flow:**
1. User sees a reverse-chronological list of posts with author avatar, handle, display name, content, timestamp, and engagement counts
2. User scrolls down; a "Load more" control reveals the next batch of posts
3. User clicks an author handle → navigates to `/profile/[handle]`
4. User clicks a post → navigates to `/post/[id]`
5. User clicks the Like button on a post → like count increments, button state toggles
6. User clicks the Bookmark button on a post → bookmark state toggles
7. User clicks the Compose button → post composer opens; user writes and submits → new post appears at top of feed

**Feedback:**
- Like/bookmark buttons update immediately with no loading state (client-side toggle)
- New composed posts appear at top of the list without page reload
- Pagination is explicit ("Load more") not infinite scroll

---

### Loop 2: Profile Exploration

**Trigger:** User navigates to `/profile/[handle]`

**Flow:**
1. User sees display name, handle, bio, follower/following counts
2. User sees Follow/Unfollow button — toggling updates the label and adjusts the follower count display
3. User sees all posts by this user in reverse-chronological order
4. User can navigate to any post from the profile
5. User can navigate back to the home feed via the nav

**Feedback:**
- Follow/Unfollow toggles immediately; count display updates to reflect the change
- State is client-side only — refreshing the page resets the follow state

---

### Loop 3: Post Thread View

**Trigger:** User navigates to `/post/[id]`

**Flow:**
1. User sees the full post content, author info, and engagement counts
2. If the post is a reply, the parent post is displayed above it
3. All replies to this post are listed below it
4. User can like/bookmark the focal post and any replies
5. User can navigate to any author's profile from the thread
6. User can navigate to any reply's full thread view

**Feedback:**
- Like/bookmark actions on any post in the thread update immediately
- Thread context (parent + replies) is fully visible without additional navigation

---

### Loop 4: Search

**Trigger:** User visits `/search`

**Flow:**
1. User types into the search input
2. Results filter in real time as each character is entered — no submit button required
3. Results show author name, handle, and post content matching the query
4. Clicking a result navigates to `/post/[id]`
5. If the query returns no results, a "No results for [query]" message is shown
6. Clearing the input returns to the empty search state (no results shown)

**Feedback:**
- Results update synchronously with no perceptible delay (in-memory filtering)
- Empty state message is friendly and shows the current query so the user knows what failed

---

### Loop 5: Explore / Discovery

**Trigger:** User visits `/explore`

**Flow:**
1. User sees trending tags displayed as clickable chips
2. User clicks a tag → post list filters to show only posts with that tag
3. User sees suggested users to follow with name, handle, bio, and a Follow button
4. User sees popular posts ordered by engagement
5. User can navigate to any profile or post from the explore page
6. Clicking a tag again or clicking a different tag updates the filter; active tag is visually indicated

**Feedback:**
- Tag filtering updates the visible post list immediately
- Active tag chip is styled distinctly from inactive tags
- Follow buttons on the explore page reflect current follow state (same client-side state as profile pages)

---

### Loop 6: Theme Toggle

**Trigger:** User clicks the theme toggle in the header (sun/moon icon)

**Flow:**
1. On first load, the theme reflects the user's OS preference (dark or light)
2. User clicks the toggle → theme switches immediately
3. Theme preference persists for the browser session (client-side state)
4. Refreshing the page re-reads OS preference unless session state overrides

**Feedback:**
- Theme switches immediately with no flicker or delay
- Toggle icon changes to reflect the active mode (sun for light, moon for dark)

## Edge Cases and Failure Modes

### Static Route Boundaries
Only profiles and posts that exist in the mock data produce rendered pages. Navigating to an unknown handle or post ID produces a static 404 from the hosting layer (GitHub Pages). This is not modeled as an application state — it is a hosting constraint acknowledged in the truths.

### Empty Search State
When the search input is empty, no results are shown. This is the correct starting state — not an error.

### No Persistence on Reload
All client-side state (likes, bookmarks, follows, composed posts) resets on page reload. The meta banner explains this expectation. It is not a bug.

### Composed Posts are Ephemeral
Posts created via the composer appear in the feed for the current session only. They are not added to the JSON data and do not persist. The composer acknowledges this behavior.

## Navigation Architecture

| Route                  | Entry Points                              | Exit Points                              |
|------------------------|-------------------------------------------|------------------------------------------|
| `/` — Home Feed        | Nav link, direct URL                      | Profile, post detail, search, explore    |
| `/profile/[handle]`    | Author handle click (any page), Nav       | Post detail, home feed                   |
| `/post/[id]`           | Post click (any page)                     | Author profile, parent/reply threads     |
| `/search`              | Nav link, direct URL                      | Post detail, author profile              |
| `/explore`             | Nav link, direct URL                      | Post detail, author profile              |

Navigation is flat — no routes are nested deeper than one level. The nav bar is always visible and always shows the five primary destinations.

## Meta Banner Interaction

The meta banner is persistent across all pages. It contains:
- A brief description of what RootFeed is and how it was built
- A link to the scaffold commit on GitHub (absolute URL)
- A link to the spec files on GitHub (absolute URL)
- A link to SEED.md on GitHub (absolute URL)

The banner is not dismissable — it is a permanent fixture of the demo experience. It supports the Transparent Construction pillar.
