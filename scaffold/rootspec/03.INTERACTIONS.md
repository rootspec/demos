# Level 3: Interaction Architecture

**RootFeed — How Users and the Product Interact**

---

## Interaction Loop Architecture

| Scale | Scope | Duration | Key Patterns |
|-------|-------|----------|--------------|
| Immediate | Single action | Seconds | Like toggle, bookmark toggle, follow toggle, search filtering |
| Session | Connected navigation | Minutes | Feed browsing → profile → post detail → back |
| Extended | Returning visitor | Page loads | Theme preference persisted; all other state resets |
| Lifetime | Developer evaluation | One visit | See demo → understand scaffold → evaluate RootSpec |

The dominant interaction scale for this product is **session**: a developer opens the app, navigates through several routes, understands what was built, and makes an evaluation. The lifetime value is a single conversion event — forming an opinion about RootSpec.

---

## Core Interaction Loops

### Loop 1: Feed Browsing

**Trigger:** Visitor arrives at `/`
**Flow:**
1. Meta banner is visible at the top — origin and links are immediately available
2. Composer is visible — user can write a post (client-side only)
3. Feed renders posts in reverse-chronological order (newest first)
4. Each post shows: author avatar + name + handle, content, timestamp, like count, repost count, like button, bookmark button
5. User can click a post to go to its detail page
6. User can click an author name/handle to go to their profile
7. User can like or bookmark a post — count updates immediately, state is local to that card
8. Composed post appears at the top of the feed immediately after submission

**Feedback:** Immediate visual toggle on like/bookmark buttons; no loading states (all data is local)

**Failure mode:** If a post has no matching author in the data, author information is omitted gracefully — the post still renders

---

### Loop 2: Post Detail & Thread

**Trigger:** User clicks on a post from the feed or profile
**Flow:**
1. Single post is displayed prominently with full content
2. If the post has a parent (is a reply), the parent post is shown above with a visual connector
3. Replies to this post are listed below in chronological order
4. Author links navigate to profile; engagement counts displayed
5. No new interactions beyond reading — compose/reply is not available on this page

**Feedback:** Static presentation; no loading states

**Failure mode:** If post ID is not found, a "Post not found" message is shown

---

### Loop 3: Profile Viewing

**Trigger:** User clicks an author name or navigates to `/profile/[handle]`
**Flow:**
1. User's avatar, display name, handle, bio, follower count, following count are displayed
2. A follow/unfollow button toggles the follow state client-side
3. Follow state is reflected in the button label immediately
4. The user's posts are listed below the profile header
5. Clicking a post navigates to its detail page

**Feedback:** Follow button label changes immediately ("Follow" → "Unfollow"); no server calls, no count update

**Failure mode:** If handle is not found, a "User not found" message is shown

---

### Loop 4: Search

**Trigger:** User navigates to `/search`
**Flow:**
1. An empty search input is focused
2. User types a query
3. Results filter in real-time (derived, no debounce — data is local)
4. Results show: author name, post content, link to post detail
5. If query has content but no results, an empty state message is shown
6. Clearing the input clears results

**Feedback:** Immediate result update on every keystroke; no spinner or delay needed

**Failure mode:** Empty query shows no results (not all posts) — this is intentional

---

### Loop 5: Explore & Discovery

**Trigger:** User navigates to `/explore`
**Flow:**
1. Trending tags are displayed as clickable chips, sorted by post count descending
2. Clicking a tag filters the post list to posts tagged with that tag
3. Active tag is visually highlighted
4. A list of suggested users is displayed below the tags
5. Each user shows display name, handle, bio, and a follow/unfollow toggle

**Feedback:** Tag selection filters the list immediately; follow button toggles immediately

**Failure mode:** If no posts match a selected tag, an empty state is shown

---

### Loop 6: Post Composition

**Trigger:** User interacts with the composer at the top of the home feed
**Flow:**
1. Textarea accepts up to [max_post_length] characters
2. Character count is displayed below the textarea
3. Submitting an empty or whitespace-only post shows an inline validation error
4. Valid submission adds the post to the top of the feed immediately
5. Textarea clears after submission; error clears

**Feedback:** Inline error for validation; post appears instantly in feed on success

**Failure mode:** Empty post is blocked with an error message; no post is created

---

### Loop 7: Theme Toggle

**Trigger:** User clicks the theme toggle in the navigation header
**Flow:**
1. Theme switches between light and dark immediately
2. Preference is persisted to localStorage
3. On next page load, the stored preference is restored
4. System preference is respected as the default if no stored preference exists

**Feedback:** Immediate visual change; no reload required

---

## Navigation Architecture

**Persistent nav:** All pages share the same nav bar with: RootFeed logo/home link, Home, Explore, Search links, RootSpec version badge, theme toggle.

**Meta banner:** Appears at the top of every page, below the nav. Links to seed, spec, and scaffold commit.

**Footer:** Appears at the bottom of every page. Contains: RootSpec version, build date, builder identity, link to rootspec/rootspec repo.

**No breadcrumbs, no back buttons** — browser navigation is sufficient for this app's depth.

## Edge Cases & Failure Modes

| Scenario | Behavior |
|----------|----------|
| Post author not found in users.json | Post renders without author details; no crash |
| Profile handle not found | "User not found" message; no crash |
| Post ID not found | "Post not found" message; no crash |
| Search with no results | "No results for [query]" message |
| Composer submitted empty | Inline validation error; post not created |
| Tag clicked with no matching posts | Empty state shown below tag section |
| Dark mode on first visit | System preference applied; no flash |
| Dark mode on return visit | localStorage preference applied |

---

## Verification Criteria

An implementation of this spec is correct if:

1. Every page shows the meta banner with working links
2. Every page shows the footer with attribution
3. The nav shows the RootSpec version on every page
4. The theme toggle works and persists across page loads
5. Feed posts show author info, content, timestamps, and engagement counts
6. Like and bookmark buttons toggle visually and update counts immediately
7. Composing a post adds it to the top of the feed
8. Profile pages show follow/unfollow toggle that responds immediately
9. Search results filter in real-time as the user types
10. Explore page tag chips filter the post list on click
11. Post detail shows parent post (if reply) and all replies
12. All navigation links work and route to the correct pages
