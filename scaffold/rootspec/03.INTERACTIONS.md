## Core User Journeys

### Journey 1: Developer Arrival and Orientation

A developer lands on the home feed (`/`). They see:
1. The meta banner at the top — "RootFeed started as a bare SvelteKit scaffold..."
2. The navigation bar with RootFeed branding, nav links, theme toggle, and RootSpec version
3. The composer for writing a post
4. A feed of posts with authors, timestamps, like/bookmark controls, and engagement counts

Within [short duration] they understand what this is and how it was built.

### Journey 2: Feed Browsing

The user scrolls the home feed. Posts render with avatars, display names, handles, relative timestamps, and engagement actions. Clicking a post navigates to the post detail page. Clicking an author name navigates to their profile. Like and bookmark buttons toggle immediately with visual feedback. A "Load more" control loads the next batch of posts.

### Journey 3: Post Composition

The user types in the composer textarea. A character counter reflects [max_post_length] characters. Submitting an empty post shows a validation error. Submitting a valid post prepends it to the feed instantly and clears the composer.

### Journey 4: Post Detail and Thread

Navigating to `/post/[id]` shows the full post. If the post is a reply, the parent post is shown above it. All replies to the post are listed below. Engagement actions work the same as on the feed.

### Journey 5: Profile Browsing and Follow

Navigating to `/profile/[handle]` shows the user's display name, handle, bio, follower and following counts, and all their posts. A follow/unfollow button toggles client-side follow state. The button label and style reflect the current state. Following/unfollowing updates the follow state immediately.

### Journey 6: Search

Navigating to `/search` shows a text input. As the user types, results update in real time — filtering posts by keyword content. If the query matches nothing, an empty state message appears. Posts in results link to their detail pages; author names link to profiles.

Tag filtering: selecting a tag from the search or explore pages filters results to posts tagged with that tag.

### Journey 7: Explore Discovery

Navigating to `/explore` shows trending tags, suggested users, and popular posts. Clicking a tag filters the explore feed to posts with that tag. Clicking a user navigates to their profile. Clicking a post navigates to the post detail.

## Interaction Patterns

### Toggling (like, bookmark, follow)

Toggle interactions change immediately on click — no loading state, no delay. Visual state updates: filled icon for active state, outlined for inactive. Count increments or decrements accordingly. These states do not persist across page loads.

### Real-time Search

Search filters update on every keystroke. No debounce delay required given the small mock dataset. Results appear below the input; clearing the input returns to the empty state.

### Tag Filtering

Tags are clickable throughout the explore and search surfaces. Selecting a tag applies a filter; selecting the same tag again removes it. Only one tag filter is active at a time.

### Load More

The home feed loads [initial_posts_count] posts initially. A "Load more" button appends the next [load_more_count] posts. When all posts are loaded, the button disappears.

### Post Composition

The composer is always visible at the top of the home feed. It shows a textarea, a character counter, and a submit button. Submitting adds the post to the top of the feed. The composer is not available on other pages.

### Theme Toggle

A toggle control in the navigation bar switches between light and dark mode. The selected theme is stored in localStorage so it persists across page loads. On first load, system preference is detected.

### Meta Banner

The meta banner is persistent on every page. It contains a short explanation of the demo and links to the seed, spec, and scaffold commit. It cannot be dismissed.

## Failure States and Edge Cases

- **Post not found** (`/post/[id]` with unknown id): Display a "Post not found" message.
- **Profile not found** (`/profile/[handle]` with unknown handle): Display a "User not found" message.
- **Empty composer submission**: Show inline validation error "Post cannot be empty."
- **Post at character limit**: Disable or warn at [max_post_length] characters.
- **Search with no results**: Show a "No results for [query]" message.
- **Explore with no posts for tag**: Show empty state for the filtered view.

## Navigation Model

The top navigation bar contains: RootFeed logo (links home), Home, Explore, Search links, theme toggle, and RootSpec version badge. Navigation is always visible. The current route is not highlighted (scaffold baseline — enhancement deferred).

Footer on every page: RootSpec version, build date, "Built by RootSpec" attribution, link to the rootspec/rootspec GitHub repository.
