## Interaction Loop Architecture

| Scale     | Scope                        | Duration    | Key Systems                          |
|-----------|------------------------------|-------------|--------------------------------------|
| Immediate | Single action                | Seconds     | Feed state toggle, search filter     |
| Session   | Browsing + composing         | Minutes     | Navigation, composer, explore        |
| Extended  | Across page visits           | N/A (static)| Theme preference (localStorage only) |
| Lifetime  | Demo evaluation              | One visit   | Meta banner, attribution, spec links |

## Core Interaction Loops

### Loop 1: Feed Browsing
**Trigger:** User lands on `/` or navigates to Home.
**Pattern:**
1. User sees paginated list of posts ([N] per page)
2. Each post shows avatar, display name, handle, content, timestamp, like count, repost count
3. User scrolls and optionally clicks "Load more" to paginate
4. User clicks a post to navigate to `/post/[id]`
5. User clicks an author to navigate to `/profile/[handle]`

**Feedback:** Post list expands on load more; navigation happens immediately (static pages).

### Loop 2: Post Engagement (Like / Bookmark)
**Trigger:** User clicks the like or bookmark button on any post.
**Pattern:**
1. Button shows toggled state immediately (filled vs. outline icon)
2. Like count increments/decrements to reflect toggle
3. State is stored in `FeedState` store — persists within session, resets on reload

**Feedback:** Visual state change is instantaneous. No loading states needed.

### Loop 3: Post Composition
**Trigger:** User opens the post composer (visible on home feed).
**Pattern:**
1. User types in composer text area (character limit enforced)
2. User submits the post
3. New post appears at the top of the feed
4. Composer clears and closes/collapses

**Feedback:** Post appears immediately. Composer shows character count. Submit disabled when empty or over limit.

### Loop 4: User Follow/Unfollow
**Trigger:** User visits a profile page and clicks Follow or Unfollow.
**Pattern:**
1. Button label reflects current follow state ("Follow" or "Unfollow")
2. Click toggles the state immediately
3. Follower count updates to reflect the change
4. State stored in `ProfileState` store

**Feedback:** Button label and style change on click. Count updates.

### Loop 5: Search
**Trigger:** User navigates to `/search` and types in the search input.
**Pattern:**
1. Results update in real time as user types (derived state, no debounce needed at mock scale)
2. Results show author name and post content
3. Empty state shown when query has no matches
4. No results shown for empty query

**Feedback:** Results list re-renders on every keystroke. Empty state appears immediately for no matches.

### Loop 6: Tag Filtering (Explore)
**Trigger:** User clicks a tag on the `/explore` page.
**Pattern:**
1. Active tag is highlighted
2. Post list filters to show only posts with that tag
3. Clicking the active tag again deselects it, showing all posts
4. Clicking a different tag switches the filter

**Feedback:** Posts list updates immediately. Tag selection is visually distinguished.

### Loop 7: Theme Toggle
**Trigger:** User clicks the theme toggle button in the navigation.
**Pattern:**
1. Theme switches between light and dark
2. All visual elements update immediately (CSS class on `<html>`)
3. Preference is stored in localStorage
4. On next visit, the stored preference is applied before render

**Feedback:** Instant visual transition. No flash of wrong theme on reload.

### Loop 8: Explore Discovery
**Trigger:** User navigates to `/explore`.
**Pattern:**
1. User sees trending tags with post counts
2. User sees suggested users with bio previews
3. User sees popular posts
4. User can click tags to filter, click users to navigate to profiles, click posts to see detail

**Feedback:** Tag filtering is local to the explore page. Navigation to profiles/posts follows standard routing.

## Navigation Architecture

All navigation is SvelteKit link-based (prerendered static pages). Navigation patterns:

- **Feed → Post detail:** Click post card or post content
- **Feed → Profile:** Click author name or avatar
- **Profile → Post:** Click post in profile timeline
- **Explore → Profile:** Click user name
- **Explore → Post:** Click post card
- **Search → Post:** Click result item
- **Search → Profile:** Click author name in result

## Failure Modes and Edge Cases

### Empty search query
**Behavior:** Show no results, no empty-state message. The input is the only element shown.

### Search with no matches
**Behavior:** Show "No results for '[query]'" message below the input.

### Like/bookmark on a post that was composed in-session
**Behavior:** Composed posts support likes and bookmarks the same as loaded posts. All posts are treated uniformly by the feed store.

### Theme flash prevention
**Behavior:** Theme class must be applied synchronously before any paint. The `initTheme()` function is called in `+layout.svelte` on mount to minimize flash.

### Pagination boundary
**Behavior:** When all posts are visible, "Load more" button is hidden. No error state needed.

### Unknown handles/IDs in URLs
**Behavior:** These routes do not exist — this is a static site. Only prerendered routes resolve. Unknown paths are handled by the hosting platform (GitHub Pages 404).
