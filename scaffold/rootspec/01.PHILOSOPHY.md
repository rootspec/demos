# Level 1: Foundational Philosophy

**Product:** RootFeed
**Tagline:** A RootSpec scaffold demo

---

## Competitors

- **Bluesky** — Decentralized social feed with AT Protocol; the primary design inspiration
- **Twitter/X** — Centralized microblogging; established the conventions this demo riffs on
- **Mastodon** — Federated social network; more complex UX than needed for demo purposes
- **Threads** — Meta's Bluesky competitor; polished but opaque about its own construction

## Table Stakes

- Scrollable feed of posts with author attribution
- Post detail view with replies/threading
- User profiles with bio and follower counts
- Search or discovery surface to find content
- Basic navigation between sections

## Pain Points

- **Black box demos** — Most "AI-built" showcases hide the spec and process; visitors can't learn from them
- **Lorem ipsum theater** — Demo content feels hollow; fake data signals low investment in the demo
- **No transparency about origin** — Apps that pretend to be real products when they're prototypes create false expectations
- **Scaffold mystery** — Developers wonder "where did this start?" and can't see the delta between scaffold and finished product

## Influences

- **Bluesky's feed design** — Clean, content-forward, minimal chrome
- **Readable codebases** — The spec-first approach of RootSpec itself; specification as documentation
- **Build-in-public culture** — Show your work, including rough edges and process artifacts
- **Museum exhibit design** — Good exhibits show you what you're looking at and why it's interesting

---

## Mission

RootFeed exists to make RootSpec's scaffold workflow legible to developers evaluating the methodology. It is a real app that demonstrates a real process — not a mockup, not a tutorial, not a polished product. Every visible rough edge is a data point.

---

## Design Pillars

### 1. Transparent by Construction
The demo never pretends to be more than it is. The scaffold origin, spec files, and build process are surfaced prominently — not hidden in a README.

**User perspective:** "I can see exactly how this was built and what RootSpec actually produced."

**Examples:**
- Persistent meta banner explaining the demo context
- Links to SEED.md, spec files, and original scaffold commit
- Footer showing RootSpec version, build date, and builder identity

### 2. Alive Enough to Believe
Mock data and interactions feel realistic enough that the social feed experience is credible — tech conversations, replies, threads — without pretending to be a real network.

**User perspective:** "This feels like a real app, but I know it's a demo. That combination is what makes it useful."

**Examples:**
- 8 distinct personas with plausible bios and handles
- 30+ posts covering real dev topics: architecture debates, security tips, indie hacking
- Interactive like/bookmark/follow with client-side state that persists during a session

### 3. Spec as First-Class Artifact
The specification isn't a side document — it's part of the experience. Viewing the spec alongside the app is the intended use.

**User perspective:** "I can read the spec and immediately see how it maps to what's on screen."

**Examples:**
- RootSpec version displayed in the header
- Every interactive feature traces back to a user story
- Meta banner links directly to the spec source

---

## Inviolable Principles

- **Never pretend to be a real social network** — every page must feel like a demo that happens to be a social feed
- **Always surface the scaffold origin** — the meta banner must appear on every page
- **Never obscure the RootSpec connection** — the name "RootFeed" and tagline "A RootSpec scaffold demo" must be present throughout
- **No server-side persistence** — all interactive state (likes, follows, new posts) is client-side only; this is a demo, not a product

---

## North-Star Experience

Six months from now, a developer evaluating RootSpec opens RootFeed, reads the meta banner, clicks through to the spec, and says: "I can see exactly what RootSpec produced from this scaffold. The feed works, the spec makes sense, and I understand the delta between where it started and where it ended up. I want to try this on my own project."
