# Level 1: Foundational Philosophy

**Product:** RootFeed
**Tagline:** A RootSpec scaffold demo — from skeleton to experience.

---

## Competitors

- **Bluesky** — Decentralized social feed with strong open-protocol ethos; complex onboarding
- **Mastodon** — Federated microblogging; fragmented UX, high barrier to self-hosting
- **Twitter/X** — Algorithmic feed, ad-heavy, trust issues with editorial decisions
- **Threads** — Instagram-integrated microblogging; locked into Meta ecosystem

## Table Stakes

- Scrollable timeline of posts with author, content, and engagement counts
- User profiles with posts and follower/following stats
- Post detail view showing replies/threads
- Search or discovery surface for finding content
- Navigation between sections without full page reloads

## Pain Points

- **Production theater** — Most social demos pretend to be real products, obscuring what was built vs. what was given
- **Opaque scaffolding** — Developers can't tell where the framework ended and the spec began
- **Blank-slate anxiety** — Starting with empty routes and no guidance on how to fill them feels paralyzing
- **Unmaintainable demos** — Demo code is usually throwaway; this demo is meant to be readable and spec-driven

## Influences

- **Bluesky's clean feed aesthetic** — Dense but readable timeline
- **GitHub's demo repositories** — Code that teaches by being well-structured
- **The SvelteKit scaffold pattern** — Start with routing, fill with purpose

---

## Mission

Demonstrate that a bare SvelteKit scaffold — five empty routes and some JSON files — can become a fully interactive, spec-driven social feed through RootSpec guidance alone, with minimal human intervention.

---

## Design Pillars

### 1. Transparency Over Illusion
Users always know what was built, what was specified, and what the AI was given to start with.

**User perspective:** "I can see exactly what was built, what was specified, and what was given to the AI."

**Examples:**
- Persistent meta banner explaining origin and linking to artifacts
- Footer with RootSpec version, build date, and repo links

### 2. Alive Enough to Believe
Interactions and content feel genuinely real — but always framed as a purposeful demo, never a live product.

**User perspective:** "These posts feel like something a real person might write. The interactions respond like a real app."

**Examples:**
- Realistic-feeling usernames and tech conversation content
- Like, bookmark, and follow state that toggles visibly
- Search that responds as you type

### 3. Scaffold Visibility
The seam between scaffold and spec is visible and celebrated — the demo earns credibility by showing its starting point.

**User perspective:** "I can see the scaffold commit and understand what the AI was actually given to work with."

**Examples:**
- Meta banner links to the scaffold commit in the GitHub repo
- Spec files accessible from the footer
- The SEED.md document linked prominently

---

## Inviolable Principles

- Never present RootFeed as a real social network — every page must feel like a demo
- Never hide the scaffold origin — link to artifacts on every page
- Never sacrifice readability of the spec or code for visual polish
- Static-first: no server-side rendering or API calls — all data from mock JSON, all state client-side only

---

## North-Star Experience

A developer evaluating RootSpec visits the demo site, reads the meta banner, and immediately understands: "This was a bare scaffold. RootSpec turned it into this — here's the spec that drove it, here's the seed that started it, here's the commit that shows the before." They explore the feed, click around, toggle the theme, search posts — and the whole time it feels like a real app that openly admits it's a demo. They leave thinking: "If RootSpec can do this from an empty scaffold, I want to know what it can do with my codebase."
