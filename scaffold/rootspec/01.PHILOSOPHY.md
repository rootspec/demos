# Level 1: Foundational Philosophy

## Competitors

- **Vibe-coded demos** — AI-generated apps built through iterative chat prompting; untested, unverified, drift from intent over time
- **Tutorial clones** — Step-by-step Twitter/Instagram clones that teach a framework but feel lifeless and generic
- **AI app generators** — Tools like v0, bolt.new that produce UI fast but without a specification layer connecting intent to output

## Table Stakes

- Social feed that feels alive — real-ish content, threading, profiles, not lorem ipsum
- Working interactions — likes, follows, compose, search — that respond immediately
- Responsive layout that looks intentional on mobile and desktop
- Obvious demo framing — never pretends to be a real social network

## Pain Points

- **Intent drift** — Vibe coding produces something *close* to what was intended, never exactly it; each iteration introduces new divergence
- **No verification** — AI-generated apps have no way to prove they match the original vision
- **Chat as spec** — Conversation history is a terrible specification medium; context degrades, decisions get lost
- **Shallow demos** — Most showcases look good on the surface but fall apart under inspection — no tests, no coherent architecture

## Influences

- **Bluesky** — Clean, chronological social feed with a developer-friendly ethos
- **Storybook** — The idea of a living showcase that demonstrates components in context
- **RootSpec greenfield demo** — Companion project showing the from-scratch workflow

## Mission

Close the intent-to-product gap. Demonstrate that a short product description, run through structured specification, produces a verified, tested social feed — not an approximation, but a faithful translation of intent.

## Design Pillars

### 1. Earned Credibility
Developers trust what they can verify. Every feature traces back to a spec, every spec traces back to the seed.

**User perspective:** "This wasn't hand-tweaked — the spec actually produced this."

### 2. Effortless Curiosity
The demo invites exploration without requiring commitment. Developers poke around, see how things connect, and learn by discovery.

**User perspective:** "I keep clicking around because I want to see how deep it goes."

### 3. Approachable Confidence
The experience makes RootSpec feel achievable, not intimidating. If this came from a seed file, I can do this too.

**User perspective:** "This looks like something I could set up for my own project this afternoon."

## Inviolable Principles

- Never disguise the demo as a real product — transparency is the point
- Every interactive feature must be spec-traced and testable — no decorative functionality
- The seed-to-product story must be visible and verifiable at all times

## North Star

A developer lands on RootFeed, scrolls through a social feed that feels surprisingly complete, notices the meta banner explaining it came from a seed file and a spec, clicks through to the spec, sees the tests passing, and thinks: "I should try this for my project." The whole experience takes five minutes and leaves no doubt that the output was generated, not hand-crafted.
