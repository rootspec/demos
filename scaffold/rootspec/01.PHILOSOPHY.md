# Level 1: Foundational Philosophy

**Product:** RootFeed
**Tagline:** A RootSpec scaffold demo
**Version:** 7.2.7
**Last Updated:** 2026-04-14

---

## Problem Statement

Developers evaluating RootSpec need to see how it works with an existing codebase — not just greenfield projects. The question is: "I already have routing, a framework, and some JSON. Where does RootSpec fit in?" RootFeed answers that question by showing exactly where structure, tested interactions, and specification methodology enter an existing scaffold.

---

## Target Audience

Developers and technical teams who:
- Have an existing SvelteKit (or similar) project and want to see how RootSpec layers on top
- Are evaluating spec-driven development for real projects
- Want to see the contrast between "bare scaffold" and "fully implemented with guidance"

---

## Design Pillars

These are emotional/experiential qualities the product must embody, not features:

### Transparent
The app never pretends to be a real social network.

### Alive
Mock data and interactions feel real-ish — genuine usernames, plausible tech conversations, responsive UI.

### Playfully Technical
The tone is developer-audience-aware; humor lives in content, not UI chrome.

### Methodologically Honest
The meta-narrative (this is a RootSpec demo) is foregrounded, not hidden.

---

## Inviolable Principles

These will not be compromised under any circumstances:

- **No pretense:** Every page must reinforce that this is a demo, not a live product.
- **Absolute GitHub URLs only:** All external links (SEED.md, spec files) must use absolute GitHub URLs — relative links break the static prerender.
- **Static-only behavior:** Dynamic route parameters are prerendered from mock data. There is no server; do not spec or build "not found" handling for unknown parameters.
- **Attribution always visible:** RootSpec version, build date, and builder identity appear in every page footer.

---

## North Star

Six months from now, a developer evaluating RootSpec opens RootFeed and within 30 seconds understands: "This started as five empty routes and some JSON. RootSpec defined what it should be and built it from there. The spec is right there in the repo." They click through the feed, search for a post, explore trending tags, and finish with a clear picture of how RootSpec works on a real project — not a toy, not a tutorial, but an honest scaffold demo.

---

## Table Stakes

Features users must see to take this demo seriously:
- Working feed with real-ish posts and authors
- Navigable profiles and post detail pages
- Search that works in real-time
- Explore with trending tags and people
- Visible meta-banner linking to the scaffold commit, spec, and seed
- Dark/light theme toggle
- Interactive likes, bookmarks, follow/unfollow (client-side)
