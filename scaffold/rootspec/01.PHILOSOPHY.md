# Level 1: Foundational Philosophy

**Product:** RootFeed
**Tagline:** A RootSpec scaffold demo

---

## Mission

RootFeed exists to show developers how RootSpec transforms a bare SvelteKit scaffold into a complete, tested, living product. It is not a social network — it is a demonstration that happens to look like one. Every page should feel like a well-made prototype, inviting developers to see how specification-driven development works with an existing codebase.

---

## Market Context

### Similar Products

- **Bluesky** — Decentralized social feed; the aesthetic inspiration for RootFeed's timeline and profile patterns
- **Twitter/X** — The canonical microblogging reference; over-engineered for what developers need to see
- **Mastodon** — Federated, complex to self-host; opaque about how the frontend is structured
- **Threads** — Mobile-first; doesn't demonstrate web-native SvelteKit patterns well

### What They Get Wrong (for this purpose)

These are real products with real complexity — authentication, persistence, moderation, federation. That complexity obscures the scaffold story. RootFeed strips all of that away: no login, no database, no server. Just a SvelteKit skeleton + RootSpec + a weekend of implementation.

### Table Stakes

For developers evaluating RootSpec via this demo, the minimum credible experience includes:
- A timeline that looks and feels like a real feed
- Profiles with posts, bios, and social counts
- Post detail with thread context (replies)
- Search that works as expected (instant, keyword)
- An explore/discovery surface with tags and suggestions
- Interactive elements that respond: likes, follows, post composer
- Dark/light mode toggle
- A prominent banner explaining this is a demo, not a product

### Influences

- Bluesky's clean, chronological feed aesthetic
- Linear's precision and restraint in UI details
- The "working prototype" energy of Vercel's and Netlify's own demo sites

---

## Design Pillars

### Transparent Delight
The product wears its demo nature openly, yet every interaction still feels polished and intentional — visitors feel curious, not confused.

### Grounded Immediacy
Everything responds instantly — likes toggle, searches filter, posts appear — all without network round-trips, so the experience feels alive even though the data is static.

### Readable Craftsmanship
The code, the spec, and the UI are all legible artifacts — a developer can read any layer and understand exactly what was intended.

### Scaffold Honesty
The app never pretends to be more than it is — the meta-layer (banner, footer, version) is a first-class feature, not an afterthought.

---

## Inviolable Principles

1. **Never pretend to be a real social network.** Every page must contain at least one signal that this is a RootSpec demo (banner, footer, branding).
2. **No server-side state.** All interactivity (likes, follows, post composition) is client-side only, reset on page reload. Do not imply persistence.
3. **Spec drives implementation.** Every UI feature must trace back to a design pillar. If it doesn't support a pillar, it doesn't belong.
4. **Only known data is rendered.** Dynamic routes (`/profile/[handle]`, `/post/[id]`) are prerendered from mock data only. No 404 handling is required or expected.
5. **Absolute URLs for GitHub links.** All links to spec files and the seed in the meta banner must use absolute GitHub URLs, never relative paths.

---

## North-Star Experience

A developer lands on RootFeed, sees a scrollable timeline of realistic tech posts, and immediately understands both the product (a social feed) and the meta-layer (this was built with RootSpec). They click through a profile, explore trending tags, write a test post that appears in the feed, then follow the "View the spec" link to read the exact specification that generated what they're looking at. They leave thinking: "I could use this workflow on my own codebase."
