## Design Philosophy

**Clarity over cleverness.** Every UI decision should make the product easier to understand — not just aesthetically pleasing. When in doubt, prefer the option that a developer skimming the demo will immediately parse.

**Demo-forward framing.** The spec methodology is the product. The social feed is the vehicle. Design decisions reinforce the meta-narrative rather than hiding it.

---

## Strategic Trade-offs

| We choose | Over | Rationale |
|---|---|---|
| Static prerendering | Server-side rendering | No backend needed; GitHub Pages deployment; simpler demo story |
| Client-side interactivity only | Persisted state | Simplicity; no API or database to maintain |
| Mock data from JSON | Real user-generated content | Controllable, curated, realistic-feeling without real accounts |
| Absolute GitHub URLs | Relative paths | Static prerender at `/demos/scaffold/` subpath breaks relative links |
| Meta-banner always visible | Hiding demo origins | Transparency is a design pillar, not a feature |
| System preference + manual toggle for theme | System preference only | Respects user agency; common developer expectation |
| 10-post pagination for feed | Infinite scroll | Explicit load-more is simpler, more obviously demo-like |
| SvelteKit with adapter-static | SSR or CSR-only | Matches existing scaffold; fits GitHub Pages deployment |

## Success Metrics

This is a demo, not a production app. Success is defined as:

1. A developer visiting the site immediately understands: "This is a RootSpec scaffold demo."
2. All 5 routes render correctly with realistic mock data.
3. All interactive features (like, bookmark, follow, search, compose, theme toggle, load more) work client-side.
4. The meta-banner links resolve correctly to GitHub.
5. The spec is visible in the repo and linked from the app.
6. All MVP user stories pass their acceptance criteria.

---

## Constraints We Accept

- **No persistence:** Interactions reset on page reload. This is explicit and acceptable.
- **No real auth:** There is no login, no real user identity.
- **Fixed data set:** Only handles and post IDs from `users.json` and `posts.json` have prerendered pages.
- **Subpath deployment:** All asset URLs and internal links must work at `/demos/scaffold/`.
- **No 404 handling:** Unknown dynamic parameters produce no page — this is a static site constraint, not a product decision.

---

## Deployment Model

- **Platform:** GitHub Pages
- **Base path:** `/demos/scaffold/`
- **Framework:** SvelteKit with `@sveltejs/adapter-static`
- **Build:** Static export; all pages prerendered at build time
- **Data:** JSON files in `src/lib/data/`; no API calls

---

## Content Strategy

Mock data should feel like real developer Twitter/Bluesky circa 2025-2026:
- Tech discussions, hot takes, debugging war stories
- Replies and threads that make sense in context
- Handles and display names that sound like real developer personas
- Tags that reflect actual developer interests (webdev, ai, architecture, etc.)

Content is curated, not random. Every post should feel like something a real developer might actually post.
