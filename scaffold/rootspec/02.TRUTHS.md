## Design Philosophy

The product is a demonstration, not a product. Every design choice prioritizes **legibility of the demo itself** over feature richness. When in doubt, show less but show it clearly.

## Strategic Commitments

### We choose demo transparency over feature completeness

RootFeed deliberately exposes its own construction. Links to the seed, spec, and scaffold commit are permanent first-class UI elements. We choose this over a cleaner, less self-referential interface because the audience (developers evaluating RootSpec) values the provenance more than a polished illusion.

### We choose client-side state over persistence

All interactions — likes, bookmarks, follows, new posts — are ephemeral. We choose this over any server or localStorage strategy because it keeps the codebase minimal and the demo focused on spec methodology, not backend engineering.

### We choose realistic mock data over lorem ipsum

The posts, users, and tags feel like real tech conversations — opinions, threads, humor. We choose this over placeholder text because realistic data is what makes the demo feel *alive*, which supports the **Alive Without Being Fake** pillar.

### We choose SvelteKit conventions over custom abstractions

File-based routing, `$state` / `$derived` reactivity, Svelte stores for shared state. We choose this over custom state management or utility layers because the scaffold scenario is most compelling when the implementation is conventional — the spec does the work, not clever code.

### We choose system detection + manual toggle for theming

Dark mode respects `prefers-color-scheme` on first visit, then follows the user's explicit choice stored in localStorage. We choose this over system-only or manual-only because dark mode is table stakes in 2026 and users expect to override it.

### We choose readable prose over dense density

Feed cards, profile pages, and post detail pages all use generous whitespace. We choose this over information-dense Twitter-style layouts because developer-audience demos benefit from clear visual hierarchy that makes the interface easy to read and evaluate.

## Success Definition

This demo succeeds when a developer reads the meta banner, follows one link (seed, spec, or scaffold commit), and understands that RootSpec produced this experience from very little human input. Any interaction before that moment is setup; that moment of recognition is the outcome.

## Accepted Constraints

- No backend, no API, no authentication. Mock JSON only.
- No cross-page state persistence (localStorage exception: theme preference only).
- No real user accounts. The "author" of new posts is always u1 (alice.dev).
- Deployment at a GitHub Pages subpath (`/demos/scaffold/`) requires base path configuration throughout.
- The RootSpec version displayed in the header is the version used to build this spec (not a live API call).
