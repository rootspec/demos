# Level 2: Stable Truths

**Product:** RootFeed
**References:** 01.PHILOSOPHY.md

---

## Truth: Static-first, no server required

RootFeed is deployed as a fully prerendered static site. Every page — including dynamic routes like `/profile/[handle]` and `/post/[id]` — is generated at build time from mock JSON data. There is no backend, no database, and no API. All interactivity is client-side and ephemeral.

**Why:** GitHub Pages deployment, zero infrastructure cost, and honest scoping for a demo. The scaffold scenario answers "how does RootSpec work on an existing codebase?" — not "how do you build a production social network?"

**Trade-off:** We choose prerendered completeness over server-side dynamism. Known handles and IDs produce pages; unknown routes do not exist. This is a constraint, not a failure mode.

---

## Truth: Mock data is personality, not placeholder

The JSON data powering RootFeed is crafted to feel like a real tech community — real-ish names, recognizable developer archetypes, authentic conversations. It supports the Honest Aliveness and Playful Credibility pillars by making the demo feel worth exploring, not just worth inspecting.

**Trade-off:** We choose curated authenticity over procedural generation. Data is handcrafted, not lorem ipsum. This makes the demo more compelling and the spec more specific.

---

## Truth: Interactivity is real, persistence is not

Like counts, bookmark states, follow states, and composed posts work as live client-side interactions. They update in the UI, respond to user actions, and feel genuine. They do not persist across page loads or browser sessions.

**Why:** The Honest Aliveness pillar requires real interactions. The static deployment constraint prevents real persistence. The demo framing makes the trade-off legible and acceptable.

**Trade-off:** We choose interactive honesty over persistent state. Every session starts fresh. The meta banner makes this expectation explicit so users are never surprised.

---

## Truth: Transparency is a first-class feature

The meta banner, footer, header version badge, and GitHub links are not afterthoughts — they are core to the product's purpose. RootFeed exists to demonstrate RootSpec, and that demonstration requires showing the seams.

**Trade-off:** We choose legible construction over polished illusion. A visitor should always be able to trace the path from SEED.md to spec to implementation.

---

## Truth: Theme follows system preference, manual override is allowed

Dark/light mode defaults to the user's system preference (OS-level). A manual toggle lets users override this without changing their system settings. The preference is stored in client-side state for the session.

**Trade-off:** We choose graceful defaults over forced choice. The product respects what the user already decided at the OS level and gives them an easy out if they want something different in this context.

---

## Truth: Discovery and search are local and synchronous

Search operates on the complete in-memory post dataset, filtering in real time as the user types. Explore surfaces trending tags, suggested users, and popular posts derived entirely from the mock data. No external search index, no async fetching, no debouncing required.

**Trade-off:** We choose simplicity and responsiveness over scale. For a static demo with [small dataset size] posts and users, in-memory filtering is instantaneous and appropriate.

---

## Definition of Success

RootFeed succeeds when a developer evaluating RootSpec can:

1. Arrive at the deployed site and immediately understand it is a demo, not a real social network
2. Explore the feed, profiles, posts, and search — and have those interactions feel genuine
3. Click through to the SEED.md, spec files, and scaffold commit on GitHub with one action
4. Leave with a clear mental model of how RootSpec turned a bare scaffold into a complete experience
