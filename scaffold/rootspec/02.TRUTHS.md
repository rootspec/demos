# Level 2: Stable Truths

**References:** 01.PHILOSOPHY.md

---

## Truth: Static beats dynamic for a demo

RootFeed is built with `@sveltejs/adapter-static`. There is no server, no database, no API. All data is preloaded from JSON files at build time. Dynamic routes are prerendered from known IDs only.

**We choose reliability and zero-ops over real-time features.**

This supports the Transparent by Design pillar — visitors see exactly what they get: a static site, no magic.

---

## Truth: Mock data is the product's content layer

The JSON files in `src/lib/data/` are not test fixtures — they are the content of the site. They must feel real enough to demonstrate the UI without being mistaken for real users or real posts.

**We choose curated realism over generated placeholder content.**

---

## Truth: Client-side interactivity is ephemeral by design

Likes, follows, bookmarks, and composed posts exist only in component state. Refreshing the page resets everything. This is not a bug — it is the correct scope for a static demo.

**We choose simplicity of implementation over persistence.**

Interaction state is never stored in localStorage or cookies. The demo is always fresh.

---

## Truth: The meta layer is a first-class feature

The meta banner, footer attribution, and transparency links are not afterthoughts. They are core features that differentiate RootFeed from a generic social feed clone.

**We choose honesty about the demo over immersion.**

Every page must surface the RootSpec connection.

---

## Truth: The scaffold commit is the origin story

RootFeed started as a bare SvelteKit scaffold. That history is part of the product. The scaffold commit SHA is linked from the meta banner so visitors can see the before/after.

**We choose provenance over polish.**

---

## Truth: Routes exist only for known data

Because this is a static site, only handles and IDs present in the mock data produce pages. There are no 404 routes, no "user not found" server responses, no fallback rendering.

**We choose simplicity of routing over defensive error handling.**

---

## Trade-offs

| We choose | Over |
|-----------|------|
| Static generation | Server-side rendering |
| Curated mock data | Auto-generated placeholder content |
| Client-side ephemeral state | Persisted interactions |
| Transparency about demo nature | Full immersive illusion |
| Pre-rendered known routes only | Dynamic routing with fallbacks |
| Minimal dependency surface | Maximum feature completeness |

---

## Definition of Success

RootFeed succeeds when a developer who has never heard of RootSpec:
1. Understands quickly that this is a demo of a methodology, not a real product
2. Can navigate to the spec, seed, and scaffold commit from any page
3. Feels that the interactions (like, follow, search, compose) are realistic enough to evaluate the UX quality
4. Leaves with a clear mental model of what RootSpec's scaffold workflow produces
