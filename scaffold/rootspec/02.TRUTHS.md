# Level 2: Stable Truths

**References:** L1 (Philosophy)

---

## Design Philosophy

RootFeed is a demonstration product. Its design strategy is shaped by a single overriding commitment: **show, don't abstract**. Every architectural decision should make the scaffold story clearer, not more obscure.

---

## Core Trade-offs

| We choose... | Over... | Because... |
|---|---|---|
| Static prerendering | Server-side rendering | Demonstrates the scaffold pattern cleanly; no server needed for a demo |
| Client-side state only | Persisted state | Reinforces the "demo, not product" identity; simplifies the architecture story |
| Mock JSON data | A real database | Developers can read the data; it's part of the artifact, not hidden infrastructure |
| Visible meta-layer | Pure product immersion | The demo identity (banner, footer, version) is a first-class feature |
| Tailwind utility classes | Custom CSS | Aligns with the SvelteKit scaffold convention; readable, composable, no build step surprises |
| Instant client-side filtering | Debounced API calls | Reinforces Grounded Immediacy; no latency to work around |
| Prerendered dynamic routes | Dynamic 404 handling | Static adapter constraint; only known data exists |
| System preference detection + manual toggle | System preference only | Serves both automatic and intentional dark/light mode users |

---

## Success Definition

RootFeed succeeds when a developer can:
1. Open the deployed site and immediately understand both the feed product and the RootSpec demo context
2. Interact with likes, follows, search, and the post composer without confusion
3. Follow a link from the meta banner to the actual spec files on GitHub and see the correspondence
4. Walk away with a concrete mental model of how RootSpec's scaffold workflow functions

---

## Constraints We Accept

- **No authentication.** There is no concept of "logged-in user." The follow/like state is anonymous and session-ephemeral.
- **No persistence.** Client state resets on page reload. This is a feature of the demo, not a limitation to apologize for.
- **No server.** Deployed as a static site on GitHub Pages at `/demos/scaffold/`. All links and asset paths must account for this subpath.
- **Fixed data set.** The mock data (8 users, 30 posts, 15 tags) is the complete universe of the product. No pagination beyond "load more" on the home feed.
- **Prerendering bounds.** Only handles and IDs present in mock data are valid routes. Unknown routes produce build-time 404s, not runtime errors.

---

## Architectural Commitments

- **SvelteKit with `@sveltejs/adapter-static`** is the immutable framework choice
- **File-based routing** (`src/routes/`) maps 1:1 to product routes
- **Svelte stores** manage all interactive client state (likes, follows, composed posts, theme)
- **TypeScript throughout** — all data shapes are typed
- **Tailwind CSS** for all styling; no CSS-in-JS, no custom stylesheets beyond `app.css`
- **No external runtime dependencies** — no analytics, no CDN scripts, no third-party widgets

---

## The Demo Identity Is Non-Negotiable

The meta banner and footer are not optional polish — they are core to the product's purpose. They must:
- Appear on every page (via the layout)
- Link to absolute GitHub URLs for the scaffold commit, the spec, and the seed
- Display the RootSpec version prominently (v7.3.5)
- Be styled distinctly from feed content (not confused with post content)
