# Level 2: Stable Truths

**RootFeed — Design Strategies & Commitments**

---

## Truth: Mock data is the product's boundary

RootFeed's entire data model lives in static JSON files. There is no backend, no database, no API. This is not a temporary shortcut — it is the correct architecture for a demo that must remain reproducible, deployable as a static site, and comprehensible to a developer reading the source.

**We choose static JSON over dynamic data storage** because reproducibility and transparency outweigh persistence. Every interaction that mutates state (likes, bookmarks, follows, new posts) is explicitly client-side only and resets on page reload. This constraint is visible, not hidden.

---

## Truth: Transparency is a design strategy, not a disclaimer

Referencing the design pillars in `01.PHILOSOPHY.md` — specifically *Transparent Construction* — transparency is embedded into the visual hierarchy, not appended as fine print. The meta banner, the footer attribution, and the RootSpec version in the header are all first-class UI elements, not afterthoughts.

**We choose transparency as UI over hiding imperfection** because the demo's audience (developers) will distrust anything that obscures its own workings.

---

## Truth: Client-side state is intentionally shallow

Interactive features (like/bookmark/follow/compose) are powered by Svelte reactive state — `$state`, reactive classes — with no serialization, no localStorage persistence (except theme), and no cross-page sync beyond what Svelte's module-level stores provide.

**We choose simple reactive state over localStorage/IndexedDB persistence** because the added complexity would obscure the scaffold's teaching purpose and is not justified by user need in a demo.

Exception: theme preference (light/dark) is persisted to localStorage, because reloading and losing your theme preference would be jarring enough to undermine the *Believable Fiction* pillar.

---

## Truth: Five routes, no more

The scaffold provides exactly five routes: `/`, `/profile/[handle]`, `/post/[id]`, `/search`, `/explore`. The spec does not add routes. The implementation lives entirely within this shape.

**We choose completing the scaffold over extending it** because the demo's thesis is about the scaffold workflow — not about building a feature-complete social app. Adding routes would muddy the comparison between the starting scaffold and the finished product.

---

## Truth: Visual design follows Tailwind conventions, not custom systems

Styling uses Tailwind utility classes directly. There is no custom design system, no component library, no CSS variables beyond what Tailwind provides. Colors, spacing, and typography use Tailwind defaults with a dark-mode layer.

**We choose Tailwind utilities over custom systems** because:
1. The target audience will recognize and trust it
2. It keeps the source readable without requiring a separate design token reference
3. It aligns with the SvelteKit scaffold's setup as delivered

---

## Truth: Success is a developer thinking "I could do this"

Success for RootFeed is not engagement, retention, or virality. Success is a developer reading the demo and saying *"I understand how this was built, and I could apply this process to my own project."*

**Metrics that matter:**
- The scaffold commit is linkable and the diff is readable
- The spec files are navigable and legible
- The meta banner is on every page
- Every interactive feature actually works
- The app is deployed and accessible as a static site

**Metrics that do not matter:**
- Time on site
- Return visits
- Post composition count
- Social engagement simulations

---

## Truth: The demo audience is always a developer, never a social media user

All content, tone, and copy is written for a technically literate audience evaluating a development methodology. The mock user personas are tech-community archetypes. The post content is tech discourse. The UI is clean and functional, not viral or emotional.

**We choose developer-focused framing over general user appeal** because mismatching the audience would undermine credibility with the actual evaluation audience.

---

## Truth: The RootSpec version is always surfaced

Per the inviolable principles in `01.PHILOSOPHY.md`, the RootSpec version (currently v7.2.5 as specified in the framework file) is always visible in the navigation header. This is a commitment, not a nice-to-have.

**We choose surfacing the tool version over clean minimalism** because the demo's function is to market a methodology. Version visibility is part of the product.
