# Level 2: Stable Truths

**References:** 01.PHILOSOPHY.md

---

## Truth: Scaffold context is permanent, not dismissible

The demo framing is never a modal, banner that can be closed, or optional feature. It is structural — woven into the layout so that every page carries the context of what this app is and how it was built. This supports the "Transparent by Construction" pillar.

We choose persistent meta-context over uncluttered UI. A clean social feed that hides its origins would be a worse demo, not a better product.

---

## Truth: Client-side interactivity, zero persistence

All interactive state — likes, bookmarks, follows, new posts — lives in Svelte stores and resets on page reload. There is no backend, no database, no auth. This constraint is a feature: it keeps the demo deployable as a static site and makes the "no real server" nature explicit.

We choose static deployability over persistence. This is a deliberate trade-off that simplifies infrastructure without compromising the demo's purpose.

---

## Truth: Mock data must earn belief

Content that reads like placeholder text undermines the demo. The 8 personas have real tech backgrounds, the 30+ posts cover topics developers actually argue about, and the threading model creates real conversational texture.

We choose realistic mock data over obviously fake data. The social experience needs to be credible for the spec-to-implementation mapping to feel meaningful.

---

## Truth: RootSpec version and build identity are always visible

The spec version (v7.2.5), build date, and attribution are structural elements of the footer — not documentation afterthoughts. Visitors should be able to immediately see what version of RootSpec produced this output.

We choose visible provenance over clean aesthetics. This aligns with the "Spec as First-Class Artifact" pillar.

---

## Truth: Dark and light mode are both required

System preference detection plus manual toggle is baseline expected behavior in 2026. This is not a nice-to-have. The app respects user environment preferences by default and allows override.

We choose both modes over defaulting to one. This is a table-stakes commitment.

---

## Truth: Performance is demo-grade, not production-grade

Load times, bundle optimization, and caching strategies are not primary concerns. The app loads mock data synchronously from JSON files. The acceptable performance bar is: loads fast enough that visitors aren't waiting, nothing more.

We choose simplicity over optimization. Premature optimization would obscure the scaffold-to-spec delta that is the demo's core value.

---

## Definition of Success

A developer who opens RootFeed can, within 5 minutes:
1. Understand that it's a RootSpec scaffold demo
2. Navigate to the spec source files
3. See all five routes working with realistic mock data
4. Interact with at least two interactive features (like a post, follow a user)
5. Find the footer showing RootSpec version and build provenance
