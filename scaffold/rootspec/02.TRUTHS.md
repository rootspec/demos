# Level 2: Stable Truths

## Design Philosophies

### Specification fidelity over creative improvisation
Every feature exists because the spec says it should. Nothing is added "because it looks good" or "because other social apps have it." The output is a faithful translation of the seed, and that discipline is the point.

### Transparency over immersion
RootFeed never tries to fool anyone into thinking it's a real social network. The demo framing is always visible — meta banners, attribution, links to source. This supports Earned Credibility: the value is in seeing how it was made, not in pretending it wasn't.

### Depth over breadth
Fewer features, fully specified and tested, beats many features loosely implemented. A like button that's spec-traced, tested, and verifiable is worth more than ten untested features. This supports Approachable Confidence: developers need to see that *what exists works*, not that *a lot exists*.

### Client-side simplicity over real infrastructure
Mock data, client-side state, static adapter. No backend, no database, no auth. This is a deliberate constraint: the demo showcases RootSpec's specification-to-implementation pipeline, not devops. Reducing infrastructure complexity keeps the focus on what matters.

## Trade-offs

- **Verifiability over polish** — We choose testable interactions over pixel-perfect design. A feature that can't be validated against the spec doesn't ship.
- **Explainability over elegance** — We choose code and architecture that's easy to trace back to the spec over clever abstractions. A developer reading the code should see the spec's structure reflected in it.
- **Demo clarity over realism** — We choose obvious demo framing over a seamless social media experience. Visitors should never wonder "is this real?"
- **Completeness of tested features over feature count** — We choose a smaller set of fully-tested features over a larger set of untested ones.

## Success Criteria

- A developer can trace any visible feature back through user story → system spec → interaction pattern → philosophy → seed
- All user stories have passing Cypress tests
- The scaffold-to-product transformation is legible: comparing the original scaffold commit to the final state tells a clear story
- Time from landing to "I should try RootSpec" is under five minutes

## Constraints

- Static deployment only — no server-side runtime
- All data is mock JSON — no persistence across page loads
- Client-side state resets on refresh — acceptable for a demo
- The spec itself is part of the deliverable — not just the app
