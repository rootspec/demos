# Level 2: Stable Truths

**Product:** RootFeed
**References:** L1 Philosophy

---

## Design Philosophy

RootFeed is built on the principle that demos should be honest artifacts. The best demo code teaches by being structurally sound, spec-driven, and transparent about its origins — not by being visually impressive or hiding complexity.

---

## Truths

### Truth 1: Static over dynamic
We choose prerendered static output over server-side rendering. All routes are prebuilt from mock data at build time. This eliminates server infrastructure, matches the SvelteKit static adapter constraint, and makes the demo reproducible and hostable from a CDN.

**Trade-off:** We choose reliability and deployability over dynamic personalization.

### Truth 2: Client-side state only
Interactive features (likes, bookmarks, follows, composed posts) live exclusively in Svelte stores — reactive, in-memory, not persisted across page loads. No backend, no localStorage sync for interaction state (theme preference is the only exception).

**Trade-off:** We choose simplicity over persistence. Users who refresh lose their session state — this is acceptable for a demo.

### Truth 3: Mock data is the source of truth for content
Users, posts, and tags come from JSON files in `src/lib/data/`. No API calls, no dynamic data fetching. Content feels realistic but is entirely fictional.

**Trade-off:** We choose predictability over freshness. The feed never changes unless the JSON changes.

### Truth 4: Spec drives implementation, scaffold provides structure
The routing skeleton was given. Everything else — data, interactions, visual design, accessibility, content — was derived from the spec. This separation is the core demo point.

**Trade-off:** We choose spec-driven discipline over developer intuition. When in doubt, the spec wins.

### Truth 5: Transparency is a feature
The meta banner, footer attribution, and links to source artifacts are not optional polish — they are core to the product's purpose. A demo that hides its scaffolding is a worse demo.

**Trade-off:** We choose honest exposition over clean aesthetics. The banner stays.

### Truth 6: Interactions must be visibly responsive
Likes, bookmarks, follows — any action a user takes must have an immediate, visible response. No optimistic UI without visual feedback. No silent state changes.

**Trade-off:** We choose user trust over implementation simplicity. Every toggle needs a visual state change.

### Truth 7: Dark/light theme respects system preference
Theme defaults to the OS preference, stored in localStorage on manual toggle, and applied without flash on load.

**Trade-off:** We choose user comfort over implementation uniformity. The extra logic for theme initialization is worth it.

## Success Definition

RootFeed succeeds when a developer who encounters the demo can:

1. Understand within [T] seconds that this is a scaffold demo, not a real product
2. Navigate to the spec, seed, and scaffold commit from the UI
3. Experience [N] interactive features (likes, search, follow, compose, theme) working as specified
4. Feel confident that RootSpec could drive similar quality output from their own codebase

---

## Constraints

- **SvelteKit static adapter** — All pages must be prerenderable; no server routes
- **No authentication** — All interactions are anonymous, client-side only
- **Mock data only** — No external API dependencies; demo works offline
- **GitHub Pages deployment** — Base path `/demos/scaffold/` must be configured; all asset URLs must be relative to this subpath
- **RootSpec version [V]** — Spec format must conform to the version detected at init time
