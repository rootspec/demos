## Design Philosophy

RootFeed is built on the principle of **demonstrated transparency over claimed quality**. We do not assert that RootSpec works — we show it, live, with the full spec publicly accessible alongside the running product. Every design decision reinforces this: the meta banner stays, the footer stays, the demo framing never hides.

## Trade-offs

| We choose... | Over... | Because... |
|---|---|---|
| Client-side ephemeral state | Server persistence | Simplicity and transparency — sessions that reset on reload reinforce the "this is a demo" framing |
| Static site generation | Server-side rendering | GitHub Pages deployment requires static output; no server required |
| Mock JSON data | Database or API | Zero infrastructure dependencies; the demo must run from a GitHub Pages URL |
| Playful but realistic content | Lorem ipsum or placeholder data | Realistic content makes interactions feel authentic, which serves the Playful Authenticity pillar |
| Visible spec links everywhere | Clean "app-like" polish | Radical Transparency means the scaffolding is part of the story |
| Minimal route surface (5 routes) | Feature expansion | The scaffold scenario is about depth over breadth — implement the spec well, not many features loosely |
| Light/dark theme support | Single-theme simplicity | System preference detection is table stakes for any credible modern app |

## How We Define Success

- A developer evaluating RootSpec can arrive at the deployed URL and immediately understand: (1) this is a demo, (2) it was spec-driven, (3) they can inspect the spec
- All interactive features are functional with no broken states
- The Cypress test suite passes against the deployed build
- The spec is publicly accessible from links in the running app

## Constraints We Accept

- **No real authentication** — the "current user" is always `alice.dev` (u1), hardcoded. No login flow.
- **No data persistence** — likes, bookmarks, follows, and composed posts reset on page reload. This is intentional and communicated to users.
- **No backend** — all data is bundled as static JSON. No API calls, no fetch to external services.
- **Static deployment** — must build with `@sveltejs/adapter-static` and deploy to a subpath (`/demos/scaffold/`). All asset and link paths must resolve from the base path.
- **Mock data bounds** — [small user count] users, [small post count] posts, [small tag count] tags. Content must feel realistic but the data set is fixed.
- **Post character limit** — posts are capped at [post_max_chars] characters in the composer. This mirrors real-world constraints and keeps the UI predictable.

## Versioning Commitment

RootSpec version (v7.2.5) is displayed in the app header. When the framework version changes, this display updates. It is not decorative — it is a commitment to version-aware transparency.

## What This Product Is Not

- Not a real social network
- Not a production app
- Not a backend for any kind of user data
- Not a showcase of SvelteKit features — it's a showcase of RootSpec on SvelteKit
